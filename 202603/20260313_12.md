## 背景        
### 11.3.5 时序数据模型案例解析  
  
为了让读者真正掌握 TimescaleDB 的时序数据建模方法，本节将通过两个典型案例进行深入解析：  
  
1. 物联网（IoT）设备监控场景  
2. 金融行情数据分析场景  
  
我们将从 数据表设计、Hypertable 创建、分区策略、连续聚合、数据保留与压缩 全流程讲解，帮助读者在实际项目中落地时序数据库。  
  
   
  
#### 一、案例一：物联网设备监控  
  
##### 1. 业务背景  
  
假设一个智能工厂部署了 10,000 台传感器，每秒采集一次温度、湿度和电压数据。目标：  
  
* 实时监控各设备状态  
* 统计每小时平均温度、湿度  
* 存储历史数据一年，超过一年压缩存储  
  
数据特点：  
  
* 高写入频率：10,000 条/秒  
* 数据量大：每天约 8640万条  
* 查询需求：实时分析与历史趋势分析并存  
  
   
  
##### 2. 数据表设计  
  
采用 窄表设计，每行包含一个时间戳、一台设备的所有指标：  
  
```sql id="iot01"  
CREATE TABLE sensor_metrics (  
    time TIMESTAMPTZ NOT NULL,  
    device_id TEXT NOT NULL,  
    temperature DOUBLE PRECISION,  
    humidity DOUBLE PRECISION,  
    voltage DOUBLE PRECISION  
);  
```  
  
优势：  
  
* 指标统一存储  
* 支持列式压缩  
* 写入高效  
  
   
  
##### 3. 创建 Hypertable  
  
```sql id="iot02"  
SELECT create_hypertable(  
    'sensor_metrics',  
    'time',  
    chunk_time_interval => INTERVAL '1 day',  
    partitioning_column => 'device_id'  
);  
```  
  
解释：  
  
* chunk_time_interval = 1 day：每天生成一个分片  
* partitioning_column = device_id：不同设备独立分区，提高写入并行度  
  
这种设计支持百万级写入并发，同时便于数据管理。  
  
   
  
##### 4. 实时分析：连续聚合  
  
创建 每小时平均指标的连续聚合视图：  
  
```sql id="iot03"  
CREATE MATERIALIZED VIEW sensor_hourly  
WITH (timescaledb.continuous)  
AS  
SELECT  
    time_bucket('1 hour', time) AS hour,  
    device_id,  
    avg(temperature) AS avg_temp,  
    avg(humidity) AS avg_hum,  
    avg(voltage) AS avg_volt  
FROM sensor_metrics  
GROUP BY hour, device_id;  
```  
  
设置自动刷新策略：  
  
```sql id="iot04"  
SELECT add_continuous_aggregate_policy(  
    'sensor_hourly',  
    start_offset => INTERVAL '7 days',  
    end_offset => INTERVAL '1 hour',  
    schedule_interval => INTERVAL '5 minutes'  
);  
```  
  
优势：  
  
* 最近7天的数据保持实时更新  
* 历史数据批量刷新，提高效率  
  
   
  
##### 5. 数据保留与压缩  
  
假设保留一年数据：  
  
```sql id="iot05"  
-- 自动删除超过1年的原始数据  
SELECT add_retention_policy(  
    'sensor_metrics',  
    older_than => INTERVAL '365 days'  
);  
  
-- 压缩超过30天的数据  
ALTER TABLE sensor_metrics SET (  
    timescaledb.compress,  
    timescaledb.compress_segmentby = 'device_id'  
);  
  
SELECT compress_chunk(i)  
FROM show_chunks('sensor_metrics') i  
WHERE i < now() - INTERVAL '30 days';  
```  
  
效果：  
  
* 1年内数据可实时分析  
* 超过30天历史数据压缩存储，减少约 90% 的磁盘占用  
* SQL 查询无需修改，自动解压  
  
   
  
##### 6. 查询示例  
  
实时查询最近5分钟设备平均温度：  
  
```sql id="iot06"  
SELECT time_bucket('5 minutes', time) AS bucket,  
       device_id,  
       avg(temperature) AS avg_temp  
FROM sensor_metrics  
WHERE time > now() - INTERVAL '5 minutes'  
GROUP BY bucket, device_id;  
```  
  
历史趋势分析（过去1个月每小时平均）：  
  
```sql id="iot07"  
SELECT *  
FROM sensor_hourly  
WHERE hour > now() - INTERVAL '30 days'  
ORDER BY hour;  
```  
  
   
  
#### 二、案例二：金融行情数据分析  
  
##### 1. 业务背景  
  
金融市场每秒产生大量交易数据，包括：  
  
* 股票/期货成交时间  
* 价格  
* 成交量  
  
目标：  
  
* 实时计算分钟级和小时级 K 线  
* 支持回测分析和趋势分析  
  
数据特点：  
  
* 写入高频：每秒百万条记录  
* 查询以时间窗口为核心  
* 历史数据量巨大，需要压缩存储  
  
   
  
##### 2. 数据表设计  
  
```sql id="fin01"  
CREATE TABLE trade_data (  
    time TIMESTAMPTZ NOT NULL,  
    symbol TEXT NOT NULL,  
    price DOUBLE PRECISION,  
    volume BIGINT  
);  
```  
  
创建 Hypertable：  
  
```sql id="fin02"  
SELECT create_hypertable(  
    'trade_data',  
    'time',  
    chunk_time_interval => INTERVAL '1 hour',  
    partitioning_column => 'symbol'  
);  
```  
  
* 每小时一个 chunk：适合金融交易场景  
* 按股票代码分区：提高并行写入和查询效率  
  
   
  
##### 3. 连续聚合计算 K 线  
  
创建 分钟级 K 线：  
  
```sql id="fin03"  
CREATE MATERIALIZED VIEW trade_minute  
WITH (timescaledb.continuous)  
AS  
SELECT  
    time_bucket('1 minute', time) AS minute,  
    symbol,  
    first(price, time) AS open,  
    max(price) AS high,  
    min(price) AS low,  
    last(price, time) AS close,  
    sum(volume) AS volume  
FROM trade_data  
GROUP BY minute, symbol;  
```  
  
自动刷新策略：  
  
```sql id="fin04"  
SELECT add_continuous_aggregate_policy(  
    'trade_minute',  
    start_offset => INTERVAL '1 day',  
    end_offset => INTERVAL '1 minute',  
    schedule_interval => INTERVAL '1 minute'  
);  
```  
  
   
  
##### 4. 压缩历史数据  
  
历史交易数据通常存储超过半年：  
  
```sql id="fin05"  
ALTER TABLE trade_data SET (  
    timescaledb.compress,  
    timescaledb.compress_segmentby = 'symbol'  
);  
  
SELECT compress_chunk(i)  
FROM show_chunks('trade_data') i  
WHERE i < now() - INTERVAL '180 days';  
```  
  
优势：  
  
* 降低存储成本  
* 支持历史趋势查询  
* 查询透明，SQL无需修改  
  
   
  
##### 5. 查询示例  
  
实时查询最近1分钟成交均价：  
  
```sql id="fin06"  
SELECT symbol,  
       avg(price) AS avg_price  
FROM trade_data  
WHERE time > now() - INTERVAL '1 minute'  
GROUP BY symbol;  
```  
  
查询过去30天小时级 K 线：  
  
```sql id="fin07"  
SELECT *  
FROM trade_hourly  
WHERE hour > now() - INTERVAL '30 days';  
```  
  
   
  
#### 小结  
  
| 维度   | IoT监控案例   | 金融行情案例     |  
| ---- | --------- | ---------- |  
| 数据量  | 每天千万级     | 每秒百万级      |  
| 分区策略 | 时间 + 设备ID | 时间 + 股票代码  |  
| 聚合策略 | 每小时平均     | 分钟/小时 K 线  |  
| 压缩策略 | 超过30天数据压缩 | 超过180天数据压缩 |  
| 数据保留 | 1年        | 3年历史       |  
| 查询模式 | 实时 + 历史趋势 | 实时 + 回测分析  |  
  
通过上述案例可以看出：  
  
1. Hypertable 分区策略是高性能时序数据库的核心  
2. 连续聚合实现实时分析与历史分析的无缝融合  
3. 压缩与数据保留策略有效降低存储成本  
4. PostgreSQL 的 SQL 兼容性保证了业务逻辑简单迁移  
  
这些实践方法可以直接应用于 IoT、金融、运维、监控等多种高吞吐场景，充分发挥 TimescaleDB 在大规模时序数据管理中的优势。
