## 背景            
  
PostgreSQL 逻辑复制迎来“减法革命”：EXCEPT TABLE     
  
> **管理的本质，不是记住所有表，而是清晰地定义“不要什么”。**  
  
你有没有这样的经历——为了搭建逻辑复制，面对几百张表的数据库，你必须在 `CREATE PUBLICATION` 时手动列出所有要复制的表：  
  
```sql  
CREATE PUBLICATION pub1 FOR TABLE   
users, orders, products, inventory, shipments, payments, ... (此处省略 200 行)  
```  
  
每次新增一张表，你都得记得修改发布定义；每次看到那个几百行的 SQL，你都胆战心惊——万一漏了哪张表，订阅端的数据就不一致了。  
  
更痛苦的是，当你只想排除几张日志表或临时表时，你不得不放弃简洁的 `FOR ALL TABLES`，回到这种“体力活”模式。  
  
**2026 年 3 月，Amit Kapila 提交的 `fd366065` 补丁，终结了这种无奈。** PostgreSQL 19 为 `CREATE PUBLICATION ... FOR ALL TABLES` 引入了 `EXCEPT TABLE` 子句，让你能用声明式的方式定义“复制所有表，除了这几张”。  
  
## 第一性原理：为什么“全量包含+排除”是逻辑复制的必然选择？  
  
让我们回到逻辑复制的本质。一个发布（Publication）定义了一组要复制的表。从集合论角度看，有两种定义方式：  
  
1. **列举法**：手动列出所有要复制的表。  
2. **描述法**：用规则描述哪些表要复制（例如“所有表”）。  
  
`FOR ALL TABLES` 就是描述法，它表达了“**所有当前及未来创建的表**”这个集合。这个语法极大地简化了管理：你不需要知道未来会新增什么表，复制会自动涵盖它们。  
  
但描述法有一个固有的缺陷： **无法表达“几乎全部，除了少数”** 。当你有 200 张表，其中 5 张不需要复制时，你被迫退回到列举法——不仅繁琐，而且失去了“自动包含未来表”的优势。  
  
**`EXCEPT TABLE` 的引入，正是为了填补这个逻辑空白。** 它让描述法具备了表达“差集”的能力，使得复制规则可以无限接近业务需求：  
  
```sql  
CREATE PUBLICATION pub1   
    FOR ALL TABLES   
    EXCEPT TABLE logs, temp_data, archive_2025;  
```  
  
这个语法既保留了“自动包含未来表”的便利，又允许你精确排除不需要复制的表。**这是对集合论在数据库语法中的一次完美应用。**  
  
## 破局者：EXCEPT TABLE 如何改变游戏规则？  
  
### 1. 语法革命：从“要什么”到“不要什么”  
  
补丁带来的核心变化是扩展了 `CREATE PUBLICATION` 语法：  
  
```sql  
CREATE PUBLICATION name  
    FOR ALL TABLES  
    [ EXCEPT TABLE table_name [, ...] ]  
```  
  
例如：  
```sql  
CREATE PUBLICATION prod_pub   
    FOR ALL TABLES   
    EXCEPT TABLE audit_log, temp_import;  
```  
  
这个发布会复制数据库中**所有当前及未来创建的表**，除了 `audit_log` 和 `temp_import` 这两张。  
  
### 2. 实现机制：微小的内核改动，巨大的管理收益  
  
补丁在系统表 `pg_publication_rel` 中新增了 `pr except` 列（布尔值），标记该关系是否被排除。对于分区表，排除是**根级别**生效的：如果你排除了一个分区根表，那么它的所有当前及未来分区都会被自动排除。  
  
**这种设计体现了工程上的克制与远见**：只用最小的内核改动（一个新列），就实现了一个强大的管理特性，并为未来 `ALTER PUBLICATION` 的动态管理预留了空间。  
  
### 3. 分区表处理：一次性排除整个家族  
  
对于分区表，排除语义非常清晰：排除根表 = 排除整个分区树。这意味着你不需要关心有多少个分区，也不需要在新增分区时担心它们意外出现在订阅端。例如：  
  
```sql  
CREATE PUBLICATION pub1 FOR ALL TABLES EXCEPT TABLE measurement;  
```  
  
如果 `measurement` 是一个按时间分区的大表，所有历史分区和未来新建的分区都不会被复制。这对于管理时序数据、日志数据的复制非常有用——你通常不需要把原始数据复制到分析库，只需复制聚合结果。  
  
## 权威数据：EXCEPT TABLE 能为 DBA 省多少时间？  
  
让我们看一个典型的中型 SaaS 公司场景：  
  
- 数据库表总数：**350 张**  
- 需要复制的表：**328 张**（22 张是日志、临时表、归档表，不需要复制）  
- 表增长率：**每月新增 8-10 张**  
  
### 旧方案：手动维护表列表  
  
每次创建发布时，你必须列出 328 张表的名称。假设每行写 5 张表，也需要 66 行 SQL。这不仅是体力活，更是**错误的高发区**——漏掉一张表，订阅端的数据就不完整，业务可能直到用户投诉才发现。  
  
**维护成本量化**：  
- 每次新增表：需要修改发布，添加表名。按每月 8 张表计算，一年 96 次操作。每次操作 2 分钟（查找表名、执行 ALTER、验证），年耗时 **3.2 小时**。  
- 每次排除表变化：如果有表不再需要复制（例如新创建的日志表），同样需要修改列表。假设一年发生 20 次，每次 2 分钟，年耗时 **0.7 小时**。  
- 故障处理：因遗漏表导致的数据不一致事故，假设每季度发生一次，每次平均修复时间 2 小时（调查、补数据），年耗时 **8 小时**。  
  
**旧方案年总耗时：约 12 小时。**  
  
### 新方案：FOR ALL TABLES EXCEPT ...  
  
创建发布时只需列出 22 张排除表，SQL 从 66 行变成 **4-5 行**。  
  
**维护成本量化**：  
- 每次新增表：**零操作**——新表自动被复制。  
- 每次排除表变化：仅当某张表需要从复制中排除时，才修改排除列表。假设一年 20 次，每次 1 分钟（因为列表短，不易出错），年耗时 **0.3 小时**。  
- 故障处理：由于排除了手动列举，因遗漏导致的故障概率**趋近于零**。  
  
**新方案年总耗时：约 0.3 小时。**  
  
**节省时间：11.7 小时/年。** 这还只是针对一个数据库。如果你管理 50 个这样的数据库，一年就能节省 **585 小时**——相当于 **73 个工作日**！  
  
更重要的是，**你从繁琐的列表维护中解放出来**，可以把精力投入到更有价值的架构设计和性能优化上。  
  
## 第一性原理的边界：什么时候 EXCEPT TABLE 不适用？  
  
任何抽象都有前提。`EXCEPT TABLE` 的核心前提是： **你要复制的表集是“全集减去固定子集”** 。当这个前提崩塌时，你需要回归基础。  
  
### 崩塌场景 1：排除规则动态变化  
  
如果你的复制需求是“复制除了昨天创建的表之外的所有表”（即排除集随时间变化），那么静态的 `EXCEPT TABLE` 无法表达。你可能需要定期手动修改排除列表，或者干脆回到列举法。  
  
**应对策略**：考虑使用**多个发布**组合。例如，创建一个 `FOR ALL TABLES` 发布，再创建另一个包含动态排除表的发布，订阅端通过多个订阅组合过滤。或者，使用外部工具在发布定义上做一层动态管理。  
  
### 崩塌场景 2：表之间存在完整性依赖  
  
假设表 A 和表 B 有外键约束，你决定排除 B 但复制 A。在订阅端，当应用来自 A 的修改时，可能会因为找不到 B 中的对应行而失败。  
  
**应对策略**：在这种情况下，`EXCEPT TABLE` 本身不是问题，但你需要理解复制拓扑的约束。要么手动确保依赖表一起复制，要么在订阅端移除或禁用外键（需谨慎）。  
  
### 崩塌场景 3：需要行级或列级过滤  
  
`EXCEPT TABLE` 是**表级**的过滤。如果你的需求是“复制 users 表，但排除其中 deactivated 列 = true 的行”，或者“复制 orders 表，但不复制 credit_card 列”，那么你需要的是**行级或列级过滤**——这是另一个特性领域。  
  
**应对策略**：目前 PostgreSQL 不直接支持行/列级发布过滤。你可以通过**物化视图**或**触发器**在应用端过滤，或者考虑使用第三方工具（如 pglogical 的 row_filter）。对于列级过滤，可以创建一个不包含敏感列的视图，然后发布该视图（但视图不可更新，需注意）。  
  
### 崩塌场景 4：混合使用表列表和排除  
  
如果你已经有一个用 `FOR TABLE` 列举了 100 张表的发布，现在想排除其中 3 张，你无法用 `EXCEPT` 直接修改（当前补丁只支持 `FOR ALL TABLES`）。你需要等到后续的 `ALTER PUBLICATION` 支持，或者重建发布。  
  
**应对策略**：对于新设计，尽量使用 `FOR ALL TABLES EXCEPT` 作为默认模式。对于现有发布，可以计划在维护窗口重建为新的语法。  
  
## 未来展望：从 EXCEPT TABLE 到声明式复制管理  
  
这个补丁只是开始。提交信息中明确提到：**后续将实现 `ALTER PUBLICATION` 支持管理这些排除表**。届时，你将可以动态地添加或移除排除表，无需重建发布。  
  
更长远看，我们可能会看到：  
- **基于模式的排除**：`FOR ALL TABLES EXCEPT ALL TABLES IN SCHEMA staging`  
- **基于标签的排除**：通过表注释或自定义属性定义复制策略  
- **与订阅端过滤集成**：订阅端也能定义自己的过滤规则  
  
## DBA 的行动指南  
  
### 1. 立即更新你的发布创建模板  
  
将新数据库的发布创建脚本从：  
```sql  
CREATE PUBLICATION data_pub FOR TABLE table1, table2, ...;  
```  
改为：  
```sql  
CREATE PUBLICATION data_pub FOR ALL TABLES EXCEPT TABLE   
    logs, temp_import, archive_2025;  
```  
  
### 2. 审查现有发布，识别可重构场景  
  
查找那些 `FOR TABLE` 列表特别长的发布，分析它们是否可以用 `FOR ALL TABLES EXCEPT` 简化。注意考虑未来新增表的预期。  
  
### 3. 为分区表设计排除策略  
  
对于分区大表，评估是否真的需要在订阅端保留所有历史数据。如果不需要，将根表加入排除列表，可以显著减少网络传输和存储成本。  
  
### 4. 监控排除表的变更  
  
虽然排除表不复制数据，但它们的结构变更（如 DDL）是否应该复制？当前 PostgreSQL 逻辑复制不复制 DDL，所以这不是问题。但如果你使用第三方工具支持 DDL 复制，需要注意排除表的 DDL 处理。  
  
## 结语  
  
PostgreSQL 19 的 `EXCEPT TABLE` 特性，本质上是**用集合论的思维简化了复制规则的定义**。它让 DBA 从繁琐的列表维护中解放出来，同时保留了 `FOR ALL TABLES` 的“未来兼容”优势。  
  
这再次印证了一个道理：**数据库可管理性的提升，往往不是靠增加复杂功能，而是靠提供更符合人类思维的表达方式。**  
  
从现在开始，创建新发布时，请记住这个简洁而强大的模式：**复制所有，除了你明确不要的那些。**

Allow table exclusions in publications via EXCEPT TABLE.  
  
Extend CREATE PUBLICATION ... FOR ALL TABLES to support the EXCEPT TABLE  
syntax. This allows one or more tables to be excluded. The publisher will  
not send the data of excluded tables to the subscriber.  
  
To support this, pg_publication_rel now includes a prexcept column to flag  
excluded relations. For partitioned tables, the exclusion is applied at  
the root level; specifying a root table excludes all current and future  
partitions in that tree.  
  
Follow-up work will implement ALTER PUBLICATION support for managing these  
exclusions.
