## 背景                   
### 11.3.1 时序数据模型设计  
  
在物联网、金融交易、运维监控等场景中，大量数据具有一个共同特点：数据按照时间顺序持续产生。这类数据被称为时序数据（Time-Series Data）。典型例子包括：  
  
* 服务器 CPU / 内存监控指标  
* IoT 传感器数据  
* 股票行情数据  
* 用户行为日志  
  
在 PostgreSQL 生态中，处理大规模时序数据最成熟的解决方案是 TimescaleDB。TimescaleDB 通过扩展机制在 PostgreSQL 上实现了专门针对时序数据优化的数据模型和存储结构，例如 Hypertable（超表） 和 Chunk（分片）。   
  
本节将介绍时序数据建模的基本原则以及 TimescaleDB 的核心数据模型设计方法。  
  
   
  
#### 一、时序数据的特点  
  
时序数据与传统业务数据相比具有明显的结构特征。  
  
| 特征      | 描述          |  
| ------- | ----------- |  
| 时间维度核心  | 每条数据都包含时间戳  |  
| 写入频率高   | 每秒可能产生大量数据  |  
| 数据量巨大   | 数十亿甚至万亿级记录  |  
| 查询以时间为主 | 大多数查询包含时间范围 |  
| 历史数据多   | 数据通常长期存储    |  
  
例如 IoT 设备监控数据：  
  
| time                | device_id | temperature | humidity |  
| ------------------- | --------- | ----------- | -------- |  
| 2026-01-01 10:00:01 | sensor_01 | 22.3        | 40       |  
| 2026-01-01 10:00:02 | sensor_01 | 22.5        | 41       |  
| 2026-01-01 10:00:03 | sensor_01 | 22.4        | 41       |  
  
在这种数据模型中：  
  
* time 是核心维度  
* device_id 是实体标识  
* 其他列是指标（metrics）  
  
  
  
#### 二、TimescaleDB的核心数据模型  
  
TimescaleDB 为时序数据提供了专门的数据结构，其中最重要的概念是：  
  
* Hypertable（超表）  
* Chunk（数据分片）  
* Dimension（维度）  
  
  
  
##### 1 Hypertable（超表）  
  
Hypertable 是 TimescaleDB 的核心抽象结构。  
  
从逻辑上看，它与普通 PostgreSQL 表完全一致；但在内部实现上，Hypertable 会自动将数据按照时间进行分区。   
  
结构示意：  
  
```  
Hypertable  
   │  
   ├── Chunk_1 (2026-01-01)  
   ├── Chunk_2 (2026-01-02)  
   ├── Chunk_3 (2026-01-03)  
```  
  
每个 Chunk 存储一段时间范围的数据。  
  
当数据写入时，TimescaleDB 会自动将记录路由到对应的 chunk 中。   
  
这种设计具有两个重要优势：  
  
1. 写入效率高  
2. 查询只扫描相关时间分区  
  
例如查询：  
  
```sql  
SELECT *  
FROM metrics  
WHERE time > now() - interval '1 day';  
```  
  
数据库只需要扫描最近一天的 chunk，而不必扫描整个表。   
  
  
  
##### 2 时间维度（Time Dimension）  
  
在时序数据库中，时间列是最重要的字段。  
  
设计时需要满足：  
  
* 使用 `TIMESTAMP` 或 `TIMESTAMPTZ`  
* 保证单调递增  
* 作为主要分区键  
  
示例表结构：  
  
```sql  
CREATE TABLE cpu_metrics (  
    time TIMESTAMPTZ NOT NULL,  
    host TEXT,  
    cpu_usage DOUBLE PRECISION  
);  
```  
  
随后可以创建 hypertable：  
  
```sql  
SELECT create_hypertable('cpu_metrics','time');  
```  
  
该命令会将普通表转换为 Hypertable。   
  
  
  
##### 3 空间维度（Space Dimension）  
  
在某些场景中，仅按时间分区仍然不足以支撑超大规模数据。  
  
TimescaleDB 允许添加 空间维度（Space Dimension），例如：  
  
* device_id  
* region  
* sensor_id  
  
这样可以形成二维分区结构：  
  
```  
Time Dimension  
      │  
      ├── device_1  
      ├── device_2  
      ├── device_3  
```  
  
例如：  
  
```sql  
SELECT create_hypertable(  
    'metrics',  
    'time',  
    partitioning_column => 'device_id'  
);  
```  
  
这种设计可以提高并行写入能力。  
  
  
  
#### 三、时序数据表设计原则  
  
为了获得最佳性能，时序数据建模通常遵循以下设计原则。  
  
  
  
##### 1 “窄表 + 高行数”设计  
  
时序数据库通常采用：  
  
窄表（Narrow Table）设计  
  
例如：  
  
| time | device_id | metric | value |  
| ---- | --------- | ------ | ----- |  
  
而不是：  
  
| time | device_id | cpu | memory | disk |  
| ---- | --------- | ------ | ----- | ----- |  
  
原因：  
  
* 指标类型更加灵活  
* 更易压缩  
* 写入模式统一  
  
但在某些监控场景中，宽表也可能更高效，因此需要根据业务选择。  
  
  
  
##### 2 时间字段必须参与索引  
  
几乎所有时序查询都包含时间条件：  
  
```sql  
WHERE time BETWEEN ...  
```  
  
因此通常需要创建索引：  
  
```sql  
CREATE INDEX ON metrics(time DESC);  
```  
  
对于设备数据，还可以建立复合索引：  
  
```sql  
CREATE INDEX ON metrics(device_id, time DESC);  
```  
  
  
  
##### 3 控制 Chunk 大小  
  
Hypertable 会按照时间范围生成 chunk。  
  
默认情况下：  
  
```  
chunk interval = 7 days  
```  
  
也可以手动指定：  
  
```sql  
SELECT create_hypertable(  
    'metrics',  
    'time',  
    chunk_time_interval => interval '1 day'  
);  
```  
  
合理的 chunk 大小能够：  
  
* 减少索引扫描  
* 提高查询效率  
  
  
  
#### 四、时序数据存储优化  
  
TimescaleDB 为时序数据提供了一些重要优化技术。  
  
  
  
##### 1 压缩存储  
  
时序数据往往具有高度重复性，因此非常适合压缩。  
  
TimescaleDB 可以对旧数据进行 chunk 级压缩。  
  
官方文档显示，压缩后数据量可以减少 90% 以上。   
  
压缩后的存储结构类似：  
  
```  
原始数据：  
10万行  
  
压缩后：  
1行 + 数组列  
```  
  
这种方式大幅降低存储成本。  
  
  
  
##### 2 行列混合存储  
  
TimescaleDB 的 Hypercore 存储引擎采用：  
  
* 行存储：用于最新数据（写入快）  
* 列存储：用于历史数据（分析快）  
  
这种混合存储结构可以同时兼顾：  
  
* 高吞吐写入  
* 高效分析查询。   
  
  
  
#### 五、企业应用案例  
  
时序数据库已经成为多个行业的关键基础设施。  
  
  
  
##### 1 IoT物联网平台  
  
物联网平台通常需要采集大量设备数据，例如：  
  
* 温度传感器  
* 电力计量设备  
* 工业设备监控  
  
典型数据规模：  
  
```  
100万设备  
每秒1条数据  
≈ 864亿条/天  
```  
  
这种规模的数据必须采用时序数据库进行管理。  
  
  
  
##### 2 IT运维监控  
  
监控系统通常会收集：  
  
* CPU  
* 内存  
* 网络  
* 磁盘  
  
例如监控平台 Prometheus 就使用时序数据模型。  
  
  
  
##### 3 金融行情系统  
  
金融市场中每秒产生大量交易数据，例如：  
  
| time | symbol | price | volume |  
| ---- | ------ | ----- | ------ |  
  
典型应用包括：  
  
* 股票行情分析  
* 高频交易  
* 市场监控  
  
  
  
#### 小结  
  
时序数据是现代数据系统中最重要的数据类型之一，其核心特征是以时间为主线的数据序列。  
  
TimescaleDB 在 PostgreSQL 基础上提供了专门的时序数据模型，包括：  
  
| 概念         | 作用        |  
| ---------- | --------- |  
| Hypertable | 逻辑时序表     |  
| Chunk      | 时间分区      |  
| Dimension  | 时间 / 空间维度 |  
  
通过这些机制，TimescaleDB 可以高效管理大规模时序数据，同时保持 PostgreSQL 的 SQL 兼容性。
