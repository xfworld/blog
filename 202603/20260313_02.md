## 背景       
### 11.1.2 多模融合数据库底层架构  
  
多模数据库能够在一个数据库系统中同时支持多种数据模型，其核心并不仅仅是“增加几种数据类型”，而是依赖于底层数据库架构的高度可扩展性与模块化设计。  
  
在数据库系统工程中，多模能力通常通过以下三种方式实现：  
  
1. 多引擎架构（Multi-Engine Architecture）  
2. 统一存储架构（Unified Storage Architecture）  
3. 可扩展插件架构（Extensible Database Architecture）  
  
不同数据库系统会选择不同的技术路线，而 PostgreSQL 采用的主要是可扩展插件架构 + 统一存储架构的方式，这也是其能够成为典型多模数据库平台的重要原因。  
  
   
  
#### 一、多模数据库的三种技术路线  
  
##### 1 多引擎架构  
  
多引擎架构的核心思想是：  
  
不同数据模型由不同存储引擎负责处理。  
  
每个引擎针对某一数据模型进行优化，例如：  
  
| 数据模型 | 存储引擎   |  
| ---- | ------ |  
| 关系数据 | 行存储引擎  |  
| 图数据  | 图存储引擎  |  
| 时序数据 | 时序引擎   |  
| 向量数据 | 向量索引引擎 |  
  
这种架构在部分云数据库中较为常见，例如 Azure Cosmos DB 通过不同 API 支持多种数据模型，包括：  
  
* SQL API  
* MongoDB API  
* Cassandra API  
* Gremlin（图数据库）API  
  
其底层实际上是多个逻辑引擎共同构成的统一服务。  
  
优势  
  
* 针对不同模型进行高度优化  
* 每种引擎可以独立演进  
  
劣势  
  
* 系统复杂度高  
* 数据共享困难  
* 跨模型查询成本较高  
  
   
  
##### 2 统一存储架构  
  
统一存储架构的核心思想是：  
  
所有数据模型共享同一底层存储结构。  
  
例如，一些现代数据库会将不同数据模型统一映射为：  
  
* Key-Value  
* Document  
* Wide Column  
  
再通过不同查询接口进行访问。  
  
典型代表是 ArangoDB，其同时支持：  
  
* 文档模型  
* 图模型  
* Key-Value模型  
  
这些数据在底层使用统一的存储格式管理。  
  
优势  
  
* 数据共享能力强  
* 跨模型查询性能较好  
  
劣势  
  
* 对底层存储要求较高  
* 需要复杂的数据映射机制  
  
   
  
##### 3 可扩展插件架构  
  
可扩展插件架构是一种更加灵活的实现方式，其核心思想是：  
  
数据库核心提供基础能力，复杂数据模型通过插件扩展实现。  
  
这种方式通常包括：  
  
* 新数据类型  
* 新索引类型  
* 新查询语法  
* 新执行算子  
  
在开源数据库领域，PostgreSQL 是这种架构最典型的代表。  
  
通过扩展机制，PostgreSQL可以支持：  
  
| 扩展          | 功能    |  
| ----------- | ----- |  
| PostGIS     | 空间数据库 |  
| TimescaleDB | 时序数据库 |  
| Apache AGE  | 图数据库  |  
| pgvector    | 向量数据库 |  
  
这种架构使 PostgreSQL 能够在保持稳定内核的同时持续扩展能力。  
  
   
  
#### 二、PostgreSQL多模架构的核心组件  
  
PostgreSQL之所以能够成为多模数据库平台，与其高度模块化的架构设计密切相关。  
  
其核心架构可以抽象为以下几个层次：  
  
```  
+--------------------------------+  
| 应用层 (SQL / API / Drivers)   |  
+--------------------------------+  
| 查询处理层 (Parser / Planner) |  
+--------------------------------+  
| 执行引擎 (Executor)           |  
+--------------------------------+  
| 扩展框架 (Extension System)   |  
+--------------------------------+  
| 存储引擎 (Storage Engine)     |  
+--------------------------------+  
| 文件系统 / 操作系统           |  
+--------------------------------+  
```  
  
下面介绍其中几个关键组件。  
  
   
  
##### 1 数据类型系统（Type System）  
  
PostgreSQL拥有非常灵活的数据类型系统，可以支持：  
  
* 基本类型  
* 数组类型  
* 复合类型  
* JSON类型  
* 用户自定义类型  
  
例如：  
  
```sql  
CREATE TYPE vector3d AS (  
    x float,  
    y float,  
    z float  
);  
```  
  
扩展可以通过注册新的数据类型，从而支持新的数据模型。  
  
例如：  
  
* pgvector 新增 `vector` 类型  
* PostGIS 新增 `geometry` 类型  
  
   
  
##### 2 索引扩展框架（Index Access Method）  
  
PostgreSQL允许开发者实现新的索引类型。  
  
数据库内置支持多种索引结构：  
  
| 索引类型   | 适用场景  |  
| ------ | ----- |  
| B-tree | 通用索引  |  
| Hash   | 等值查询  |  
| GIN    | 文档索引(多值列索引)  |  
| GiST   | 空间索引  |  
| BRIN   | 大规模数据(分片索引) |  
  
扩展数据库可以基于这些机制构建新的索引算法。  
  
例如：  
  
* pgvector 实现 HNSW / IVFFLAT 向量索引  
* PostGIS 使用 GiST / SP-GiST 构建空间索引  
  
这种索引扩展能力是实现多模数据库的重要基础。  
  
   
  
##### 3 查询优化器扩展  
  
PostgreSQL的查询优化器允许扩展注册新的执行节点和优化策略。  
  
例如：  
  
图数据库扩展 Apache AGE 在 PostgreSQL 中引入了 Cypher 查询语言，并在执行计划中增加图查询算子。  
  
示例：  
  
```  
MATCH (a)-[:FRIEND]->(b)  
RETURN a,b  
```  
  
查询最终会被转换为 PostgreSQL 的执行计划节点。  
  
这种机制使得不同数据模型仍然可以共享同一个SQL执行框架。  
  
   
  
##### 4 扩展机制（Extension Framework）  
  
PostgreSQL的扩展系统是多模能力的核心。  
  
扩展通常包含：  
  
* SQL对象  
* 新函数  
* 新数据类型  
* 新索引方法  
* C语言实现的执行逻辑  
  
安装扩展非常简单：  
  
```sql  
CREATE EXTENSION postgis;  
```  
  
安装后数据库立即获得新的能力。  
  
例如：  
  
| 扩展          | 新能力     |  
| ----------- | ------- |  
| PostGIS     | 空间计算函数  |  
| TimescaleDB | 时序分区    |  
| pgvector    | 向量相似度计算 |  
  
   
  
#### 三、多模数据库的查询统一能力  
  
多模数据库不仅要支持多种数据模型，还需要支持统一查询能力。  
  
PostgreSQL的SQL语言可以直接访问不同类型数据，例如：  
  
```sql  
SELECT  
    id,  
    embedding <-> query_vector AS distance  
FROM documents  
ORDER BY distance  
LIMIT 10;  
```  
  
在这个示例中：  
  
* `embedding` 为向量数据  
* `<->` 为向量相似度算子  
  
同样，SQL也可以直接查询JSON数据：  
  
```sql  
SELECT data->>'name'  
FROM users  
WHERE data->>'city' = 'Shanghai';  
```  
  
或者执行空间计算：  
  
```sql  
SELECT ST_Distance(a.geom, b.geom)  
FROM locations a, locations b;  
```  
  
这种能力使得不同数据模型能够在同一SQL查询中协同工作。  
  
   
  
#### 四、真实案例：Uber 的PostgreSQL多模应用  
  
全球出行平台 Uber 在其技术架构中广泛使用 PostgreSQL 生态。  
  
在部分数据处理场景中，Uber使用 PostgreSQL + PostGIS 处理：  
  
* 地理位置查询  
* 距离计算  
* 区域匹配  
  
同时结合其他系统处理实时数据。  
  
类似的案例也出现在：  
  
* 物流路径规划  
* 共享出行平台  
* 智慧城市系统  
  
这些系统往往需要同时处理：  
  
* 关系数据  
* 空间数据  
* 时序数据  
  
多模数据库能够显著简化系统架构。  
  
   
  
#### 五、多模数据库架构的发展趋势  
  
随着 AI 和实时分析需求不断增长，多模数据库架构正在向以下方向发展：  
  
1. HTAP融合  
  
   同时支持事务处理与分析处理。  
  
2. AI原生数据库  
  
   内置向量索引与机器学习能力。  
  
3. 统一数据平台  
  
   一个数据库同时支持：  
  
   * OLTP  
   * OLAP  
   * AI  
   * GIS  
   * 图计算  
  
在这一趋势下，PostgreSQL凭借其开放架构和庞大生态，已经成为多模数据库技术的重要基础平台。  
  
   
  
#### 小结  
  
多模数据库的核心不在于简单支持多种数据类型，而在于底层架构的可扩展性与统一查询能力。  
  
PostgreSQL通过以下关键技术实现多模能力：  
  
* 灵活的数据类型系统  
* 可扩展索引框架  
* 强大的扩展机制  
* 统一SQL查询引擎  
  
这些能力使其能够在同一数据库平台中同时支持：  
  
* 图数据  
* 时序数据  
* 向量数据  
* 空间数据
