## 背景  
上一篇文章我提到一个核心观点 : OpenClaw 将被取代, 核心原因是这个仅仅是 AI 入口产品, 没有什么的技术门槛, 但是它可以接入任意模型, 作为顶级大模型厂商, 它为什么不自己打造这样的入口呢? 更好的捆绑用户. 从大模型的动作上也能看到这样的趋势.  
  
[《德说-第462期, 别着急学 OpenClaw，它即将被取代》](../202603/20260324_02.md)  
  
同时, 我真对已经在使用 openClaw 的用户, 给了一个非常明确的建议: 如果你现在已经在养虾, 我建议你要有复利思维, 把精力放在能把龙虾变“聪明”并且可以迁移的能力建设上. 例如各种 skills 的扩充和迭代(让小龙虾可以稳定完成该技能支撑的任务), 它是未来可复用的能力, 并且可以迁移到龙虾的替代品上.  
  
今天就来聊聊如何让 AI 把你日常重复要做的事情, 做成 skill , 例如我每天都要看一下 github daily 热门项目, 并进行解读. 这就可以搞成 skill.  

之前用 opencode 加免费的 kimi 2.5 pro 模型写了一个 [《PostgreSQL 日常巡检 SKILL》](../202602/20260204_01.md)  
  
今天使用 Codex 来写, 参考:       
- [《白嫖的 Codex 真香!》](../202603/20260317_05.md)  
- [《用 Codex 白嫖 ChatGPT 商业版》](../202603/20260316_02.md)  
- https://mp.weixin.qq.com/s/eGWJY8iauSd1Ix_OEHOjsQ  
  
Prompt 如下:  
````  
写一个skill, 命名为每日 github 热门项目解读, 包括全网热门项目 `https://github.com/trending?since=daily&spoken_language_code=` 和中文热门项目 `https://github.com/trending?since=daily&spoken_language_code=zh` .  
这个 skill 的目的是依次解读当日github热门开源项目.  
1、把项目地址存储到一个文件中, 每天一个 markdown 文件, 每行存储对应项目的 URL,  
2、如果同一个 github 项目在多日重复出现, 请不要对这个项目进行重复解读,  
3、解读方法, 阅读项目 readme, 调用 deepwiki mcp 获得对应 github 项目相关知识, 生产完内容后, 再用 deepwiki 检测内容是否正确, 如果有偏差, 请继续修正, 直到内容被 deepwiki 完全认可 .  
4、解读提示词:  
你是资深产品经理, 阅读这个 github 开源项目的 readme : ${这里填入项目readme URL} , 面向用户用浅显易懂的风格写一篇文章, 介绍这个产品, 如果有关键内容的实操最好, 标题要抓住重点且有爆款文章风格.  
内容必须逻辑清晰, 有理有据，有权威数据、案例支撑，不能用个例以偏概全，要有符合第一性原理的前提条件假设来支撑你的观点，如果前提条件崩塌，引出其他观点.  
5、解读后的文章每个项目保存成一个markdown文件, 文件格式: yyyymmdd_xx.md  
6、生成每天的所有项目的解读文章摘要, 保存成一个文件, 文件格式: yyyymmdd_summary.md  
备注: 如果还没有添加 deepwiki mcp, 则先配置.  
````
