## 背景 
  

在 PostgreSQL 的日常运维中，表的 REPACK（或 CLUSTER）操作是整理表物理存储、回收空间、消除表膨胀的常用手段。然而，传统的 REPACK 操作需要获取 `ACCESS EXCLUSIVE` 锁，这意味着在整个操作期间，所有对该表的读写操作都会被阻塞——这对 7x24 运行的业务系统来说是不可接受的。

本篇文章深入解读 PostgreSQL 提交 `28d534e`（Add CONCURRENTLY option to REPACK），该提交为 REPACK 命令引入了 `CONCURRENTLY` 选项，允许在保持表大部分时间可读写的情况下完成表维护工作。其核心实现原理是利用逻辑解码（Logical Decoding）捕获 repack 期间的业务数据变更，并在最终切换文件前将这些变更应用到新表文件中，从而**显著缩短阻塞窗口**。需要注意的是，最终仍需获取 `ACCESS EXCLUSIVE` 锁进行文件切换，不是绝对意义上的"零停机"。

  

**背景痛点**: 传统 REPACK 在整个操作期间持有 `ACCESS EXCLUSIVE` 锁，业务表无法读写。对于大型表，操作可能持续数分钟到数小时，这对高可用业务系统是致命的——期间所有查询和更新都会挂起或报错。

**核心价值**: 通过 `CONCURRENTLY` 选项，REPACK 可以在整个操作期间（除最后的锁升级、第二阶段回放和文件切换窗口外）保持表可读写，大幅减少对业务的影响，同时仍能完成表膨胀清理和数据整理工作。

## 一、原理介绍

### 1.1 传统 REPACK 的锁机制问题

在原有实现中，REPACK（或 CLUSTER）的执行流程是：

```
1. 获取 ACCESS EXCLUSIVE 锁（阻塞所有读写）
2. 创建新表文件
3. 将数据从旧表复制到新表
4. 重建索引
5. 交换 relfilenode
6. 删除旧表
7. 释放锁
```

问题在于：整个操作期间，锁一直持有，表完全不可访问。对于一个 100GB 的表，如果 IO 速度是 500MB/s，仅复制数据就需要约 3.5 分钟，这期间业务完全中断。

### 1.2 CONCURRENTLY 的核心设计思想

`REPACK (CONCURRENTLY)` 的设计目标是：**将 ACCESS EXCLUSIVE 锁的持有时间压缩到最短**。

实现思路借鉴了 `CREATE INDEX CONCURRENTLY` 的经验：
1. 用 `ShareUpdateExclusiveLock`（与 VACUUM 同级的锁）替代 `ACCESS EXCLUSIVE`
2. 使用 MVCC 快照复制当前数据
3. 启动后台 worker 利用逻辑解码捕获复制期间的数据变更
4. 将变更先缓存起来
5. **两阶段变更回放**：
   - **第一阶段**：复制完成后，先应用一次变更（catch-up），然后才请求 `AccessExclusiveLock`
   - **第二阶段**：获取锁后，再次 flush WAL 并应用在「等待锁期间提交的变更」
6. 最后执行文件切换

### 1.3 核心架构与组件

```
┌──────────────────────────────────────────────────────────────────┐
│                        REPACK CONCURRENTLY                        │
│                                                                   │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │                      Leader Process                          │ │
│  │  1. 获取 ShareUpdateExclusiveLock                           │ │
│  │  2. 设置 MVCC 快照                                           │ │
│  │  3. 创建新表并复制数据（此时可读写）                          │ │
│  │  4. 等待 bgworker 完成初始快照导出                            │ │
│  │  5. 第一阶段变更回放（可读写）                                │ │
│  │  6. 升级为 AccessExclusiveLock（等待中阻塞）                  │ │
│  │  7. 第二阶段变更回放（持锁期间执行）                           │ │

│  └─────────────────────────────────────────────────────────────┘ │

│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### 1.4 关键代码解析

#### 1.4.1 锁级别选择

```c
/*
 * 在非并发模式，我们全程持有 AccessExclusiveLock 以避免锁升级风险。
 * 在并发模式，我们像 VACUUM 一样只获取 ShareUpdateExclusiveLock，
 * 只在最后交换文件时才获取 AccessExclusiveLock。
 */
static LOCKMODE
RepackLockLevel(bool concurrent)
{
    if (concurrent)
        return ShareUpdateExclusiveLock;  // 并发模式
    else
        return AccessExclusiveLock;       // 普通模式
}
```

#### 1.4.2 变更类型定义

REPACK 期间捕获的并发变更有四种类型（见 `repack_internal.h`）：

```c
typedef char ConcurrentChangeKind;

#define CHANGE_INSERT       'i'   // 新插入的元组
#define CHANGE_UPDATE_OLD   'u'   // UPDATE 的旧值（用于定位）
#define CHANGE_UPDATE_NEW   'U'   // UPDATE 的新值
#define CHANGE_DELETE       'd'   // 删除的元组
```

#### 1.4.3 逻辑解码输出插件 (pgrepack)

自定义的输出插件 `pgrepack` 负责将 WAL 中的变更解码并写入共享文件：

```c
static void
repack_process_change(LogicalDecodingContext *ctx, ReorderBufferTXN *txn,
                      Relation relation, ReorderBufferChange *change)
{
    switch (change->action)
    {
        case REORDER_BUFFER_CHANGE_INSERT:
            repack_store_change(ctx, relation, CHANGE_INSERT, newtuple);
            break;
        case REORDER_BUFFER_CHANGE_UPDATE:
            if (oldtuple != NULL)
                repack_store_change(ctx, relation, CHANGE_UPDATE_OLD, oldtuple);
            repack_store_change(ctx, relation, CHANGE_UPDATE_NEW, newtuple);
            break;
        case REORDER_BUFFER_CHANGE_DELETE:
            repack_store_change(ctx, relation, CHANGE_DELETE, oldtuple);
            break;
    }
}
```

#### 1.4.4 并行 worker 的启动与通信

Leader 和 worker 通过 `DecodingWorkerShared` 结构通信：

```c
typedef struct DecodingWorkerShared
{
    bool        initialized;           // worker 是否已初始化
    XLogRecPtr  lsn_upto;            // 解码到此 LSN 即可停止

    SharedFileSet sfs;                // 存储变更文件的共享文件集
    int         last_exported;        // 已导出的文件编号
    slock_t     mutex;                // 保护上述字段的锁
    Oid         dbid;                 // 数据库 OID
    Oid         roleid;               // 运行 REPACK 的用户 OID
    Oid         relid;                // 被 repack 的表 OID
    ConditionVariable cv;             // 条件变量，用于同步
    PGPROC     *backend_proc;         // leader 的 proc
    pid_t       backend_pid;          // leader 的 pid
    ProcNumber  backend_proc_number;   // leader 的 proc number
    dsm_segment *dsm_seg;             // 共享内存段
#define REPACK_ERROR_QUEUE_SIZE  16384
    char        error_queue[FLEXIBLE_ARRAY_MEMBER];  // 错误队列
} DecodingWorkerShared;
```

### 1.6 变更捕获与回放机制

pgrepack 插件的输出格式设计巧妙地将变更按类型分类存储：

```
┌─────────────────────────────────────────────────────┐
│ File 0 (快照): WORKER_FILE_SNAPSHOT                │
│   [序列化后的 MVCC 快照（而非表数据）]               │
├─────────────────────────────────────────────────────┤
│ File 1 (后续变更):                                  │
│   i,<tuple1>    -- INSERT                           │
│   u,<oldtuple1> -- UPDATE old value (key)           │
│   U,<newtuple1> -- UPDATE new value                 │
│   d,<oldtuple1> -- DELETE                          │
│   i,<tuple2>    -- INSERT                           │
│   ...                                              │
└─────────────────────────────────────────────────────┘
```

Leader 进程回放变更时：

```c
switch (change_kind)
{
    case CHANGE_INSERT:
        // 直接插入新元组
        break;
    case CHANGE_UPDATE_OLD:
        // 用旧值定位并标记为待更新
        break;
    case CHANGE_UPDATE_NEW:
        // 用之前标记的信息执行更新
        break;
    case CHANGE_DELETE:
        // 定位并删除
        break;
}
```

## 二、应用场景及最佳实践

### 2.1 场景一：7x24 电商订单表维护

**背景**: 某电商平台的 `orders` 表约 200GB，20+ 个索引，每天承受数万订单的插入和状态更新。由于订单数据的重要性，不能长时间锁表，但表已经出现明显膨胀（实际占用 350GB）。

**传统方式的问题**:

```sql
-- 传统 REPACK（会阻塞所有操作约 30 分钟）
REPACK orders;

-- 业务影响：
-- - 所有 SELECT 查询挂起
-- - 所有 INSERT/UPDATE/DELETE 报错或等待
-- - 长事务会显著拉长最终切换阶段的阻塞窗口
```

**使用 CONCURRENTLY 后**:

```sql
-- 在业务低峰期执行（仍然有短暂阻塞，但总时间大幅缩短）
REPACK (CONCURRENTLY) orders;

-- 预期的阶段时间：
-- - 数据复制阶段（可读写）：约 25 分钟
-- - 第一阶段变更回放（可读写）：约 2-5 分钟
-- - 等锁 + 第二阶段变更回放（持锁阻塞）：取决于等锁时间和变更量
-- - 文件切换（持锁阻塞）：通常很短
```

**关键配置**:

```sql
-- 确保有足够的 replication slot（postmaster 级参数，修改后需重启生效）
ALTER SYSTEM SET max_replication_slots = 10;  -- 需重启

-- 调整 maintenance_work_mem 以加速处理
SET maintenance_work_mem = '2GB';

-- 监控变更量（如果变更太多，切换前的应用时间会变长）
```

### 2.2 场景二：分区表逐分区维护

**背景**: 某金融系统按月分区存储交易历史，当前需要清理历史分区的表膨胀。

**处理方式**:

```sql
-- 逐个分区执行 REPACK (CONCURRENTLY)
REPACK (CONCURRENTLY) orders_history_2025_01;
REPACK (CONCURRENTLY) orders_history_2025_02;
-- ...

-- 注意：不能对分区父表执行 CONCURRENTLY
-- ERROR: REPACK (CONCURRENTLY) is not supported for partitioned tables
-- 提示：Consider running the command on individual partitions.
```

**对比效果**:

| 操作 | 阻塞时间 | 对业务影响 |
|------|----------|-----------|
| `VACUUM FULL orders_history_2025_01` | 全程阻塞，约 10-15 分钟 | 高：历史查询全部挂起 |
| `REPACK orders_history_2025_01` | 全程阻塞，约 8-12 分钟 | 高：同上 |
| `REPACK (CONCURRENTLY) orders_history_2025_01` | 仅切换时阻塞，约 10-20 秒 | 低：历史查询仍可执行 |

### 2.3 场景三：不能使用 CONCURRENTLY 的情况

**案例**: 某表 `user_sessions` 用于存储会话数据，15GB，既没有主键也没有合适的 replica identity（不是 TEMP 表，而是一个普通表）。

```sql
REPACK (CONCURRENTLY) user_sessions;

-- 错误：
-- ERROR: cannot repack relation "user_sessions"
-- HINT: Relation "user_sessions" has insufficient replication identity.
```

**解决方案**:

```sql
-- 先为表设置 replica identity（需要管理员权限）
-- 注意：用作 replica identity 的索引须满足以下条件：
--   ① 唯一；② IMMEDIATE（不可 defer）；③ 不能是表达式索引；④ 不能是部分索引；⑤ 所有列均需为 NOT NULL
ALTER TABLE user_sessions REPLICA IDENTITY USING INDEX idx_user_session_id;

-- 如果没有合适索引，先创建（需满足上述全部约束）：
CREATE UNIQUE INDEX CONCURRENTLY idx_user_session_id ON user_sessions(session_id);
ALTER TABLE user_sessions REPLICA IDENTITY USING INDEX idx_user_session_id;

-- 然后再执行 REPACK (CONCURRENTLY)
```

### 2.4 场景四：与其他 DDL 操作的冲突

**问题**: 在 REPACK (CONCURRENTLY) 执行期间，如果其他会话执行了 DDL（如 ALTER TABLE ADD COLUMN），操作会失败。

```sql
-- 会话 A：启动 REPACK (CONCURRENTLY)
SESSION A: REPACK (CONCURRENTLY) users;

-- 会话 B：尝试添加列
SESSION B: ALTER TABLE users ADD COLUMN last_login TIMESTAMP;

-- 可能的结果：
-- - REPACK 可能报错中止
-- - 或者 ALTER 可能等待锁超时
-- - 取决于执行顺序和时机
```

**最佳实践**: 在维护窗口期内，确保没有其他 DDL 操作同时进行。

## 三、限制与已知问题

根据提交中的说明，以下限制需要注意：

### 3.1 Replication Slot 消耗

```
Note: The CONCURRENTLY option requires a replication slot for logical
decoding. Ensure max_replication_slots is set high enough.
```

Replication slot 是稀缺资源，如果当前已接近上限，执行会失败：

```sql
-- 检查当前 slot 使用
SELECT * FROM pg_replication_slots;

-- 检查上限
SHOW max_replication_slots;

-- 如果不够，需要增加（但 max_replication_slots 是 postmaster 级参数，
-- 修改后必须重启实例才能生效）
ALTER SYSTEM SET max_replication_slots = 20;
-- 然后重启 PostgreSQL
```

### 3.2 全局单实例限制

```
Due to the way the historic snapshot is initially set up, only one
REPACK CONCURRENTLY process can be running at any one time on the
whole system.
```

这意味着：
- 同一时间只能有一个表在执行 REPACK CONCURRENTLY
- 尝试同时对两个表执行会报错

### 3.3 锁升级死锁风险

```
There is a danger of deadlocking (and thus abort) due to the
lock upgrade required at the final phase.
```

原因是在最后阶段需要将 `ShareUpdateExclusiveLock` 升级为 `AccessExclusiveLock`，如果其他进程同时持有该表的锁，可能导致死锁。建议在业务低峰期执行。

### 3.4 不支持的场景

| 限制类型 | 说明 |
|----------|------|
| UNLOGGED 表 | 不支持（数据不写入 WAL，无法解码） |
| TEMP 表 | 不支持（不是永久表） |
| 分区表 | 不支持（必须对单个分区执行） |
| 无主键/REPLICA IDENTITY | 不支持（无法追踪变更） |
| 系统目录表 | 不支持 |
| TOAST 表 | 不支持 |
| 事务块内执行 | 不允许 |

## 四、监控与故障排查

### 4.1 监控 REPACK 进度

```sql
-- 查看当前正在执行的 REPACK
SELECT pid, query, state, query_start, wait_event_type, wait_event
FROM pg_stat_activity
WHERE query LIKE '%REPACK%';

-- 查看 replication slot 状态（REPACK CONCURRENTLY 会创建临时 slot）
SELECT slot_name, plugin, active, restart_lsn
FROM pg_replication_slots
WHERE slot_name LIKE 'repack%';

-- 注意：REPACK 完成后 slot 会自动清理
```

### 4.2 查看 REPACK 进度

REPACK (CONCURRENTLY) 的进度首选通过 `pg_stat_progress_repack` 视图监控，同时也可结合 `pg_stat_activity` 和 `pg_replication_slots` 进行辅助观察：

```sql
-- 1. 查看当前 REPACK 进度（首选视图）
SELECT * FROM pg_stat_progress_repack;

-- 2. 查看当前 REPACK 状态（通过 pg_stat_activity）
SELECT pid, state, query, query_start, wait_event_type, wait_event
FROM pg_stat_activity
WHERE query LIKE '%REPACK%';

-- 3. 查看 replication slot 状态（REPACK CONCURRENTLY 会创建临时 slot）
SELECT slot_name, plugin, active, restart_lsn, confirmed_flush_lsn
FROM pg_replication_slots
WHERE slot_name LIKE 'repack%';

-- 4. 查看是否有排他锁等待（文件切换阶段）
SELECT pid, relation::regclass, mode, granted
FROM pg_locks
WHERE relation = 'orders'::regclass;

-- 注意：REPACK 完成后 slot 会自动清理
```

### 4.3 常见错误处理

**错误 1: replication slot 不足**

```
ERROR: could not create replication slot "repack_xxxxx": maximum number of
replication slots exceeded
```

解决：增加 `max_replication_slots` 或等待其他 slot 释放。

**错误 2: 无主键/identity**

```
ERROR: cannot repack relation "xxx"
HINT: Relation "xxx" has insufficient replication identity.
```

解决：先为表设置 `REPLICA IDENTITY`。

**错误 3: 锁升级超时**

```
ERROR: could not obtain lock on relation "xxx" during repack
```

解决：在业务低峰期重试，或考虑使用非 CONCURRENTLY 模式。

if ((params.options & CLUOPT_CONCURRENT) != 0)
{
    /*
     * 确保不在事务块中执行，因为 repack_setup_logical_decoding()
     * 可能无限等待我们的 XID 完成（可能导致死锁检测无法识别的情况）

#### 5.2.2 约束检查

```c
static void
check_concurrent_repack_requirements(Relation rel, Oid *ident_idx_p)
{
    // 不支持系统目录表（数据变更不会被逻辑解码）
    if (IsCatalogRelation(rel))
        ereport(ERROR, ...);

    // 不支持 TOAST 表
    if (IsToastRelation(rel))
        ereport(ERROR, ...);

    // 只支持永久表
    if (rel->rd_rel->relpersistence != RELPERSISTENCE_PERMANENT)
        ereport(ERROR, ...);

    // REPLICA IDENTITY 为 NOTHING 时，WAL 不包含旧元组，无法追踪变更
    if (rel->rd_rel->relreplident == REPLICA_IDENTITY_NOTHING)
        ereport(ERROR, ...);
}
```

#### 5.2.3 数据复制（支持 MVCC 快照）

```c
copy_table_data(NewHeap, OldHeap, index, snapshot, verbose,
                &swap_toast_by_content, &frozenXid, &cutoffMulti);
```

当 `snapshot` 不为 NULL 时，复制操作会使用 MVCC 快照，保证复制开始时刻的数据一致性视图。

## 六、小结和思考

### 6.1 核心价值总结

`REPACK CONCURRENTLY` 为 PostgreSQL 的表维护能力带来了质的飞跃：

| 维度 | 传统 REPACK | REPACK CONCURRENTLY |
|------|-------------|---------------------|
| 锁级别 | 全程 AccessExclusive | 仅切换时 AccessExclusive |
| 表可访问性 | 完全不可读写 | 大部分时间可读写 |
| 业务影响 | 高（长时间阻塞） | 低（仅短暂阻塞） |
| 实现复杂度 | 简单 | 复杂（依赖逻辑解码） |
| 资源消耗 | 低 | 需要 replication slot |

### 6.2 最佳实践建议

1. **优先使用 CONCURRENTLY**: 对于生产环境的大表，始终使用 `CONCURRENTLY` 选项
2. **确保有主键或 REPLICA IDENTITY**: 这是并发 repack 的前提条件
3. **保留足够的 replication slot**: `max_replication_slots` 应至少比实际使用多 2-3 个
4. **在业务低峰期执行**: 虽然大部分时间可读写，但最终切换时仍需排他锁，且持锁期间可能还需再做一轮变更回放（处理等锁期间的新提交）
5. **监控变更量**: 如果期间变更太多，最终阻塞窗口会进一步拉长（等锁 + 回放）

### 6.3 与类似功能的对比

| 功能 | 锁级别 | 并发读 | 并发写 | 依赖 |
|------|--------|--------|--------|------|
| VACUUM FULL | AccessExclusive | No | No | 无 |
| CLUSTER/REPACK | AccessExclusive | No | No | 无 |
| REPACK (CONCURRENTLY) | AccessExclusive（仅切换时） | Yes | Yes | Replication slot |

### 6.4 未来改进方向

根据提交中提到的 "loose ends"，以下改进值得期待：

1. **解决全局单实例限制**: 让多个表的 REPACK CONCURRENTLY 可以并行执行
2. **优化锁升级策略**: 减少或消除最终阶段的死锁风险
3. **更智能的变更管理**: 当变更量过大时，自动分流或警告

### 6.5 升级路径

对于正在使用 PostgreSQL 的 DBA：

```sql
-- 1. 升级后检查是否支持 CONCURRENTLY
SELECT 1;  -- 应该正常返回

-- 2. 检查 max_replication_slots 是否足够
SHOW max_replication_slots;

-- 3. 为需要维护的大表确保有 replica identity index，或存在主键可作为 identity index
--    REPLICA IDENTITY NOTHING 不支持，FULL 目前也未实现
SELECT relname, relreplident
FROM pg_class
WHERE relkind = 'r'
  AND relnamespace = 'public'::regnamespace;

-- 4. 尝试对测试表执行 REPACK (CONCURRENTLY)
REPACK (CONCURRENTLY) test_table;

-- 5. 在生产环境推广
```
