## 背景 

本期 PostgreSQL 提交包含两个重要的修复和增强，虽然看起来是"小"改动，但对于 PostgreSQL 的稳定运行和可靠性有着不可忽视的价值。

第一个提交 `93dc1ace` 修复了 slotsync worker 遗漏释放继承的 PostmasterContext 的问题——该 worker 在启动时未清理 postmaster 的工作内存上下文，导致该上下文在 slotsync 进程生命周期内持续保留而未被正确释放。第二个提交 `ca2b544` 为 `log_lock_waits` 功能添加了完整的 TAP 测试覆盖，确保了锁等待日志的正确性和一致性。

  

**背景痛点**:

- **slotsync 遗漏释放 PostmasterContext**: 其他后台 worker（如 autovacuum、syslogger）在启动时会正确释放 `PostmasterContext`，但 slotsync worker 遗漏了这一步。继承的 postmaster 工作内存上下文未被及时释放。
- **log_lock_waits 缺乏测试**: `log_lock_waits` 是一个重要的诊断参数，但之前没有专门的 TAP 测试来验证其行为正确性。

**核心价值**:

- 修复了 slotsync worker 遗漏释放继承的 PostmasterContext 的问题，提升复制槽管理的稳定性
- 通过完整的测试覆盖，确保 `log_lock_waits` 日志行为的正确性

## 一、原理介绍

### 1.1 PostmasterContext 的生命周期

PostgreSQL 的 postmaster 进程在启动时会创建一个 `PostmasterContext`（工作内存上下文），用于分配启动数据。当子进程（如 autovacuum worker、slotsync worker）被 fork 出来时，它们会继承这个内存上下文。

**问题所在**: 子进程实际上不需要 postmaster 的工作内存上下文，因为：
1. 子进程有自己独立的地址空间
2. postmaster 分配的启动数据在子进程中已经无用

**正确做法**: 子进程应该在启动后立即释放 `PostmasterContext`，让这部分内存能够被正确回收。

### 1.2 slotsync worker 的角色

slotsync worker 是 PostgreSQL 逻辑复制的一个重要组件，负责把 primary 上启用 failover 的逻辑复制槽同步到 standby。它的主要职责包括：

1. 从 primary 获取逻辑 failover 槽的状态信息
2. 在 standby 上创建并定期同步这些逻辑槽
3. 处理复制槽的持久化

```c
/*
 * slotsync worker 的主要入口函数
 */
ReplSlotSyncWorkerMain(const void *startup_data, size_t startup_data_len)
{
    // 在修复前，这里没有释放 PostmasterContext
    // 导致内存泄漏

    init_ps_display(NULL);
    Assert(GetProcessingMode() == InitProcessing);
    // ... 后续工作
}
```

### 1.3 其他 worker 的正确做法

对比其他 worker 的实现，可以看到它们都正确地释放了 `PostmasterContext`：

**autovacuum.c**:
```c
/* autovacuum worker */
Assert(startup_data_len == 0);

/* Release postmaster's working memory context */
if (PostmasterContext)
{
    MemoryContextDelete(PostmasterContext);
    PostmasterContext = NULL;
}

init_ps_display(NULL);
```

**syslogger.c**:
```c
/*
 * Now that we're done reading the startup data, release postmaster's
 * working memory context.
 */
if (PostmasterContext)
{
    MemoryContextDelete(PostmasterContext);
    PostmasterContext = NULL;
}
```

### 1.4 log_lock_waits 的工作机制

`log_lock_waits` 参数控制当会话等待锁的时间超过 `deadlock_timeout` 时是否记录日志。这是一个重要的诊断工具，帮助 DBA 识别锁竞争问题。

**日志消息格式**:

1. **等待开始时**（proc.c:1648）:
   ```
   process %d still waiting for %s on %s after %ld.%03d ms
   ```
   示例（`DescribeLockTag` 输出的是 OID 而非表名）：
   ```
   process 12345 still waiting for AccessExclusiveLock on relation 1234 of database 5678 after 1001.234 ms
   ```

2. **等待结束时**（proc.c:1658）:
   ```
   process %d acquired %s on %s after %ld.%03d ms
   ```
   示例：
   ```
   process 12345 acquired AccessExclusiveLock on relation 1234 of database 5678 after 1001.456 ms
   ```

### 1.5 TAP 测试的设计

TAP（Test Anything Protocol）测试脚本验证了以下行为：

```perl
# 检查等待开始时的日志
$node->wait_for_log(qr/still waiting for AccessExclusiveLock on relation/,
    $log_offset);

# 唤醒等待的后端并确认它被唤醒
$node->safe_psql('postgres',
    q[SELECT pg_log_backend_memory_contexts(pid)
      FROM pg_locks WHERE locktype = 'relation' AND ...]);

# 确认即使有 wakeup，"still waiting" 消息仍然只记录一次
my @still_waiting = ($log_contents =~ /still waiting for/g);
is(scalar @still_waiting, 1,
    "still waiting logged exactly once despite wakeups");
```

## 二、应用场景及最佳实践

### 2.1 场景一：监控长事务导致的锁等待

**背景**: 某银行系统中有批量处理任务，可能长时间持有锁，导致其他查询阻塞。

**配置**：

```sql
-- 某些 CI/测试配置可能默认开启；生产默认通常为 off（如需开启，可显式设置为 on）
ALTER SYSTEM SET log_lock_waits = on;
ALTER SYSTEM SET deadlock_timeout = '1s';  -- 1秒后记录
SELECT pg_reload_conf();

-- 设置日志级别
ALTER SYSTEM SET log_min_messages = 'log';
SELECT pg_reload_conf();
```

**观察日志**：

```
2026-04-07 10:30:45.123 +08 [00001] LOG:  process 12345 still waiting for AccessExclusiveLock on relation 1234 of database 16384 after 1234.567 ms
2026-04-07 10:30:45.124 +08 [00001] LOG:  process 12345 acquired AccessExclusiveLock on relation 1234 of database 16384 after 1235.678 ms
```

### 2.2 场景二：诊断复制槽同步问题

**背景**: 某主从复制环境，从库的复制延迟增加，怀疑是 slotsync worker 的问题。

**排查步骤**：

1. 检查 slotsync worker 是否运行：
   ```sql
   -- 本地 slotsync worker 的稳定标识是 backend_type
   SELECT pid, state, backend_start
   FROM pg_stat_activity
   WHERE backend_type = 'slotsync worker';

   -- 主库侧看到的对应连接（对端设置的 application_name）
   -- 可能为 "slotsync worker" 或 "<cluster_name>_slotsync worker"
   ```

2. 确认修复后的内存行为：
   - 通过源码确认修复位置在 `slotsync.c` 第 1497-1502 行
   - 或通过内存上下文 dump/采样工具（如 `pg_log_backend_memory_contexts`）验证 slotsync worker 的内存上下文是否正确释放
   - 在修复前，slotsync worker 继承的 postmaster 工作内存上下文未被及时释放
   - 在修复后，这部分内存会被立即释放

### 2.3 场景三：验证锁统计的重置行为

**问题**: DBA 想知道 `pg_stat_reset_shared('lock')` 是否同时影响 `log_lock_waits` 的行为。

**测试验证**（修复后的测试确保）：
- `pg_stat_reset_shared('lock')` 只重置统计计数器
- `log_lock_waits` 只影响日志，不影响锁统计计数器

```perl
# 测试确认 log_lock_waits 消息被正确发出
$node->wait_for_log(qr/acquired AccessExclusiveLock on relation/,
    $log_offset);

# 确认统计重置后 log_lock_waits 仍然工作
$node->wait_for_log(qr/still waiting for AccessExclusiveLock on relation/,
    $log_offset);
```

## 三、代码层面的关键变更

### 3.1 slotsync.c 的修复

```c
/*
 * ReplSlotSyncWorkerMain - slotsync worker 的主入口
 */
void
ReplSlotSyncWorkerMain(const void *startup_data, size_t startup_data_len)
{
    // ... 变量声明 ...

    Assert(startup_data_len == 0);

    /* 新增：释放 postmaster 的工作内存上下文 */
    if (PostmasterContext)
    {
        MemoryContextDelete(PostmasterContext);
        PostmasterContext = NULL;
    }

    init_ps_display(NULL);
    // ... 后续初始化 ...
}
```

**修复位置**: `src/backend/replication/logical/slotsync.c` 第 1497-1502 行（Assert 在第 1495 行）

### 3.2 TAP 测试的增强

测试文件：`src/test/modules/test_misc/t/011_lock_stats.pl`

**新增的测试场景**:

| 测试场景 | 验证内容 |
|----------|----------|
| Relation lock 等待 | 确认 "still waiting" 和 "acquired" 消息都被记录 |
| TransactionID lock 等待 | 确认 "still waiting" 和 "acquired" 消息都被记录 |
| Advisory lock 等待 | 确认 "still waiting" 和 "acquired" 消息都被记录 |
| Wakeup 后的行为 | 确认即使 backend 被信号唤醒，"still waiting" 仍只记录一次 |
| log_lock_waits = off | 确认关闭时不产生日志 |

**关键测试逻辑**:

```perl
# 1. 触发锁等待
$s1->query_safe("LOCK TABLE test_stat_tab");
$s2->query_safe("LOCK TABLE test_stat_tab");  # 等待

# 2. 检查等待开始日志
$node->wait_for_log(qr/still waiting for AccessExclusiveLock on relation/,
    $log_offset);

# 3. 唤醒等待的后端（通过 pg_log_backend_memory_contexts）
$node->safe_psql('postgres',
    q[SELECT pg_log_backend_memory_contexts(pid) FROM pg_locks ...]);

# 4. 验证唤醒后仍只记录一次 "still waiting"
my @still_waiting = ($log_contents =~ /still waiting for/g);
is(scalar @still_waiting, 1,
    "still waiting logged exactly once despite wakeups");
```

## 四、修复的影响分析

### 4.1 slotsync 遗漏释放 PostmasterContext 的影响

**修复前**:
```
slotsync worker 启动
    │
    ├──► 继承 PostmasterContext（包含 postmaster 分配的启动数据）
    │
    ├──► 继承的 postmaster 工作内存上下文未被及时释放
    │
    └──► 长时间运行可能导致异常内存占用
```

**修复后**:
```
slotsync worker 启动
    │
    ├──► 立即释放 PostmasterContext
    │
    ├──► 创建自己的独立内存上下文
    │
    └──► 正常工作，不影响 postmaster 内存
```

### 4.2 测试覆盖的价值

| 方面 | 修复前的状态 | 修复后的改进 |
|------|-------------|--------------|
| log_lock_waits 测试 | 只有模糊的集成测试 | 完整的 TAP 测试 |
| 边界情况 | 可能有遗漏 | 明确覆盖 wakeup 场景 |
| 回归风险 | 高（没有专门测试） | 低（有测试保护） |

## 五、小结和思考

### 5.1 核心价值总结

这两个提交虽然代码量不大，但都是重要的质量改进：

| 提交 | 类型 | 核心价值 |
|------|------|----------|
| `93dc1ace` | Bug 修复 | 解决 slotsync worker 遗漏释放 PostmasterContext，防止潜在内存问题 |
| `ca2b544` | 测试增强 | 为 log_lock_waits 提供完整测试覆盖，防止回归 |

### 5.2 对 DBA 的实际意义

1. **slotsync 修复**: 对于运行逻辑复制的系统，这意味着更稳定的内存使用和更可预测的性能
2. **log_lock_waits 测试**: 确保锁等待诊断功能的正确性，让 DBA 在排查问题时能依赖这些日志

### 5.3 验证修复是否生效

**检查 slotsync worker 的内存上下文**:

```sql
-- 通过 pg_stat_activity 找到 slotsync worker 的 PID
SELECT pid, backend_type, application_name, state, backend_start
FROM pg_stat_activity
WHERE backend_type = 'slotsync worker';

-- 请求该 worker 输出其内存上下文日志（贴近源码语义的验证方式）
-- 日志中应能看到 PostmasterContext 已被释放，不再包含 postmaster 继承的内存上下文
SELECT pg_log_backend_memory_contexts(pid)
FROM pg_stat_activity
WHERE backend_type = 'slotsync worker';
```

**辅助手段：进程级内存观察**（作为补充，不能直接证明 PostmasterContext 释放）:

```bash
# 查看 slotsync 进程的内存使用（RSS 只能辅助观察，不能直接证明内存上下文释放）
ps aux | grep postgres | grep slotsync

# 监控内存变化趋势（结合内存上下文日志一起看）
watch -n 1 'ps -o pid,rss,vsz -p $(pgrep -f slotsync)'
```

**验证 log_lock_waits 日志**:

```sql
-- 开启 log_lock_waits
SET log_lock_waits = on;
SET deadlock_timeout = '1s';

-- 在两个会话中测试锁等待
-- 会话1:
BEGIN;
LOCK TABLE test_table IN ACCESS EXCLUSIVE MODE;

-- 会话2:
SET log_lock_waits = on;
LOCK TABLE test_table IN ACCESS EXCLUSIVE MODE;  -- 会等待

-- 检查日志
-- 应该看到 "still waiting" 和 "acquired" 消息
```

### 5.4 最佳实践建议

1. **监控后台 worker 的内存使用**:
   ```sql
   -- 查看 slotsync worker 的 PID 和状态
   SELECT pid, backend_type, application_name, state, backend_start
   FROM pg_stat_activity
   WHERE backend_type = 'slotsync worker';

   -- 若需观察主库侧看到的连接名（即 slotsync 连主库时的 application_name）
   -- 在主库上查询：application_name LIKE '%slotsync%'

   -- 查看具体内存上下文（通过 pg_log_backend_memory_contexts 看日志）
   -- SELECT pg_log_backend_memory_contexts(pid) FROM pg_stat_activity ...;
   ```

2. **配置合理的 deadlock_timeout**:
   ```sql
   -- 对于 OLTP 系统，建议设置较短
   ALTER SYSTEM SET deadlock_timeout = '500ms';

   -- 对于批处理系统，可以设置较长
   ALTER SYSTEM SET deadlock_timeout = '5s';
   ```

3. **利用 log_lock_waits 诊断锁问题**:
   - 当发现查询响应变慢时，检查是否有 "still waiting" 日志
   - 分析哪些表/事务是锁竞争的源头

### 5.5 未来展望

1. **自动化锁分析工具**: 基于 log_lock_waits 日志自动生成锁竞争报告
2. **slotsync 监控增强**: 提供更详细的 slotsync worker 性能指标
3. **扩展 TAP 测试覆盖**: 为更多后台进程添加类似的内存管理测试
