## 背景 

本文解读 PostgreSQL 19 中引入的两个 OAuth 认证相关 commit，分别针对**调试机制**和 **HBA 精细化配置**进行了重要改进。第一个 commit 将 `PGOAUTHDEBUG` 环境变量从简单的"全开"模式拆分为细粒度的选项列表，让开发者可以按需启用特定调试功能而无需承担暴露敏感信息的风险。第二个 commit 为 OAuth 验证器模块引入了在 `pg_hba.conf` 中注册和使用自定义选项的能力，使得多租户场景下单个验证器能够为不同用户组提供差异化的访问控制策略。

  

**背景痛点**：在 PostgreSQL 引入 OAuth 认证机制后，社区发现两个核心问题制约着其在生产环境中的应用：一是调试 OAuth 流程时只能"全有或全无"地启用所有调试功能，其中包含会打印敏感令牌信息的 `trace` 选项，开发者往往只想要一个轻量的 `call-count` 统计却不得不暴露所有 secrets；二是 OAuth 验证器的配置只能通过全局 GUC 参数设定，无法针对特定 IP 范围、用户或数据库在 HBA 层面进行细粒度控制，这在多租户数据库环境中是刚性需求。

**核心价值**：这两个 commit 大幅提升了 PostgreSQL OAuth 认证的可用性和安全性。前者让调试OAuth流程时可以根据实际需求选择性启用特定功能，避免误操作导致的敏感信息泄露；后者首次在 HBA 层面为 OAuth 验证器提供了per-entry配置能力，使得多租户场景下的差异化访问控制成为可能，同时保持了对未来更通用化配置方案的兼容性预留。

## 一、原理介绍

### 1.1 PGOAUTHDEBUG 调试机制的演进

#### 旧方案的问题

在原有设计中，`PGOAUTHDEBUG=UNSAFE` 是一个单一开关，启用后会同时开启所有调试功能：

```bash
# 旧的用法：启用所有调试功能（包括危险的 HTTP 明文传输和令牌日志）
PGOAUTHDEBUG=UNSAFE
```

这种设计的缺陷在于：

| 调试选项 | 风险等级 | 开发者实际需求频率 |
|---------|---------|------------------|
| HTTP 明文传输 | 高危 | 仅本地开发 |
| HTTP 流量日志（包含 bearer tokens） | 高危 | 仅调试协议细节 |
| 零秒重试间隔 | 中危 | 仅单元测试 |
| 调用次数统计 | 安全 | 日常调试 |
| 插件加载错误 | 安全 | 排错时 |

最让人困扰的是 `trace` 选项——它会将 HTTP 流量（包括 `Authorization: Bearer xxx` 等敏感头信息）直接打印到 stderr，完全掩盖设备授权流程的交互提示。

#### 新方案：逗号分隔的选项列表

新设计将 `PGOAUTHDEBUG` 拆分为支持逗号分隔的选项列表：

```bash
# 仅启用安全的调用统计
PGOAUTHDEBUG=call-count

# 同时启用安全的调用统计和插件错误（混合模式）
PGOAUTHDEBUG=UNSAFE:http,call-count

# 传统的全开模式（保持向后兼容）
PGOAUTHDEBUG=UNSAFE
```

**关键安全机制**：不安全选项（`http`、`trace`、`dos-endpoint`）必须以 `UNSAFE:` 前缀开头才能生效。如果直接写 `PGOAUTHDEBUG=trace`，系统会输出警告并忽略该选项：

```
WARNING: PGOAUTHDEBUG option "trace" is unsafe (ignored)
```

这一设计通过在环境变量值前面增加明确的"危险意图"标记，来打断开发者下意识输入不安全选项的惯性操作。

#### 核心实现：oauth-debug.h

新的解析逻辑封装在 `src/interfaces/libpq/oauth-debug.h` 中（这是 libpq 和 libpq-oauth 共享的私有头文件）：

```c
/* 不安全选项（需要 UNSAFE: 前缀） */
#define OAUTHDEBUG_UNSAFE_HTTP          (1<<0)   /* 允许 HTTP 明文 */
#define OAUTHDEBUG_UNSAFE_TRACE         (1<<1)   /* 打印 HTTP 流量 */
#define OAUTHDEBUG_UNSAFE_DOS_ENDPOINT  (1<<2)   /* 零秒重试 */

/* 安全选项（可直接使用） */
#define OAUTHDEBUG_CALL_COUNT           (1<<16)  /* 打印调用统计 */
#define OAUTHDEBUG_PLUGIN_ERRORS        (1<<17)  /* 打印插件加载错误 */

/* 掩码用于区分安全/不安全选项 */
#define OAUTHDEBUG_UNSAFE_MASK          ((uint32) 0x0000FFFF)
```

`oauth_parse_debug_flags()` 函数解析环境变量并返回标志位掩码：

```c
static uint32
oauth_parse_debug_flags(void)
{
    uint32 flags = 0;
    const char *env = getenv("PGOAUTHDEBUG");
    bool unsafe_allowed = false;

    if (strcmp(env, "UNSAFE") == 0)
        return OAUTHDEBUG_LEGACY_UNSAFE;  // ~0，即所有位

    if (strncmp(env, "UNSAFE:", 7) == 0) {
        unsafe_allowed = true;
        env += 7;
    }

    // 按逗号分隔解析各选项...
    for (option = strtok_r(options_str, ",", &saveptr); ...) {
        if (!unsafe_allowed && (flag & OAUTHDEBUG_UNSAFE_MASK)) {
            fprintf(stderr, "WARNING: PGOAUTHDEBUG option \"%s\" is unsafe (ignored)\n", option);
            continue;
        }
        flags |= flag;
    }
    return flags;
}
```

在 `oauth-curl.c` 中，原本的 `bool debugging` 字段被替换为 `uint32 debug_flags`，使用位运算检查各选项：

```c
// 启用 HTTP 明文传输（仅在明确标记为 UNSAFE 时）
if (actx->debug_flags & OAUTHDEBUG_UNSAFE_HTTP) {
    protos = CURLPROTO_HTTPS | CURLPROTO_HTTP;  // 允许 HTTP
}

// 启用流量跟踪（会打印敏感信息）
if (actx->debug_flags & OAUTHDEBUG_UNSAFE_TRACE) {
    // 设置 CURLOPT_DEBUGFUNCTION 回调
}

// 安全选项：打印调用次数
if (actx->debug_flags & OAUTHDEBUG_CALL_COUNT) {
    actx->dbg_num_calls++;
}
```

### 1.2 OAuth 验证器自定义 HBA 选项机制

#### 问题的本质

PostgreSQL 的 OAuth 验证器在 PG18 中引入了通过 `postgresql.conf` 中的 GUC 参数进行全局配置的能力。然而，这种设计在多租户场景下暴露了根本性的局限：

```ini
# postgresql.conf —— 全局配置，所有 OAuth 连接共用同一套参数
oauth_validator.authn_id = '固定的用户标识'
oauth_validator.error_detail = '详细错误模式'
```

假设你运营一个多租户数据库服务：
- tenant_A 需要使用 `authn_id = 'tenant-a-id'`，且需要详细的错误信息
- tenant_B 需要使用 `authn_id = 'tenant-b-id'`，但错误信息应该保密

在原有设计下，这两个需求无法在单集群内同时满足——GUC 是全局性的，无法针对特定 HBA 条目差异化配置。

#### 解决方案：per-HBA 选项注册

PG19 为 OAuth 验证器引入了两个新 API，允许验证器在 `pg_hba.conf` 中声明和使用自定义选项：

```c
// 注册可在 HBA 中使用的选项（在 startup_cb 中调用）
void RegisterOAuthHBAOptions(ValidatorModuleState *state, int num, const char *opts[]);

// 获取用户在 HBA 中设置的值（在 validate_cb/shutdown_cb 中调用）
const char *GetOAuthHBAOption(const ValidatorModuleState *state, const char *optname);
```

#### 验证器侧实现

以测试模块 `oauth_validator` 为例，它注册了两个 HBA 选项：

```c
static const char *hba_opts[] = {
    "authn_id",   // 覆盖默认的身份标识
    "log",        // 记录任意字符串到日志
};

static void
validator_startup(ValidatorModuleState *state)
{
    // 在启动回调中注册选项
    RegisterOAuthHBAOptions(state, lengthof(hba_opts), hba_opts);
}

static bool
validate_token(const ValidatorModuleState *state, ...)
{
    // 在验证回调中获取 HBA 选项值
    if (GetOAuthHBAOption(state, "log"))
        elog(LOG, "%s", GetOAuthHBAOption(state, "log"));

    if (authn_id)
        res->authn_id = pstrdup(authn_id);
    else if (GetOAuthHBAOption(state, "authn_id"))
        res->authn_id = pstrdup(GetOAuthHBAOption(state, "authn_id"));
    else
        res->authn_id = pstrdup(role);

    return true;
}
```

#### pg_hba.conf 中的使用

用户现在可以在 HBA 条目中直接设置验证器选项：

```ini
# pg_hba.conf

# 为 admin 用户配置 OAuth：使用特定的 authn_id 和自定义日志消息
hostssl  postgres  admin  0.0.0.0/0  \
    oauth issuer=https://admin.example.com \
    scope="pg-admin openid email" \
    map=oauth-email \
    validator.authn_id="admin-access" \
    validator.log="admin login attempt"

# 为普通用户配置 OAuth：使用不同的 authn_id
hostssl  postgres  all  0.0.0.0/0  \
    oauth issuer=https://www.example.com \
    scope="pg-user openid email" \
    map=oauth-email \
    validator.authn_id="user-access"
```

#### 后端解析流程

HBA 解析逻辑在 `src/backend/libpq/hba.c` 的 `parse_hba_auth_opt()` 函数中扩展：

```c
else if (strncmp(name, "validator.", strlen("validator.")) == 0)
{
    const char *key = name + strlen("validator.");

    REQUIRE_AUTH_OPTION(uaOAuth, name, "oauth");

    // 验证选项名称的合法性（仅 ASCII 字母数字、下划线、连字符）
    if (!valid_oauth_hba_option_name(key)) {
        ereport(elevel, ...);
        return false;
    }

    // 将选项存入 HbaLine 结构体
    hbaline->oauth_opt_keys = lappend(hbaline->oauth_opt_keys, pstrdup(key));
    hbaline->oauth_opt_vals = lappend(hbaline->oauth_opt_vals, pstrdup(val));
}
```

存储结构在 `HbaLine` 中新增了两个 List 指针：

```c
typedef struct HbaLine {
    // ... 其他字段 ...
    char    *oauth_scope;
    char    *oauth_validator;
    bool     oauth_skip_usermap;
    List    *oauth_opt_keys;   // 新增：选项名称列表
    List    *oauth_opt_vals;   // 新增：选项值列表
} HbaLine;
```

#### 连接时的选项校验

由于验证器模块不需要在 postmaster 启动时加载（可以使用 `session_preload_libraries` 而非 `shared_preload_libraries`），选项名称的合法性无法在 `pg_ctl reload` 时检查。因此，校验被延迟到 OAuth 认证流程的早期（`oauth_exchange()` 函数中）：

```c
static bool
check_validator_hba_options(Port *port, const char **logdetail)
{
    HbaLine *hba = port->hba;

    foreach_ptr(char, key, hba->oauth_opt_keys) {
        bool found = false;

        // 在验证器注册的选项列表中查找
        foreach_ptr(char, optname, ValidatorOptions) {
            if (strcmp(key, optname) == 0) {
                found = true;
                break;
            }
        }

        if (!found) {
            // 未注册的选项 —— 连接将被拒绝
            ereport(WARNING, ...);
            return false;
        }
    }

    ValidatorOptionsChecked = true;  // 允许 GetOAuthHBAOption() 现在返回有效值
    return true;
}
```

这意味着：如果用户在 HBA 中设置了一个验证器未注册的选项，`pg_ctl reload` 会成功，但实际建立连接时会失败并伴随明确的错误信息：

```
WARNING: unrecognized authentication option name: "validator.bad"
DETAIL: The installed validator module ("my_validator") did not define an option named "bad".
HINT: All OAuth connections matching this line will fail. Correct the option and reload the server configuration.
```

## 二、应用场景及最佳实践

### 2.1 场景一：本地开发时的精细化调试

#### 改进前

开发者在调试 OAuth 设备授权流程时，为了查看调用次数统计，必须启用全部不安全选项：

```bash
# 改进前：虽然只想要 call-count，但必须开启所有危险选项
$ PGOAUTHDEBUG=UNSAFE psql "host=... sslmode=require"

# stderr 输出：被大量 HTTP 流量淹没，难以找到关键信息
==摆渡车信息==
* Rebuilt URL to: https://...
* SSLv3, TLS alert, Client write key (arg length):   22
Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...
* Connection timeout...
```

#### 改进后

开发者可以精确选择所需的调试功能：

```bash
# 改进后：仅启用安全的调用统计，不暴露任何敏感信息
$ PGOAUTHDEBUG=call-count psql "host=... sslmode=require"

# stderr 输出：简洁明了
OAuth flow completed in 3 calls

# 调试插件加载问题时
$ PGOUTHDEBUG=plugin-errors psql "host=... sslmode=require"
failed dlopen for libpq-oauth: library not found

# 需要同时查看协议细节时，明确启用 trace
$ PGOAUTHDEBUG=UNSAFE:trace,call-count psql "host=... sslmode=require"
```

**最佳实践**：
- 日常开发调试使用 `PGOAUTHDEBUG=call-count`
- 排查插件加载问题时使用 `PGOAUTHDEBUG=plugin-errors`
- 仅在需要分析 HTTP 协议细节时使用 `PGOAUTHDEBUG=UNSAFE:trace`
- 生产故障排查时，优先使用 `call-count` 和 `plugin-errors`，避免使用 `trace`

### 2.2 场景二：多租户数据库服务的差异化访问控制

#### 背景

某 SaaS 提供商使用 PostgreSQL 为多个客户提供数据库服务。每个客户使用不同的 OAuth 发行方（issuer），需要：

1. 为企业客户配置更严格的身份映射和审计日志
2. 为普通客户提供简化的认证流程
3. 不同客户群体使用不同的 OAuth 提供商

#### 改进前

```ini
# postgresql.conf
oauth_validator.authn_id = 'default-user'  # 所有租户共用
oauth_validator.authorize_tokens = true     # 无法差异化配置
```

```ini
# pg_hba.conf —— 无法区分不同租户的配置
hostssl  postgres  all  0.0.0.0/0  oauth issuer=https://issuer-1.example.com
hostssl  postgres  all  0.0.0.0/0  oauth issuer=https://issuer-2.example.com
```

问题：所有连接都使用相同的 `authn_id`，无法根据不同租户设置不同的身份标识或审计策略。

#### 改进后

```ini
# postgresql.conf
# 仅设置全局默认值
oauth_validator.authorize_tokens = true
```

```ini
# pg_hba.conf —— 每个租户可以有不同的验证器配置

# 企业客户 A：使用专有发行方 + 详细的审计日志
hostssl  postgres  tenant_a  10.0.1.0/24  \
    oauth issuer=https://corp-a.auth.example.com \
    scope="openid postgres" \
    map=tenant-a-mapping \
    validator.authn_id="corp-a-user" \
    validator.audit_level=verbose

# 企业客户 B：使用专有发行方 + 不同的身份映射
hostssl  postgres  tenant_b  10.0.2.0/24  \
    oauth issuer=https://corp-b.auth.example.com \
    scope="openid postgres" \
    map=tenant-b-mapping \
    validator.authn_id="corp-b-user" \
    validator.audit_level=minimal

# 普通客户：使用共享发行方
hostssl  postgres  public  0.0.0.0/0  \
    oauth issuer=https://public.auth.example.com \
    scope="openid postgres" \
    validator.authn_id="public-user"
```

验证器模块根据 `validator.audit_level` 选项的值决定日志详细程度：

```c
static bool
validate_token(const ValidatorModuleState *state, ...)
{
    const char *audit_level = GetOAuthHBAOption(state, "audit_level");

    if (audit_level && strcmp(audit_level, "verbose") == 0) {
        // 记录完整的审计信息
        elog(LOG, "OAuth validation: tenant=%s, issuer=%s, scope=%s",
             GetOAuthHBAOption(state, "authn_id"),
             MyProcPort->hba->oauth_issuer,
             MyProcPort->hba->oauth_scope);
    }

    // ...
}
```

#### 配置验证器选项的最佳实践

1. **选项命名规范**：仅使用 ASCII 字母数字、下划线和连字符，避免与未来 HBA 语法扩展冲突
2. **值类型**：目前仅支持字符串类型，如需数值或布尔值，验证器负责在回调中进行类型转换
3. **错误处理**：如果用户设置了无效的选项值，应在 `validate_cb` 中通过 `result->error_detail` 返回清晰的错误信息
4. **reload 行为**：修改 `validator.*` 选项后，`pg_ctl reload` 会成功但连接可能失败，因此生产环境修改后应立即测试连接

## 三、小结和思考

### 核心改动总结

| 改动 | 文件 | 核心意义 |
|------|------|---------|
| PGOAUTHDEBUG 选项化 | `oauth-debug.h`, `oauth-curl.c` | 调试机制从"全有或全无"变为按需启用，安全性大幅提升 |
| RegisterOAuthHBAOptions API | `auth-oauth.c` | 验证器可声明自定义 HBA 选项 |
| GetOAuthHBAOption API | `auth-oauth.c` | 验证器可在认证回调中获取 per-HBA 配置值 |
| HBA 解析扩展 | `hba.c` | 支持 `validator.*` 命名空间的选项解析 |

### 设计哲学的体现

这两个 commit 体现了 PostgreSQL 社区在功能扩展上的一贯谨慎态度：

1. **渐进式改进**：`validator.*` 选项机制被设计为"架构死胡同"（原文 architectural dead end），作者明确表示这只是 PG19 的临时妥协，真正的解决方案是未来实现通用的 per-HBA GUC 覆盖机制。这种坦诚避免了为短期需求引入过度复杂的抽象。

2. **安全优先**：`PGOAUTHDEBUG` 的不安全选项必须显式声明 `UNSAFE:` 前缀，通过增加操作成本来防止误用。

3. **向后兼容**：`PGOAUTHDEBUG=UNSAFE` 仍然有效，确保现有脚本和文档不会突然失效。

### 对 DBA 的实际价值

- **调试效率提升**：日常 OAuth 调试不再需要冒着泄露令牌的风险
- **多租户支持**：首次能够在 HBA 层面为不同租户配置差异化的 OAuth 策略
- **运维风险降低**：HBA 配置错误会在连接时明确报错，而不是无声地使用错误的认证策略

这两个 commit 共同为 PostgreSQL 的 OAuth 认证从"能用"走向"好用"奠定了坚实基础，是 PG19 中值得关注的重要增强。
