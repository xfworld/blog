## 背景

[数据库筑基课大纲](../202409/20240914_01.md)

<b>本节: 列存之 Greenplum Append-Optimized Column-Oriented Storage, 简称 AOCO / AOCS</b>

很多同学一听 Greenplum 列存, 第一反应是: 不就是 `WITH (appendoptimized=true, orientation=column)` 吗?

如果只是记住这个语法, 基本等于没有学会. 因为真正影响生产效果的不是语法, 而是下面这条链路:

```text
批量写入 -> 按列落盘 -> 每列独立压缩/编码 -> 只读查询需要的列 -> visimap 处理删除可见性 -> vacuum/compaction 回收空间 -> gp_toolkit 验证效果
```

这篇不是语法备忘录, 而是把 Greenplum AOCO 当成一个数据库内核存储模型来看: 它为什么适合数仓大宽表? 为什么不适合高频小事务? 为什么 delete/update 后空间不立刻回来? 为什么建索引后会多出 block directory? 为什么 blocksize 不是越大越好?

## 先说结论

Greenplum AOCO 的核心价值不是“列存”两个字, 而是 **把 MPP 数仓里的大表扫描成本从按行搬运, 改成按列、按块、按压缩单元搬运**.

它适合:

- 明细事实表、日志表、事件表、宽表、历史流水表
- 批量导入, 批量追加, 少量更新删除
- 查询只访问少数列, 或对少数列聚合
- 列值局部重复、有序、低基数、适合压缩
- 分区表, 按时间/租户/业务域冷热分层

它不适合:

- 高频单行 `INSERT`
- 高频 `UPDATE` / `DELETE`
- 大量点查、强随机访问
- 每次查询都读取绝大多数字段的小窄表
- 低延迟 OLTP 事务表

官方文档说得很直接: append-optimized table storage 最适合 data warehouse 里的 denormalized fact tables, 批量装载、只读查询场景; single row `INSERT` 不推荐. 列式存储适合只读取少数列、聚合单列、压缩收益明显的场景; 但写入时一行的多个列值要写到不同磁盘位置, 所以不为写操作优化.

一句话:

> AOCO 是 Greenplum 给 OLAP 大表准备的“数据库内置列式段文件 + MVCC 元数据层 + 压缩编码 + 辅助目录”方案. 不是 OLTP heap 的平替.

## 一、药引子: 为什么 Greenplum 需要 AOCO?

假设有一张订单明细宽表:

```sql
CREATE TABLE dwd_order_detail (
  order_id      bigint,
  user_id       bigint,
  tenant_id     int,
  order_time    timestamp,
  province      text,
  sku_id        bigint,
  category_id   int,
  pay_amount    numeric(18,2),
  status        int,
  ext_json      text
)
DISTRIBUTED BY (order_id);
```

典型分析 SQL 可能只关心 3 到 5 列:

```sql
SELECT tenant_id, category_id, sum(pay_amount)
FROM dwd_order_detail
WHERE order_time >= date '2026-04-01'
  AND order_time <  date '2026-04-02'
  AND status = 1
GROUP BY tenant_id, category_id;
```

行存的问题是: 即使查询只需要 `tenant_id, category_id, pay_amount, order_time, status`, 也容易把同一行里的大量无关字段一起搬进 IO、解压、CPU cache、executor.

列存的第一性原理是:

```text
查询只需要哪些列, 存储层就尽量只搬运哪些列.
列值类型相同、分布相近, 就给压缩和编码创造条件.
```

Greenplum 是 MPP 架构, 一张大表会按分布键拆到多个 segment. AOCO 再在每个 segment 本地把数据按列组织, 于是查询代价被拆成两层:

```text
MPP 分布: 多 segment 并行扫
AOCO 列存: 每个 segment 内只扫需要列
```

这就是它的价值基点.

## 二、怎么创建 AOCO 表?

Greenplum 7 支持两种常见写法.

传统 storage option 写法:

```sql
CREATE TABLE fact_sales (
  sale_id     bigint,
  sale_time   timestamp,
  shop_id     int,
  amount      numeric(18,2),
  status      int
)
WITH (
  appendoptimized = true,
  orientation = column
)
DISTRIBUTED BY (sale_id);
```

Table access method 写法:

```sql
CREATE TABLE fact_sales (
  sale_id     bigint,
  sale_time   timestamp,
  shop_id     int,
  amount      numeric(18,2),
  status      int
)
USING ao_column
DISTRIBUTED BY (sale_id);
```

官方文档特别提到, `appendoptimized` 是 legacy `appendonly` storage option 的 thin alias; catalog 里仍按 `appendonly` 记录和展示. 所以看到 `appendonly=true` 不要误解为“用户不能 update/delete”. 在新版本 Greenplum 里, 用户视角可以更新和删除 AO/AOCO 表; 只是底层 segment file 物理上不原地覆盖.

## 三、列级压缩: AOCO 真正值钱的一层

AOCO 可以使用表级 compression, 也可以对列使用 `ENCODING` 指定列级 compression. Greenplum 7 官方文档列出的 AOCO 压缩类型包括:

| 压缩类型 | 适合场景 | 备注 |
|---|---|---|
| `zstd` | 通用压缩, 追求压缩率与速度平衡 | Greenplum 7 支持, 级别 1-19 |
| `zlib` | 通用压缩, 兼容性好 | 级别 1-9 |
| `RLE_TYPE` | 连续重复值、排序后低变化列 | AOCO 特有价值明显, 级别 1-6 |
| `none` | 不压缩 | 适合已压缩数据或不值得压缩的列 |

例子:

```sql
CREATE TABLE fact_event (
  event_id     bigint,
  event_time   timestamp ENCODING (compresstype=zstd, compresslevel=3),
  tenant_id    int       ENCODING (compresstype=rle_type, compresslevel=5),
  event_type   int       ENCODING (compresstype=rle_type, compresslevel=5),
  payload      text      ENCODING (compresstype=zstd, compresslevel=1)
)
WITH (
  appendoptimized = true,
  orientation = column
)
DISTRIBUTED BY (event_id);
```

这里的判断原则不是“全表一个压缩算法打天下”, 而是按列看:

- 时间列如果按时间写入, delta/RLE 类思路更容易受益.
- 状态、枚举、租户、类别这类低基数字段, `RLE_TYPE` 常常值得测试.
- 大 text/json 如果本身重复度不高, 更像通用压缩问题, 可以测试 `zstd` 或 `zlib`.
- 已经压缩过的二进制内容, 再压缩可能浪费 CPU.

官方文档还强调, `blocksize` 范围是 8192 到 2097152 字节, 必须是 8192 的倍数, 默认 32768. 大 block 通常有利于压缩, 但会占更多内存; Greenplum 对 column-oriented table 是按 partition、按 column 维护 buffer, 所以“分区很多 + 列很多 + blocksize 很大”会把内存压力放大.

这就是列存调优最容易踩的坑: **压缩率不是唯一目标, scan latency、CPU、内存、并发才是生产目标.**

## 四、AOCO 的物理模型: 不是一张表一个文件

从本地源码 `src/backend/access/appendonly/README.md` 看, append-optimized tables 又被历史性地称为 append-only. 关键解释是:

- 用户视角可以 update/delete.
- 物理 segment files 是 append-only, 不做 in-place update/overwrite.
- update/delete 依赖 AO visibility map 和 metadata table 的 MVCC.
- AOCS/AOCO 和 AO row layout 相似, 但需要额外元数据跟踪“哪个 segment file 对应哪个 column”.

可以把 AOCO 的存储理解成:

```text
用户表 rel
├── pg_appendonly: 记录 AO 辅助表 OID
├── pg_aoseg.pg_aoseg_xxx: segment file 元数据
├── pg_aoseg.pg_aovisimap_xxx: 删除/更新可见性覆盖层
├── pg_aoseg.pg_aoblkdir_xxx: block directory, 支持随机定位/索引访问
└── data files: 每列、每 segment 的物理数据文件
```

源码里 `pg_appendonly` 的结构很清楚:

```c
CATALOG(pg_appendonly,6105,AppendOnlyRelationId)
{
    Oid relid;
    Oid segrelid;
    Oid blkdirrelid;
    Oid visimaprelid;
    int16 version;
} FormData_pg_appendonly;
```

对应文件: `src/include/catalog/pg_appendonly.h`.

`pg_aoseg` 记录每个 segfile 的 `segno`, `eof`, `tupcount`, `varblockcount`, `eofuncompressed`, `modcount`, `formatversion`, `state`. 对应文件: `src/include/access/aosegfiles.h`.

`pg_aovisimap` 只有三个核心字段:

```text
segno, firstrownum, visimap
```

对应文件: `src/include/catalog/aovisimap.h`.

`pg_aoblkdir` 记录:

```text
segno, columngroupno, firstrownum, minipage
```

对应文件: `src/include/catalog/aoblkdir.h`.

这些辅助表是理解 AOCO 的钥匙. 很多“为什么 delete 后空间没回来”“为什么索引 scan 多一层跳转”“为什么 vacuum 会移动 tuple”的问题, 都能从这里解释.

## 五、写入路径: 一行进来, 多列分流

AOCO 的 table access method 在 `src/backend/access/aocs/aocsam_handler.c` 里注册:

```c
static const TableAmRoutine ao_column_methods = {
    ...
    .scan_begin = aoco_beginscan,
    .scan_getnextslot = aoco_getnextslot,
    .tuple_insert = aoco_tuple_insert,
    .multi_insert = aoco_multi_insert,
    .tuple_delete = aoco_tuple_delete,
    .tuple_update = aoco_tuple_update,
    .relation_vacuum = aoco_vacuum_rel,
    ...
};
```

插入时, `aoco_tuple_insert()` 获取或创建 AOCS insert descriptor, 然后调用 `aocs_insert()`. 真正按列写入在 `src/backend/access/aocs/aocsam.c` 的 `aocs_insert_values()`:

```c
for (i = 0; i < RelationGetNumberOfAttributes(rel); ++i)
{
    Datum datum = d[i];
    int err = datumstreamwrite_put(idesc->ds[i], datum, null[i], &toFree1);
    ...
    datumstreamwrite_block(idesc->ds[i], &idesc->blockDirectory, i);
}

AOTupleIdInit(aoTupleId, idesc->cur_segno, idesc->lastSequence);
```

这段代码说明了三个事实:

1. AOCO 是每列一个 datum stream, 一行进来后拆到多个列流.
2. block 满了或遇到大对象时, 写 block, 同时维护 block directory.
3. 每行仍然需要一个逻辑 tuple id, 也就是 `AOTupleId`.

`AOTupleId` 定义在 `src/include/access/appendonlytid.h`: 它把 AO tuple 的 `(segfile#, row#)` 映射成系统其它部分可以当作 heap TID 处理的结构. 这就是 Greenplum 在“完全不同的物理布局”和 PostgreSQL 执行器/索引接口之间做兼容的桥.

## 六、扫描路径: 只扫需要的列, 但不是免费的

AOCO 的 scan 入口是 `aoco_beginscan()` 和 `aoco_getnextslot()`. 源码注释里有一个很关键的点: AO_COLUMN tables 不支持 parallel SeqScan, 对 Greenplum 来说主要并行来自 MPP segments, 而不是 PostgreSQL 单节点 parallel seq scan.

查询时, 只有被投影或过滤需要的列才值得打开和读取. 在 table AM routine 里能看到 Greenplum 扩展了 extract columns 相关入口:

```c
.scan_begin_extractcolumns = aoco_beginscan_extractcolumns,
.scan_begin_extractcolumns_bm = aoco_beginscan_extractcolumns_bm,
```

这就是列存和行存的关键差异: 行存 scan 的自然单位是一行, 列存 scan 的自然单位是“本次查询需要的列集合”.

但列存不是永远更快. 如果查询是:

```sql
SELECT *
FROM fact_event
WHERE event_id = 123;
```

AOCO 可能并不占便宜. 因为:

- 要读很多列.
- 点查随机定位依赖 block directory/索引, 多一层间接访问.
- 压缩块可能需要解压后才能取少量值.
- AOCO 本身不是为高频点查设计.

列存的收益来自“少读列 + 批量扫描 + 压缩友好 + MPP 并行”, 不是来自所有查询都神奇变快.

## 七、delete/update: 物理不改旧数据, visimap 标记不可见

很多 DBA 第一次用 AOCO 会困惑:

```sql
DELETE FROM fact_event WHERE event_time < date '2025-01-01';
```

为什么磁盘空间没有马上下降?

原因是 AOCO 的物理文件 append-only. 删除并不原地改旧 block, 而是在 visibility map 里标记 tuple 不可见. 源码路径:

```text
aoco_tuple_delete()
  -> aocs_delete()
    -> AppendOnlyVisimapDelete_Hide()
```

对应文件:

- `src/backend/access/aocs/aocsam_handler.c`
- `src/backend/access/aocs/aocsam.c`
- `src/backend/access/appendonly/appendonly_visimap*.c`

`UPDATE` 也类似 heap 的思想: 逻辑上是 delete old tuple + insert new tuple. AOCO 的旧 tuple 通过 visimap 隐藏, 新 tuple 追加写入.

所以 AOCO 表的维护要关注 hidden tuples. 测试文件 `src/test/regress/sql/uaocs_catalog_tables.sql` 就用:

```sql
select * from gp_toolkit.__gp_aovisimap_hidden_info('uaocs_table_check_hidden_tup_count_after_delete'::regclass);
delete from uaocs_table_check_hidden_tup_count_after_delete where i = 1;
select * from gp_toolkit.__gp_aovisimap_hidden_info('uaocs_table_check_hidden_tup_count_after_delete'::regclass);
vacuum full uaocs_table_check_hidden_tup_count_after_delete;
select * from gp_toolkit.__gp_aovisimap_hidden_info('uaocs_table_check_hidden_tup_count_after_delete'::regclass);
```

验证 delete/update 与 vacuum 后 hidden tuple 的变化.

## 八、VACUUM/compaction: AOCO 的“垃圾回收”

AOCO 物理文件不能原地删除旧 tuple, 所以 vacuum 的基本模式不是 heap 那种就地清理页面, 而是:

```text
读旧 segment file -> 跳过 visimap 标记的 dead tuples -> 把 live tuples 写入新 segment file -> 旧 segment 标记/回收
```

`src/backend/access/appendonly/README.md` 里把它类比成 heap table 的 `VACUUM FULL`: 被移动的 tuple 会产生新的 index entries.

AOCO compaction 的测试在 `src/test/regress/sql/uaocs_compaction/threshold.sql`, 里面通过:

```sql
SET gp_appendonly_compaction_threshold=10;
...
SELECT * FROM gp_toolkit.__gp_aovisimap_compaction_info('uaocs_threshold_boundary'::regclass);
```

观察 hidden tuples 比例与 compaction 阈值.

生产建议:

- 大批量 delete 后不要只看 `count(*)`, 要看 hidden tuple 和 aoseg 状态.
- 对按时间淘汰的数据, 优先用分区 `DROP/TRUNCATE PARTITION`, 少做大范围 delete.
- 对必须 delete/update 的 AOCO 表, 规划 vacuum/compaction 窗口.
- delete/update 频繁的表, 重新评估是否应该用 heap 或 AO row.

## 九、block directory: 为什么建索引后多一层目录?

AOCO 的物理文件是 append-only varblock 流, 不像 heap 那样天然有 block number + offset. 但 PostgreSQL/Greenplum 的索引需要 TID 定位 tuple.

解决办法是 block directory:

```text
AOTupleId(segfile#, row#)
  -> pg_aoblkdir
  -> segment file offset / minipage
  -> 找到包含该 row 的 block
```

`src/backend/access/appendonly/README.md` 说得很明确: block directory 用来支持 append-only table 的随机访问; 通常顺序扫描从头到尾读 blocks, 但 index scan 需要一个 map 找到给定 TID 所在 block 的物理 segment file 和 offset. 第一次在 AO 表上 `CREATE INDEX` 时才会按需创建 block directory.

这也解释了一个取舍:

- 没索引时, AOCO 最自然的路径是批量顺序扫描.
- 有索引时, 能点到 block, 但比 heap 多一层 indirection.
- 对低选择率点查, AOCO + index 不等于 OLTP heap + btree.

## 十、怎么观测 AOCO?

先看表的存储选项:

```sql
SELECT
  c.oid::regclass AS rel,
  c.reloptions,
  a.segrelid::regclass AS aoseg,
  a.blkdirrelid::regclass AS aoblkdir,
  a.visimaprelid::regclass AS aovisimap,
  a.version
FROM pg_class c
JOIN pg_appendonly a ON a.relid = c.oid
WHERE c.oid = 'fact_event'::regclass;
```

看每列大小和压缩比, 可参考 Greenplum 测试 `src/test/isolation2/sql/aoco_column_size.sql`:

```sql
SELECT *
FROM gp_toolkit.get_column_size('fact_event'::regclass)
ORDER BY 1, 2;

SELECT relname, attnum, size, size_uncompressed, compression_ratio
FROM gp_toolkit.gp_column_size_summary
WHERE relname = 'fact_event';
```

看 AOCO segment 元数据:

```sql
SELECT *
FROM gp_toolkit.__gp_aocsseg('fact_event')
ORDER BY segment_id, column_num, segno;
```

看 visimap hidden tuple:

```sql
SELECT *
FROM gp_toolkit.__gp_aovisimap_hidden_info('fact_event'::regclass);
```

看 compaction 建议:

```sql
SELECT *
FROM gp_toolkit.__gp_aovisimap_compaction_info('fact_event'::regclass);
```

看 block directory, 需要表上有索引:

```sql
CREATE INDEX ON fact_event(event_time);

SELECT gp_segment_id, (gp_toolkit.__gp_aoblkdir('fact_event')).*
FROM gp_dist_random('gp_id');
```

这些 SQL 的意义不是“为了看而看”, 而是回答四个生产问题:

| 问题 | 观察对象 |
|---|---|
| 压缩是否有效? | `gp_column_size_summary.compression_ratio` |
| delete/update 是否造成垃圾堆积? | `__gp_aovisimap_hidden_info` |
| vacuum/compaction 是否应该做? | `__gp_aovisimap_compaction_info`, `__gp_aocsseg.state` |
| 索引访问是否走 block directory? | `__gp_aoblkdir`, `EXPLAIN` |

## 十一、选型口诀: 不是所有大表都该 AOCO

可以用下面这个表做架构评审.

| 场景 | 推荐 |
|---|---|
| 小维表, 频繁 update/delete | heap |
| 大事实表, 批量导入, 多列一起读取 | AO row 或 AOCO 测试对比 |
| 大宽表, 查询只扫少数列 | AOCO |
| 时间序列/日志, 按时间批量写入和聚合 | AOCO + 时间分区 |
| 低基数字段很多, 重复值多 | AOCO + `RLE_TYPE` 测试 |
| 高频单行写入 | heap 或重新设计写入链路 |
| 高频 delete 历史数据 | 分区表, 用 drop/truncate partition 代替 delete |
| 需要大量点查 | heap + index, 或单独服务化存储 |

更简单一点:

```text
批量写、批量读、少列读、少改动 -> AOCO
频繁小写、频繁更新、点查为主 -> 不要优先 AOCO
```

## 十二、一个可复现实验

下面给一个最小 POC, 用来验证 AOCO 的列级压缩、列大小、delete hidden tuple 和 vacuum.

```sql
DROP TABLE IF EXISTS aoco_lab;

CREATE TABLE aoco_lab (
  id          bigint,
  tenant_id   int       ENCODING (compresstype=rle_type, compresslevel=5),
  event_day   date      ENCODING (compresstype=rle_type, compresslevel=5),
  amount      numeric(18,2) ENCODING (compresstype=zstd, compresslevel=3),
  payload     text      ENCODING (compresstype=zstd, compresslevel=1)
)
WITH (
  appendoptimized = true,
  orientation = column
)
DISTRIBUTED BY (id);

INSERT INTO aoco_lab
SELECT
  i,
  (i % 10)::int,
  date '2026-04-01' + ((i % 30)::int),
  (random() * 1000)::numeric(18,2),
  repeat('payload-', 10) || (i % 100)::text
FROM generate_series(1, 1000000) AS s(i);

ANALYZE aoco_lab;

SELECT *
FROM gp_toolkit.get_column_size('aoco_lab'::regclass)
ORDER BY attnum;

DELETE FROM aoco_lab WHERE event_day < date '2026-04-10';

SELECT *
FROM gp_toolkit.__gp_aovisimap_hidden_info('aoco_lab'::regclass);

VACUUM aoco_lab;

SELECT *
FROM gp_toolkit.__gp_aovisimap_compaction_info('aoco_lab'::regclass);
```

验收点:

- `tenant_id`, `event_day` 这类重复/有序字段应比随机字段更容易压缩.
- delete 后 hidden tuple 应增加.
- vacuum/compaction 后 segment 状态和 hidden tuple 信息应变化.
- 如果 `SELECT *` 与只查少数列差异不明显, 说明你的查询并没有真正吃到列裁剪收益.

## 小结

Greenplum AOCO 的本质是: 在 MPP 数据库内部实现的一套列式 append-optimized 存储. 它用列裁剪减少 IO, 用列级压缩减少存储和扫描成本, 用 `pg_aoseg` 记录 EOF/tuple count 等元数据, 用 `pg_aovisimap` 承担 delete/update 可见性, 用 `pg_aoblkdir` 支持索引随机定位.

真正的使用原则只有一句:

> 把 AOCO 用在“批量追加、批量分析、少列访问、压缩友好”的表上; 把频繁小事务、点查、更新删除压力留给更合适的存储模型.

数据库选型和调优最怕只记参数. `appendoptimized=true, orientation=column` 只是入口; 能不能用好, 要看你是否理解它背后的 IO、压缩、MVCC 元数据、vacuum/compaction 和查询形态.
