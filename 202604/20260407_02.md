## 背景 

PostgreSQL 的并行查询执行通过 DSM（Dynamic Shared Memory）实现 worker 进程与 leader 进程之间的数据共享。在并行索引扫描场景中，EXPLAIN ANALYZE 需要收集每个 worker 的执行统计数据（instrumentation），而并行感知扫描（parallel-aware）则需要另一套数据结构。

本篇文章深入解读 PostgreSQL 提交 `dd78e69`（Allocate separate DSM chunk for parallel Index[Only]Scan instrumentation），该提交将并行索引扫描的 DSM 分配拆分为两个独立的部分：并行扫描描述符和性能监控数据结构，从而让这两套机制能够按需独立初始化，互不依赖。

  

**背景痛点**: 在此之前，并行索引扫描将并行扫描描述符和共享 instrumentation 打包在同一个 DSM 分配中。这种设计导致两个独立的功能被强耦合：即使只需要其中之一，也必须分配整个结构。更重要的是，当扫描是并行感知但不需要 instrumentation（或反之）时，代码逻辑变得复杂且容易出错。

**核心价值**: 本次提交通过将 DSM 拆分为两个独立的 chunk，实现了：
- 并行感知扫描和 instrumentation 的真正解耦
- 更清晰的代码逻辑和更易维护的结构
- 为未来扩展到其他扫描节点类型奠定基础

## 一、原理介绍

### 1.1 并行查询的 DSM 机制

PostgreSQL 的并行查询使用 DSM（Dynamic Shared Memory）让 leader 和 worker 进程共享数据。DSM 通过 TOC（Table of Contents）机制组织，每个数据块有一个唯一的 TOC key。

```
┌─────────────────────────────────────────────────────────────┐
│              ParallelContext (DSM Segment)                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                    TOC (Table of Contents)           │   │
│  │  ┌──────────────────┬──────────────────────────┐  │   │
│  │  │  Key: 123        │  Chunk: ParallelScanDesc │  │   │
│  │  ├──────────────────┼──────────────────────────┤  │   │
│  │  │  Key: 0xD...     │  Chunk: Instrumentation  │  │   │
│  │  └──────────────────┴──────────────────────────┘  │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 原有设计的问题

在原有设计中，`ParallelIndexScanDescData` 结构体本身只包含并行扫描描述符，但旧实现将 instrumentation 空间的计算和分配耦合在并行扫描的 DSM 路径里，导致两者无法独立处理：

```c
typedef struct ParallelIndexScanDescData
{
    RelFileLocator ps_locator;	/* physical table relation to scan */
    RelFileLocator ps_indexlocator; /* physical index relation to scan */
    Size		ps_offset_am;	/* Offset to am-specific structure */
    char		ps_snapshot_data[FLEXIBLE_ARRAY_MEMBER];
} ParallelIndexScanDescData;
```

**问题**：

1. **强耦合**: 旧实现需要同时兼容 `parallel_aware=false` 但 `instrument=true`（EXPLAIN ANALYZE 路径），以及 `parallel_aware=true` 但无需 instrumentation 的路径，instrumentation 空间的计算被打包在并行扫描的 DSM 分配逻辑中，无法独立处理
2. **共享 TOC key**: 两者共用 `plan_node_id` 作为 TOC key，无法独立分配
3. **逻辑复杂**: 初始化函数必须同时处理两种情况，条件分支多

### 1.3 新的设计：分离的 DSM chunks

提交引入了 `PARALLEL_KEY_SCAN_INSTRUMENT_OFFSET` 常量：

```c
#define PARALLEL_KEY_SCAN_INSTRUMENT_OFFSET    UINT64CONST(0xD000000000000000)
```

现在并行扫描使用 `plan_node_id` 作为 TOC key，而 instrumentation 使用 `plan_node_id + PARALLEL_KEY_SCAN_INSTRUMENT_OFFSET`：

```
┌────────────────────────────────────────────────────────────────┐
│                拆分后的 DSM 分配结构                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Chunk 1: Parallel Scan Descriptor                             │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │  TOC Key: plan_node_id (如 123)                         │ │
│  │  内容: ps_locator, ps_indexlocator, ps_snapshot_data... │ │
│  └──────────────────────────────────────────────────────────┘ │
│                                                                │
│  Chunk 2: Shared Instrumentation (EXPLAIN ANALYZE 用)        │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │  TOC Key: plan_node_id + 0xD000000000000000            │ │
│  │  内容: SharedIndexScanInstrumentation                   │ │
│  │         - num_workers                                    │ │
│  │         - winstrument[] (每个 worker 的统计数据)        │ │
│  └──────────────────────────────────────────────────────────┘ │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 1.4 SharedIndexScanInstrumentation 结构

```c
typedef struct SharedIndexScanInstrumentation
{
    int         num_workers;
    IndexScanInstrumentation winstrument[FLEXIBLE_ARRAY_MEMBER];
} SharedIndexScanInstrumentation;

typedef struct IndexScanInstrumentation
{
    uint64      nsearches;  /* 索引搜索次数 */
} IndexScanInstrumentation;
```

### 1.5 代码变更详解

#### 1.5.1 index_parallelscan_estimate 的简化

**修改前**：
```c
Size
index_parallelscan_estimate(Relation indexRelation, int nkeys, int norderbys,
                          Snapshot snapshot, bool instrument,
                          bool parallel_aware, int nworkers)
{
    nbytes = offsetof(ParallelIndexScanDescData, ps_snapshot_data);
    nbytes = add_size(nbytes, EstimateSnapshotSpace(snapshot));

    if (instrument)
    {
        // 计算 instrumentation 空间
        sharedinfosz = offsetof(SharedIndexScanInstrumentation, winstrument) +
            nworkers * sizeof(IndexScanInstrumentation);
        nbytes = add_size(nbytes, sharedinfosz);
    }

    if (parallel_aware && indexRelation->rd_indam->amestimateparallelscan != NULL)
        // 添加 AM-specific 空间
        nbytes = add_size(nbytes, ...);
}
```

**修改后**：
```c
Size
index_parallelscan_estimate(Relation indexRelation, int nkeys, int norderbys,
                          Snapshot snapshot)
{
    nbytes = offsetof(ParallelIndexScanDescData, ps_snapshot_data);
    nbytes = add_size(nbytes, EstimateSnapshotSpace(snapshot));

    // instrumentation 的计算被移到了独立的函数
    // 只处理 parallel-aware 的 AM-specific 空间
    if (indexRelation->rd_indam->amestimateparallelscan != NULL)
        nbytes = add_size(nbytes, ...);
}
```

#### 1.5.2 新的 instrumentation 估算函数

```c
/*
 * 计算共享 instrumentation 所需的空间并通知 estimator
 */
void
ExecIndexScanInstrumentEstimate(IndexScanState *node, ParallelContext *pcxt)
{
    Size    size;

    // 只有在需要 instrumentation 且有 worker 时才分配
    if (!node->ss.ps.instrument || pcxt->nworkers == 0)
        return;

    size = offsetof(SharedIndexScanInstrumentation, winstrument) +
        pcxt->nworkers * sizeof(IndexScanInstrumentation);

    shm_toc_estimate_chunk(&pcxt->estimator, size);
    shm_toc_estimate_keys(&pcxt->estimator, 1);
}
```

#### 1.5.3 execParallel.c 的调用逻辑变更

**修改前**：
```c
case T_IndexScanState:
    ExecIndexScanEstimate(...);  // 同时处理 parallel 和 instrumentation
    break;
```

**修改后**：
```c
case T_IndexScanState:
    // 只有在 parallel-aware 时才分配并行扫描描述符
    if (planstate->plan->parallel_aware)
        ExecIndexScanEstimate(...);

    // 即使不是 parallel-aware，只要处于并行查询且需要 EXPLAIN ANALYZE 就会分配 instrumentation
    // 注意：ExecIndexScanInstrumentEstimate() 内部会检查 pcxt->nworkers，nworkers == 0 时直接返回
    ExecIndexScanInstrumentEstimate(...);
    break;
```

### 1.6 shm_toc 的 TOC key 机制

TOC（Table of Contents）是 DSM 中的索引结构，每个 chunk 有一个 64-bit 的 key：

```c
// 并行扫描描述符的 TOC key
shm_toc_insert(pcxt->toc, plan_node_id, piscan);

// instrumentation 的 TOC key（使用不同的 key）
shm_toc_insert(pcxt->toc,
               plan_node_id + PARALLEL_KEY_SCAN_INSTRUMENT_OFFSET,
               node->iss_SharedInfo);
```

**为什么用 0xD000000000000000？**

这是一个随意选择的高位值，确保与正常的 plan_node_id（通常是较小的值）不会冲突。这只是一个临时方案，提交注释也提到：

> In the future, if nodes need more DSM allocations, we would need a more robust system.

## 二、应用场景及最佳实践

### 2.1 场景：理解 EXPLAIN (ANALYZE, BUFFERS) 的输出

当执行 `EXPLAIN (ANALYZE, BUFFERS)` 时，如果查询使用了并行索引扫描，PostgreSQL 需要收集每个 worker 的索引扫描统计数据。

**示例查询**：

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders
WHERE customer_id = 123
  AND order_date > '2024-01-01';
```

**可能的执行计划**：

```
Gather  (cost=0.56..1234.56 rows=456 width=128)
       (actual time=0.12..45.67 rows=456 loops=1)
  ->  Parallel Index Scan on orders  (cost=0.56..1234.56 rows=456 width=128)
                                 (actual time=0.12..45.67 rows=456 loops=1)
        Index Cond: (customer_id = 123)
        Buffers: shared hit=1234 read=56
```

在并行执行时，leader 与 workers 的 `nsearches`（索引搜索次数）会汇总后显示在 EXPLAIN 输出里：

```
Gather  (cost=... rows=...)
       (actual time=...)
  ->  Parallel Index Scan on orders  (cost=... rows=...)
                                 (actual time=...)
        Index Cond: (customer_id = 123)
        Index Searches: 1234
```

### 2.2 场景：诊断并行扫描性能问题

当怀疑并行扫描没有按预期工作时，可以通过 EXPLAIN ANALYZE 检查。

**优化前**（原有设计的问题）：

假设你想使用 `parallel_aware` 扫描但不需要 instrumentation（为了减少开销）。在原有设计中，这很难做到，因为两者打包在同一 DSM chunk 中，分配路径无法区分。

**优化后**（新设计）：

DSM 分离后，执行器内部可分别为 parallel-aware 扫描描述符和 instrumentation 估算/分配 DSM：

```sql
-- 启用并行扫描（无 ANALYZE）
SET parallel_tuple_cost = 0;  -- 将 tuple cost 设为 0，鼓励使用并行
SET max_parallel_workers_per_gather = 4;

-- 这个查询会使用并行索引扫描
-- 注意：EXPLAIN 只生成计划，不执行查询，无法证明运行期 DSM 分配行为
EXPLAIN SELECT * FROM orders WHERE customer_id = 123;
```

如果你需要 ANALYZE 数据来诊断性能问题：

```sql
-- 启用带 instrumentation 的并行扫描
-- EXECUTE 才能真正触发 execParallel.c / nodeIndexscan.c 中的 DSM 分配路径
EXPLAIN (ANALYZE) SELECT * FROM orders WHERE customer_id = 123;
```

### 2.3 场景：查看索引搜索统计

当执行 `EXPLAIN (ANALYZE)` 时，并行查询的每个 worker 都会收集统计数据到 `SharedIndexScanInstrumentation` 中。最终 EXPLAIN 输出会显示一个汇总的 "Index Searches" 值（所有 worker + leader 的 nsearches 之和）：

```
Gather  (cost=... rows=...)
       (actual time=...)
  ->  Parallel Index Scan on orders  (cost=... rows=...)
                                 (actual time=...)
        Index Cond: (customer_id = 123)
        Index Searches: 1234
```

**注意**：EXPLAIN ANALYZE 输出**不会**显示每个 worker 的 nsearches 分开值，只会显示汇总后的 "Index Searches" 总数。标准视图（如 `pg_stat_activity`）无法查看 per-worker `nsearches`，如需查看每个 worker 的详细统计，需要依赖自定义调试或额外插桩。

这些 `nsearches` 值来自 `IndexScanInstrumentation` 结构体中的 `nsearches` 字段，通过 DSM 共享给 leader。

### 2.4 对比：新旧架构的行为差异

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| Parallel-aware + Instrument | 单一 DSM chunk | 两个独立 DSM chunk |
| Parallel-aware + No Instrument | 仍需处理 instrumentation 逻辑 | 只分配并行扫描 chunk |
| Not parallel-aware + Instrument | 难以区分 | 独立分配 instrumentation chunk |
| 代码可读性 | 复杂的条件分支 | 清晰的职责分离 |

## 五、小结和思考

### 5.1 核心价值总结

本次提交 `dd78e69` 是一个代码重构，为未来的扩展奠定了基础：

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| DSM 分配 | 单一 chunk 包含所有数据 | 两个独立 chunk |
| 耦合度 | parallel-aware 和 instrumentation 强耦合 | 完全解耦 |
| 代码逻辑 | 复杂的条件分支 | 清晰的职责分离 |
| 可扩展性 | 难以扩展到其他场景 | 为其他扫描类型提供范式 |

### 5.2 对 DBA 的实际意义

这是一个内部重构，对 DBA 的直接可见影响较小，但带来了以下间接好处：

1. **更可靠的 EXPLAIN ANALYZE**: parallel-aware 和 instrumentation 的解耦使代码更稳定
2. **已扩展到其他扫描类型**: 这个模式已经从 Index[Only]Scan 扩展到了 BitmapIndexScan（参见 `nodeBitmapIndexscan.c`），进一步统一了并行查询的 instrumentation 架构
3. **更清晰的性能数据**: DSM 分离确保即使在复杂场景下也能正确收集统计数据

### 5.3 查看并行查询的统计信息

```sql
-- 查看当前会话的并行查询设置
SHOW max_parallel_workers_per_gather;
SHOW parallel_tuple_cost;
SHOW parallel_leader_participation;

-- 执行一个并行查询并查看 ANALYZE 输出
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders
WHERE customer_id BETWEEN 100 AND 200;

-- 检查是否有 parallel-aware 扫描
-- 在执行计划中搜索 "Parallel" 关键字
```

### 5.4 未来展望

这个模式已经扩展到了 Bitmap Index Scan。PostgreSQL 社区可能会继续扩展到其他扫描节点类型：

- Sequential Scan（并行 seq scan 的 instrumentation）
- 其他需要并行 instrumentation 的节点类型

这将统一并行查询的 instrumentation 架构，使 PostgreSQL 的性能监控更加一致和可靠。
