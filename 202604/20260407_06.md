## 背景 

PostgreSQL 的 `auto_explain` 扩展是 DBA 监控慢查询执行计划的利器，但长期以来只能记录标准的 EXPLAIN 输出。对于使用扩展（如 `pg_overexplain`、`pg_plan_advice`）添加的自定义 EXPLAIN 选项，auto_explain 无法将其纳入日志。

本篇文章深入解读 PostgreSQL 提交 `e972dff` 和 `0442f1c`，这两个提交协同工作，为 `auto_explain` 引入了扩展选项支持：允许在自动记录执行计划时包含扩展提供的自定义 EXPLAIN 选项，并通过 GUC 检查机制确保只有合法的选项值才能被设置。

  

**背景痛点**: 此前 `auto_explain` 只能记录标准的 EXPLAIN 输出（如 `ANALYZE`, `VERBOSE`, `COSTS` 等）。随着第三方扩展（如 `pg_overexplain`、`pg_plan_advice`）引入自定义 EXPLAIN 选项（如 `DEBUG`, `RANGE_TABLE`, `plan_advice`），这些扩展提供的丰富调试信息无法通过 auto_explain 自动记录到日志中。

**核心价值**: 本次提交让 `auto_explain` 能够：
- 自动记录扩展提供的自定义 EXPLAIN 选项
- 通过 GUC 检查机制验证选项合法性，避免把非法配置写入 GUC
- 为 DBA 提供更丰富的自动调优和故障排查数据

## 一、原理介绍

### 1.1 原有机制的限制

在原有实现中，`auto_explain` 的日志选项是固定的几项（实际字段名和默认值可能有差异，此处为示意）：

```c
/* 示意性伪代码 — 实际字段名和默认值请参阅 auto_explain.c */
static bool auto_explain_log_analyze = false;
static bool auto_explain_log_verbose = false;
static bool auto_explain_log_buffers = false;
static bool auto_explain_log_timing = true;   /* 注意：默认值是 true */
static bool auto_explain_log_wal = false;
static bool auto_explain_log_triggers = false;
static bool auto_explain_log_settings = false;
/* ... 以及 log_format、log_level、log_min_duration 等 */
```

原有实现中不存在 `log_costs` 字段（COSTS 选项通过其他方式控制）。当扩展（如 pg_overexplain）注册了自定义 EXPLAIN 选项（如 `DEBUG`, `RANGE_TABLE`）时，`auto_explain` 无法知道这些选项的存在，更无法将它们纳入日志。

### 1.2 扩展的 EXPLAIN 选项机制

PostgreSQL 提供了一套扩展 EXPLAIN 选项的接口，允许第三方模块注册自定义选项：

```c
/* 注册一个新的 EXPLAIN 选项 */
void RegisterExtensionExplainOption(const char *option_name,
                                   ExplainOptionHandler handler,
                                   ExplainOptionGUCCheckHandler guc_check_handler);
```

**参数说明**：
- `option_name`: 选项名称（如 "debug", "range_table"）
- `handler`: 处理函数，当 EXPLAIN 使用该选项时调用
- `guc_check_handler`: GUC 检查处理函数，用于验证 GUC 值的合法性

**已有的扩展示例**：

| 扩展 | 注册名 | 说明 |
|------|--------|------|
| `pg_overexplain` | `debug` | 输出更详细的调试信息 |
| `pg_overexplain` | `range_table` | 输出 range table 信息 |
| `pg_plan_advice` | `plan_advice` | 输出查询优化建议 |

> **注意**：SQL 输入大小写不敏感（如 `DEBUG`、`Debug`、`debug` 均可识别），但扩展注册时使用的是小写名称。

### 1.3 GUC 检查处理机制

`0442f1c` 提交引入了一个关键的类型和两个关键函数：

```c
/* 新类型：GUC 检查处理函数类型 */
typedef bool (*ExplainOptionGUCCheckHandler) (const char *option_name,
                                              const char *option_value,
                                              NodeTag option_type);

/* 检查扩展选项是否合法（由 auto_explain 在设置 GUC 时调用） */
bool GUCCheckExplainExtensionOption(const char *option_name,
                                    const char *option_value,
                                    NodeTag option_type);

/* Boolean 类型选项的检查实现 */
bool GUCCheckBooleanExplainOption(const char *option_name,
                                   const char *option_value,
                                   NodeTag option_type);
```

**工作流程**：

```
用户设置 GUC
    │
    ▼
auto_explain.log_extension_options = 'DEBUG, RANGE_TABLE'
    │
    ▼
check_log_extension_options() 钩子被调用
    │
    ▼
解析选项字符串 'DEBUG, RANGE_TABLE'
    │
    ▼
对每个选项调用 GUCCheckExplainExtensionOption()
    │
    ├──► 查找注册的选项名（如 "DEBUG"）
    │        │
    │        ▼
    │    调用该选项的 guc_check_handler
    │        │
    │        ▼
    │    返回 true（合法）或 false（非法）
    │
    └──► 如果找到但 guc_check_handler 返回 false → GUC 设置失败，报错
```

### 1.4 Boolean 选项的检查逻辑

`GUCCheckBooleanExplainOption` 的检查逻辑与 `defGetBoolean` 保持一致：

```c
bool
GUCCheckBooleanExplainOption(const char *option_name,
                              const char *option_value,
                              NodeTag option_type)
{
    bool valid = false;

    if (option_value == NULL)
    {
        /* 无参数值视为有效（相当于省略） */
        valid = true;
    }
    else if (option_type == T_String)
    {
        /* Boolean 字符串只接受这些值 */
        if (pg_strcasecmp(option_value, "true") == 0 ||
            pg_strcasecmp(option_value, "false") == 0 ||
            pg_strcasecmp(option_value, "on") == 0 ||
            pg_strcasecmp(option_value, "off") == 0)
            valid = true;
    }
    else if (option_type == T_Integer)
    {
        /* 整数只接受 0 或 1 */
        long        value;
        char       *end;

        errno = 0;
        value = strtol(option_value, &end, 0);
        /* 完整验证：解析成功、无尾随字符、无溢出、值为 0 或 1 */
        if (errno == 0 && *end == '\0' && end != option_value &&
            value == (int) value && (value == 0 || value == 1))
            valid = true;
    }

    if (!valid)
        GUC_check_errmsg("EXPLAIN option \"%s\" requires a Boolean value",
                          option_name);

    return valid;
}
```

**可接受的值形式**：
- 无参数值（视为有效）
- `true` / `false` / `on` / `off`（不区分大小写）
- 整数 `0` 或 `1`

**不接受的值形式**：
- 带引号但内容不是合法布尔字面量的字符串（如 `"0"`）—— 会作为 `T_String` 处理，但内容不是 true/false/on/off
- 注意：`"off"`、`'false'` 这类布尔字符串是**可接受**的
- 浮点数（如 `3.14`）
- 其他整数（如 `2`）

### 1.5 新增的 GUC: log_extension_options

`e972dff` 提交添加了新的 GUC：

```sql
-- 语法（需加 SET 前缀，才是合法 SQL）
SET auto_explain.log_extension_options = 'option1, option2 value, option3 "string_value"';

-- 示例
SET auto_explain.log_extension_options = 'DEBUG, RANGE_TABLE';
SET auto_explain.log_extension_options = 'DEBUG TRUE, RANGE_TABLE FALSE';
SET auto_explain.log_extension_options = 'DEBUG 1, RANGE_TABLE "off"';
```

在 `postgresql.conf` 中的写法（无需 SET 前缀）：

```ini
auto_explain.log_extension_options = 'DEBUG, RANGE_TABLE'
auto_explain.log_extension_options = 'DEBUG TRUE, RANGE_TABLE FALSE'
auto_explain.log_extension_options = 'DEBUG 1, RANGE_TABLE "off"'
```

**支持的语法**：

| 语法 | 说明 |
|------|------|
| `DEBUG` | 无参数的 Boolean 选项 |
| `DEBUG TRUE` | 带 Boolean 参数 |
| `DEBUG 1` | 带整数参数（0 或 1） |
| `DEBUG "off"` | 带字符串参数 |
| `option1, option2` | 多个选项用逗号分隔 |

### 1.6 选项解析流程

```c
/*
 * 解析 'DEBUG, RANGE_TABLE FALSE' 为:
 * [
 *   { name: "DEBUG", value: NULL, type: T_Invalid },
 *   { name: "RANGE_TABLE", value: "FALSE", type: T_String }
 * ]
 */
static int
auto_explain_split_options(char *rawstring, auto_explain_option *options,
                           int maxoptions, char **errmsg)
{
    // 1. 使用 scanner_isspace() 跳过前导空白
    // 2. 循环解析每个选项:
    //    a. scan_identifier() 解析选项名
    //    b. 检查是否有参数值（逗号或字符串结尾前）
    //    c. 根据参数类型解析: 数字 / 单引号字符串 / 双引号标识符
    //    d. 验证格式并存储
}
```

## 二、应用场景及最佳实践

### 2.1 场景一：自动记录 pg_overexplain 的 DEBUG 信息

**背景**: `pg_overexplain` 扩展提供了 `DEBUG` 选项，可以输出更详细的执行计划信息，包括每个节点的并行安全属性、别名信息等。

**配置步骤**：

```sql
-- 1. 加载扩展
-- 方式 A：LOAD（仅影响当前会话，立即生效，适合开发和测试）
LOAD 'auto_explain';
LOAD 'pg_overexplain';

-- 方式 B：session_preload_libraries（仅影响新连接，不影响当前会话）
-- 只在连接启动时生效，可通过 postgresql.conf、ALTER ROLE SET、ALTER DATABASE SET 等方式为新会话生效

-- 方式 C：shared_preload_libraries（需重启实例，适合生产环境）
-- 以下为 postgresql.conf 中的配置写法，不是 SQL：
```

```ini
shared_preload_libraries = 'pg_overexplain,auto_explain'
```

```sql
-- 预加载主要是部署选择，不是功能前提
-- 选择哪种方式取决于：是否需要重启、是否影响所有会话、是否需要持久化配置

-- 2. 配置 auto_explain
SET auto_explain.log_min_duration = '3s';      -- 记录超过 3 秒的查询
SET auto_explain.log_analyze = on;              -- 显示实际运行时间
SET auto_explain.log_buffers = on;             -- 显示缓冲区使用
SET auto_explain.log_extension_options = 'debug';  -- 启用 debug 选项

-- 3. 使用 LOAD 时立即生效；使用预加载方式时需重启或等待新连接
-- 注意：pg_reload_conf() 不能让当前已连接会话重新加载预加载的库
```

**效果**：在已加载 `auto_explain`、当前会话已启用（`auto_explain.log_min_duration` 未被禁用）且查询时长超过阈值时，符合采样条件的查询会自动记录其执行计划，并且包含 `DEBUG` 选项提供的扩展信息（如 `Parallel Safe: true/false`, `Command Type: select` 等）。

### 2.2 场景二：调试分区表的 RANGE_TABLE 输出

**背景**: 在排查复杂查询的性能问题时，需要查看完整的 range table 信息。

**配置**：

```sql
-- 启用 RANGE_TABLE 选项（只有加载并注册该扩展选项的模块后才能使用）
SET auto_explain.log_extension_options = 'RANGE_TABLE';

-- 执行一个复杂的多表连接查询
EXPLAIN (RANGE_TABLE) SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN products p ON o.product_id = p.id
WHERE ...;

-- 注意：只有当 pg_overexplain（或其他提供 RANGE_TABLE 选项的扩展）
-- 被正确加载并注册后，EXPLAIN 才能使用 RANGE_TABLE 选项
-- 使用 auto_explain 时，只要查询被自动 explain，就会带上 RANGE_TABLE
```

**优化前**（无此功能）：
- DBA 只能事后手动对慢查询执行 `EXPLAIN (RANGE_TABLE) ...`
- 如果问题已经发生，可能已经错失关键信息

**优化后**：
- auto_explain 自动记录 RANGE_TABLE 信息
- 可以追溯分析历史慢查询的表连接情况

### 2.3 场景三：结合多个扩展选项

```sql
-- 同时启用多个扩展选项
SET auto_explain.log_extension_options = 'DEBUG, RANGE_TABLE';

-- 日志输出将包含：
-- - 标准 EXPLAIN 输出（根据其他 auto_explain.* 设置）
-- - DEBUG 提供的扩展信息
-- - RANGE_TABLE 提供的表信息
```

### 2.4 错误处理示例

**无效的选项名**：

```sql
SET auto_explain.log_extension_options = 'UNKNOWN_OPTION';

-- 错误：
-- ERROR:  invalid value for parameter "auto_explain.log_extension_options": "UNKNOWN_OPTION"
-- DETAIL:  unrecognized EXPLAIN option "UNKNOWN_OPTION"
```

**无效的选项值类型**：

```sql
-- range_table 选项期望 Boolean 值
SET auto_explain.log_extension_options = 'range_table maybe';

-- 错误：
-- ERROR:  invalid value for parameter "auto_explain.log_extension_options": "range_table maybe"
-- DETAIL:  EXPLAIN option "range_table" requires a Boolean value
```

```sql
-- 传入整数 2（不是合法的 Boolean）
SET auto_explain.log_extension_options = 'range_table 2';

-- 错误：
-- ERROR:  invalid value for parameter "auto_explain.log_extension_options": "range_table 2"
-- DETAIL:  EXPLAIN option "range_table" requires a Boolean value
```

### 2.5 语法错误处理

```sql
-- 尾部逗号
SET auto_explain.log_extension_options = 'DEBUG,';
-- ERROR:  invalid value for parameter "auto_explain.log_extension_options": "DEBUG,"
-- DETAIL:  trailing comma in option list

-- 未闭合的引号（双引号）
SET auto_explain.log_extension_options = '"DEBUG';
-- ERROR:  invalid value for parameter "auto_explain.log_extension_options": ""DEBUG"
-- DETAIL:  unterminated double-quoted string

-- 未闭合的引号（单引号，需用 dollar-quoting 避免 SQL 解析层先行拦截）
SET auto_explain.log_extension_options = $$'$$;
-- ERROR:  invalid value for parameter "auto_explain.log_extension_options": "'"
-- DETAIL:  unterminated single-quoted string
```

### 3.3 auto_explain.c 的核心变更

#### 3.3.1 新增 GUC 定义

```c
DefineCustomStringVariable("auto_explain.log_extension_options",
                           "Extension EXPLAIN options to be added.",
                           NULL,
                           &auto_explain_log_extension_options,
                           NULL,
                           PGC_SUSET,
                           0,
                           check_log_extension_options,    // GUC check hook
                           assign_log_extension_options,  // GUC assign hook
                           NULL);
```

#### 3.3.2 应用扩展选项到 ExplainState

```c
static void
apply_extension_options(ExplainState *es,
                        auto_explain_extension_options *ext)
{
    if (ext == NULL)
        return;

    for (int i = 0; i < ext->noptions; i++)
    {
        auto_explain_option *opt = &ext->options[i];
        DefElem    *def;
        Node       *arg;

        /* 根据类型构建参数节点 */
        if (opt->value == NULL)
            arg = NULL;
        else if (opt->type == T_Integer)
            arg = (Node *) makeInteger(strtol(opt->value, NULL, 0));
        else if (opt->type == T_Float)
            arg = (Node *) makeFloat(opt->value);
        else
            arg = (Node *) makeString(opt->value);

        /* 构建 DefElem 并应用 */
        def = makeDefElem(opt->name, arg, -1);
        ApplyExtensionExplainOption(es, def, NULL);
    }
}
```

#### 3.3.3 在 explain_ExecutorEnd 中调用

```c
explain_ExecutorEnd(QueryDesc *queryDesc)
{
    /* ... */

    if (es->str != NULL)
    {
        es->format = auto_explain_log_format;
        es->settings = auto_explain_log_settings;

        /* 应用扩展选项 */
        apply_extension_options(es, extension_options);

        ExplainBeginOutput(es);
        /* ... */
    }
}
```

## 四、扩展选项处理流程全图

```
┌──────────────────────────────────────────────────────────────────────┐
│                        auto_explain 工作流程                           │
└──────────────────────────────────────────────────────────────────────┘

1. 模块加载时 (_PG_init)
   │
   ├──► 定义 log_extension_options GUC（含 check 和 assign hooks）
   │
   └──► 当用户设置 GUC 时：
           │
           ▼
2. check_log_extension_options() 被调用
   │
   ├──► 解析选项字符串（如 'DEBUG, RANGE_TABLE'）
   │
   ├──► 对每个选项调用 GUCCheckExplainExtensionOption()
   │        │
   │        ▼
   │    查找注册的选项（如 "DEBUG"）
   │        │
   │        ▼
   │    调用选项的 guc_check_handler（如 GUCCheckBooleanExplainOption）
   │        │
   │        ▼
   │    返回 true/false
   │        │
   └──► 如果所有检查通过，存储解析结果到 *extra
           │
           ▼
3. assign_log_extension_options() 被调用
   │
   └──► 将 extra 指针保存到 extension_options 全局变量
           │
           ▼
4. 查询执行完成时 (explain_ExecutorEnd)
   │
   └──► 调用 apply_extension_options(es, extension_options)
           │
           ▼
5. ApplyExtensionExplainOption() 被调用
   │
   ├──► 查找注册的选项处理器（如 overexplain_debug_handler）
   │
   └──► 调用处理器，更新 ExplainState
           │
           ▼
6. ExplainQueryText/ExplainPrintPlan 输出时
   │
   └──► 扩展选项的影响已体现在 ExplainState 中
           │
           ▼
7. 最终日志输出包含扩展信息
```

## 五、小结和思考

### 5.1 核心价值总结

这两个提交协同工作，为 auto_explain 带来了扩展选项支持：

| 提交 | 功能 | 核心价值 |
|------|------|----------|
| `0442f1c` | GUC 检查处理机制 | 防止无效选项导致所有查询失败 |
| `e972dff` | `log_extension_options` GUC | 自动记录扩展 EXPLAIN 选项 |

### 5.2 使用前提

1. **加载提供扩展选项的模块**：
   ```ini
   # postgresql.conf 中的配置（需重启）
   shared_preload_libraries = 'pg_overexplain'  -- 需重启实例
   # 或使用 session_preload_libraries（新会话才生效）
   ```

   **替代方案——会话级 `LOAD`**：若扩展自身无 shared_preload 依赖（即不需要在 postmaster 启动时申请共享内存等资源），可直接在会话中执行 `LOAD 'pg_overexplain'`，无需修改配置文件，也无需重启：

   ```sql
   -- 会话级加载（扩展自身无 shared_preload 依赖时可用）
   LOAD 'pg_overexplain';
   SET auto_explain.log_extension_options = 'DEBUG, RANGE_TABLE';
   ```

2. **然后设置 GUC**：
   ```sql
   SET auto_explain.log_extension_options = 'DEBUG, RANGE_TABLE';
   ```

**注意**:
- `shared_preload_libraries` 修改后必须重启 PostgreSQL 才能生效
- `session_preload_libraries` 只影响后续新建立的连接，不影响当前已连接的会话
- `pg_reload_conf()` 不能让当前已连接会话重新加载这些预加载库
- 顺序很重要——扩展必须先加载，其选项才能被注册和验证
- 预加载（`shared_preload_libraries`/`session_preload_libraries`）只在扩展自身有 shared_preload 依赖时才必须；若仅需提供 auto_explain 扩展选项，`LOAD` 即可满足

### 5.3 最佳实践建议

1. **结合 `log_min_duration` 使用**：
   ```sql
   SET auto_explain.log_min_duration = '5s';
   SET auto_explain.log_analyze = on;
   SET auto_explain.log_extension_options = 'DEBUG';
   ```
   只记录超过 5 秒的慢查询，同时包含 DEBUG 扩展信息。

2. **结合多个扩展**：
   ```sql
   -- 假设同时使用 pg_overexplain 和 pg_plan_advice
   SET auto_explain.log_extension_options = 'DEBUG, RANGE_TABLE, plan_advice';
   ```

3. **在 postgresql.conf 中配置**：
   ```ini
   # shared_preload_libraries 需要重启，session_preload_libraries 新连接生效
   shared_preload_libraries = 'pg_overexplain,auto_explain'
   auto_explain.log_min_duration = '3s'
   auto_explain.log_analyze = on
   auto_explain.log_extension_options = 'DEBUG'
   ```
   
   **注意**：`pg_reload_conf()` 不能让当前已连接会话重新加载这些预加载库

### 5.4 安全性考量

- `auto_explain.log_extension_options` 是 PGC_SUSET 级别的参数：默认由超级用户修改，也可以通过参数 ACL 有条件地授权给其他角色（通过 `pg_parameter_acl` 授予）；是否允许非超级用户修改还取决于参数上下文、现有参数 ACL 配置和当前环境
- GUC 检查机制确保无效选项会被拒绝，避免因配置错误导致查询失败
- 这比让 auto_explain 盲目尝试设置选项要安全得多

### 5.5 未来展望

1. **更多内置扩展选项**: PostgreSQL 未来可能内置更多扩展选项
2. **更丰富的检查类型**: 目前只有 `GUCCheckBooleanExplainOption`，未来可能支持更多类型（如整数、字符串等）
3. **动态选项发现**: 可能通过 `pg_explain_options` 等系统目录动态发现可用选项
