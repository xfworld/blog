## WAIT FOR 优化：避免在 Hot Standby 上触发恢复冲突

## 背景 

WAIT FOR 是 PostgreSQL 用于等待复制事件（如 LSN 回放、刷新、写入）的重要命令，常用于主从复制环境中的复制进度协调。然而，在某些场景下（如热 Standby），WAIT FOR 命令构建返回元组描述符时可能会触发意外的 syscache 访问，进而重新建立 catalog snapshot，可能导致 Standby 上的恢复冲突。

本篇文章深入解读 PostgreSQL 提交 `834038c`（Avoid syscache lookup while building a WAIT FOR tuple descriptor），该提交通过使用 `TupleDescInitBuiltinEntry` 替代 `TupleDescInitEntry`，完全避免了 WAIT FOR 命令中的 syscache 访问，从而防止了在 Standby 上可能出现的恢复冲突问题。
  

**背景痛点**: 在主从复制环境中，当使用 WAIT FOR 命令等待 LSN 复制进度时，如果 WAIT FOR 在构建返回元组描述符时调用 `TupleDescInitEntry`，该函数会访问 syscache。Syscache 访问在某些情况下会重新建立 catalog snapshot，这在热 Standby 上可能导致恢复冲突（recovery conflict），可能导致 WAIT FOR 会话被取消或报错。

**核心价值**: 通过使用 `TupleDescInitBuiltinEntry` 替代 `TupleDescInitEntry`，完全避免了 catalog 访问，避免了 WaitStmtResultDesc() 结果描述符构建路径中的 syscache 访问（见 wait.c line 334）。

## 一、原理介绍

### 1.1 WAIT FOR 命令概述

WAIT FOR 是 PostgreSQL 的一个内置命令，用于等待指定的 LSN（Log Sequence Number）达到特定状态：

- **LSN 回放完成**: `WAIT FOR LSN 'lsn' WITH (MODE 'standby_replay')`
- **LSN 刷新完成**: `WAIT FOR LSN 'lsn' WITH (MODE 'standby_flush')`
- **LSN 写入完成**: `WAIT FOR LSN 'lsn' WITH (MODE 'standby_write')`
- **主库 LSN 刷新**: `WAIT FOR LSN 'lsn' WITH (MODE 'primary_flush')`

**典型使用场景**:

```sql
-- 等待 LSN 0/306EE20 在 Standby 上回放完成
WAIT FOR LSN '0/306EE20' WITH (MODE 'standby_replay');

-- 等待 LSN 在 Standby 上刷新（持久化）
WAIT FOR LSN '0/306EE20' WITH (MODE 'standby_flush');

-- 带超时和 NO_THROW 的用法
WAIT FOR LSN '0/306EE20' WITH (MODE 'standby_write', TIMEOUT '100ms', NO_THROW);
```

WAIT FOR 命令返回单列单行的文本结果：`success`（成功）。当指定了 `NO_THROW` 选项时，还会返回 `timeout`（超时）或 `not in recovery`（不在恢复状态）；否则这些状态会作为错误抛出。

### 1.2 元组描述符构建的问题

当 WAIT FOR 命令返回结果时，它需要构建一个元组描述符（Tuple Descriptor）来描述返回的列信息。

**修改前的代码**:

```c
TupleDesc
WaitStmtResultDesc(WaitStmt *stmt)
{
    TupleDesc  tupdesc;

    /* 创建一个单列表描述符 */
    tupdesc = CreateTemplateTupleDesc(1);

    /* 问题在这里：TupleDescInitEntry 会访问 syscache */
    TupleDescInitEntry(tupdesc, (AttrNumber) 1, "status",
                       TEXTOID, -1, 0);

    TupleDescFinalize(tupdesc);
    return tupdesc;
}
```

### 1.3 TupleDescInitEntry vs TupleDescInitBuiltinEntry

这两个函数都能初始化元组描述符，但行为截然不同：

#### TupleDescInitEntry（修改前使用）

```c
void
TupleDescInitEntry(TupleDesc desc,
                   AttrNumber attributeNumber,
                   const char *attributeName,
                   Oid oidtypeid,
                   int32 typmod,
                   int attdim)
{
    HeapTuple  tuple;
    Form_pg_type typeForm;

    // 1. 从 pg_type syscache 中查找类型信息
    tuple = SearchSysCache1(TYPEOID, ObjectIdGetDatum(oidtypeid));
    if (!HeapTupleIsValid(tuple))
        elog(ERROR, "cache lookup failed for type %u", oidtypeid);

    typeForm = (Form_pg_type) GETSTRUCT(tuple);

    // 2. 使用查找到的信息填充元组描述符
    att->atttypid = oidtypeid;
    att->attlen = typeForm->typlen;
    att->attbyval = typeForm->typbyval;
    // ... 其他字段 ...

    // 3. 释放 syscache
    ReleaseSysCache(tuple);
}
```

**关键问题**: `SearchSysCache1` 可能重新建立 catalog snapshot！

#### TupleDescInitBuiltinEntry（修改后使用）

```c
void
TupleDescInitBuiltinEntry(TupleDesc desc,
                          AttrNumber attributeNumber,
                          const char *attributeName,
                          Oid oidtypeid,
                          int32 typmod,
                          int attdim)
{
    // 直接根据 OID 查表初始化属性
    // 不访问 syscache！
    // 只支持内置类型（如 TEXTOID）

    switch (oidtypeid)
    {
        case TEXTOID:
            att->attlen = -1;        /* variable length */
            att->attbyval = false;
            att->attalign = TYPALIGN_INT;
            att->attstorage = TYPSTORAGE_EXTENDED;
            // ...
            break;
        default:
            elog(ERROR, "unsupported type %u", oidtypeid);
    }
}
```

**优势**: 完全不访问 syscache，不会有任何 catalog snapshot 重新建立的风险。

### 1.4 为什么 syscache 访问会导致恢复冲突？

问题的根源在于 PostgreSQL 的 catalog snapshot 机制：

```
┌─────────────────────────────────────────────────────────────────┐
│                    Catalog Snapshot 管理                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 事务开始时获取 catalog snapshot                              │
│                                                                  │
│  2. WaitStmtResultDesc 可能在 snapshot 已释放之后被调用          │
│     (源码注释: may be called after snapshots have been released)  │
│                                                                  │
│  3. 如果此时调用 TupleDescInitEntry → SearchSysCache            │
│                                                                  │
│  4. SearchSysCache 可能重新建立 catalog snapshot                 │
│                                                                  │
│  5. 在 Standby 上，这可能导致 recovery conflict                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**在热 Standby 上的影响**:

当 SearchSysCache 重新建立 catalog snapshot 时，它会让 Standby 认为有新的查询正在访问 catalogs，这可能与 Standby 的恢复过程冲突，导致 "recovery conflict" 错误。

## 二、代码层面的关键变更

### 2.1 变更内容

**文件**: `src/backend/commands/wait.c`

**修改前**:
```c
TupleDesc
WaitStmtResultDesc(WaitStmt *stmt)
{
    TupleDesc  tupdesc;

    tupdesc = CreateTemplateTupleDesc(1);
    TupleDescInitEntry(tupdesc, (AttrNumber) 1, "status",
                        TEXTOID, -1, 0);
    TupleDescFinalize(tupdesc);
    return tupdesc;
}
```

**修改后**:
```c
TupleDesc
WaitStmtResultDesc(WaitStmt *stmt)
{
    TupleDesc  tupdesc;

    tupdesc = CreateTemplateTupleDesc(1);
    TupleDescInitBuiltinEntry(tupdesc, (AttrNumber) 1, "status",
                              TEXTOID, -1, 0);
    TupleDescFinalize(tupdesc);
    return tupdesc;
}
```

### 2.2 为什么可以安全替换？

`TupleDescInitBuiltinEntry` 只支持内置类型。在 WAIT FOR 命令中：

- 返回的 "status" 列类型是 `TEXT`（`TEXTOID`）
- TEXT 是内置类型，`TupleDescInitBuiltinEntry` 完全支持

这意味着：
1. 功能完全相同
2. 性能更好（无 syscache 访问）
3. 避免潜在的恢复冲突问题

### 2.3 支持的内置类型

根据 `TupleDescInitBuiltinEntry` 的实现，支持的类型包括：

| 类型 OID | 类型名称 | 说明 |
|----------|---------|------|
| `TEXTOID` | text | 可变长度文本 |
| `TEXTARRAYOID` | text[] | 文本数组 |
| `BOOLOID` | bool | 布尔值 |
| `INT4OID` | int4 | 32位整数 |
| `INT8OID` | int8 | 64位整数 |
| `OIDOID` | oid | 对象标识符 |

WAIT FOR 命令只使用 `TEXTOID`，完全在支持范围内。如果传入其他不支持的类型（如 `NUMERICOID`），会报错：`ERROR: unsupported type <具体OID>`

## 三、应用场景及最佳实践

### 3.1 场景一：主从复制环境中的进度等待

**背景**: 某电商系统使用流复制进行读写分离，主库写入后需要确保数据已复制到 Standby 才返回成功。

**使用 WAIT FOR**:

```sql
-- =============================================
-- 【主库侧·会话 A】
-- =============================================
INSERT INTO orders (customer_id, amount) VALUES (123, 999.99);

-- 获取写入产生的 LSN（需将此 LSN 传到 Standby 侧）
SELECT pg_current_wal_lsn() AS lsn;
-- 结果可能是: 0/306EE20
-- 将 0/306EE20 交给 Standby 侧执行 WAIT FOR

-- =============================================
-- 【Standby 侧·会话 B】
-- =============================================
-- 等待指定 LSN 被主库发送过来并完成回放
WAIT FOR LSN '0/306EE20' WITH (MODE 'standby_replay');
```

> **注意**：`WAIT FOR` 必须在 **Standby 侧**（或逻辑复制订阅侧）执行，`MODE 'standby_replay'` 才会正确等待本地 WAL 回放进度。将 `WAIT FOR` 放在主库本机执行毫无意义。

**优化效果**:

| 阶段 | 优化前 | 优化后 |
|------|--------|--------|
| 构建元组描述符 | 访问 syscache | 无 syscache 访问 |
| Standby 恢复冲突风险 | 存在 | 消除了由该路径引入的一类冲突风险 |
| 等待时间 | 可能因冲突延长 | 稳定 |

### 3.2 场景二：基于 WAIT FOR 监控复制延迟

通过尝试等待一个较旧的 LSN，可以检测复制是否滞后：

```sql
-- 获取当前 LSN
SELECT pg_current_wal_lsn() AS current_lsn;
-- 假设备份节点返回: 0/306EE20

-- 尝试等待一个稍旧的 LSN（如果已回放则立即返回，超时则说明滞后）
WAIT FOR LSN '0/306EE20' WITH (MODE 'standby_replay', TIMEOUT '1s');

-- 如果执行成功，说明该 LSN 已在 Standby 回放
-- 如果超时，说明复制存在延迟
```

## 四、相关知识：PostgreSQL 的 Snapshot 机制

### 4.1 Snapshot 的作用

Snapshot 用于确保事务看到一致的数据视图。在 PostgreSQL 中：

- **MVCC snapshot**: 保证事务看到一致的数据
- **Catalog snapshot**: 保证 DDL 操作看到一致的 catalog 状态

### 4.2 Recovery Conflict 的原因

在 Standby 上执行查询时：

```
┌──────────────────────────────────────────────────────────────────┐
│                     Standby 上的 Recovery Conflict                │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│  1. Standby 持续在恢复模式（recovery）                            │
│                                                                   │
│  2. 用户查询开始在 Standby 上执行                                │
│     ↓                                                             │
│  3. 查询访问 syscache（如 TupleDescInitEntry）                   │
│     ↓                                                             │
│  4. syscache 访问重新建立了 catalog snapshot                      │
│     ↓                                                             │
│  5. 这个 snapshot 与 recovery 过程冲突                           │
│     ↓                                                             │
│  6. Recovery conflict 导致查询被取消                             │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### 4.3 何时需要避免 syscache 访问

以下场景需要避免 syscache 访问：

1. **WAIT FOR 命令**: 在释放 snapshot 后执行
2. **Standby 上的查询**: 需要尽量减少与 recovery 的冲突
3. **关键路径上的 catalog 访问**: 当返回列类型是内置类型，且代码路径必须避免 syscache 时，可优先使用 `TupleDescInitBuiltinEntry`

## 五、小结和思考

### 5.1 核心价值总结

本次提交 `834038c` 是一个精确的性能和稳定性优化：

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| syscache 访问 | 有（TupleDescInitEntry） | 无（TupleDescInitBuiltinEntry） |
| Catalog snapshot 重建风险 | 存在 | 消除 |
| Recovery conflict 风险 | 可能发生 | 消除了由该路径引入的一类风险 |
| 代码行数变化 | - | 仅增加注释，实际代码微调 |

### 5.2 对 DBA 的实际意义

虽然这是一个很小的代码变更，但对于运行以下环境的 DBA 来说很重要：

1. **高可用主从复制环境**: 确保 WAIT FOR 命令不会触发意外的恢复冲突
2. **热 Standby**: 减少 Standby 上的查询被取消的可能性
3. **对复制延迟敏感的应用**: 确保等待逻辑稳定可靠

### 5.3 最佳实践建议

1. **在复制协调逻辑中使用 WAIT FOR**:
   ```sql
   -- 第一步：获取当前 LSN
   SELECT pg_current_wal_lsn();
   -- 结果可能是: 0/306EE20

   -- 第二步：将上述 LSN 作为字符串传入 WAIT FOR
   -- 确保数据已复制后再做后续操作
   WAIT FOR LSN '0/306EE20' WITH (MODE 'standby_replay');
   ```

2. **监控 Standby 的恢复冲突**:
   ```sql
   -- 查看恢复冲突统计（pg_stat_database_conflicts 是视图，不接受参数）
   SELECT d.datname, c.*
   FROM pg_stat_database_conflicts c
   JOIN pg_database d ON c.datid = d.oid
   WHERE d.datname = current_database();
   ```

3. **合理设置 hot_standby_feedback**:
   ```sql
   -- 在 Standby 上启用反馈，让主库知道 Standby 的状态
   -- 它可能减少部分恢复冲突，但会增加主库膨胀风险
   ```

### 5.4 未来展望

这类优化可能进一步扩展到其他命令和场景：

1. **其他可能受影响的地方**: 检查是否有类似的 snapshot 问题
2. **更多的内置类型支持**: `TupleDescInitBuiltinEntry` 可能支持更多内置类型
3. **文档增强**: 明确说明 WAIT FOR 在 Standby 上的行为
