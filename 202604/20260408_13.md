# PostgreSQL 19 最值得期待的特性  
  
如果只看这段 PostgreSQL 19 开发周期里的 commit message，而不去展开代码实现细节，有一个很清晰的结论：PostgreSQL 19 的重点不是单点小修小补，而是同时在 SQL 表达力、逻辑复制、分区管理、在线运维和可观测性上继续往前推了一大步。对应用开发者来说，最值得关注的是 SQL 语义越来越强，很多以前要绕路写的逻辑，现在开始有更直接的原生命令；对 DBA 来说，复制一致性、在线整理、autovacuum 调优和 standby 观测这几个长期痛点，都看到了更接近生产现实的增强。  
  
需要先说明一点：下面的判断严格依据这些提交的完整描述文本，不分析 diff，因此重点放在“这个特性想解决什么问题、会带来什么变化、适合谁使用”。  
  
## 一、SQL 能力继续增强，PostgreSQL 正在减少“绕路写法”  
  
### 1. `GROUP BY ALL` 让探索式查询更顺手  
提交 `ef38a4d9` 引入 `GROUP BY ALL`。它的语义是：自动把 `SELECT` 目标列表里那些不含聚合函数、也不含窗口函数的表达式，加入到 `GROUP BY` 列表中。  
这项特性的意义不在于“以前做不到”，而在于“以前必须手写、现在可以少写”。对于交互式分析、临时统计、BI 查询原型设计，这会显著降低 SQL 的样板代码量。  
  
更值得注意的是，commit message 里提到 SQL 标准委员会已经接受了这个语法。这意味着它不是一个随意拼出来的小语法糖，而是有标准化趋势的能力。对于 PostgreSQL 来说，这种兼顾易用性和标准演进方向的改进，通常都很有长期价值。  
  
### 2. 窗口函数支持 `IGNORE NULLS / RESPECT NULLS`  
提交 `25a30bbd` 为 `lead`、`lag`、`first_value`、`last_value`、`nth_value` 增加了 `IGNORE NULLS/RESPECT NULLS`。  
这看似只是窗口函数补了一块拼图，但对分析型 SQL 的实际影响很大。因为很多业务数据天然含空值，比如传感器采样、用户行为流、财务流水、宽表画像字段。以前用户往往需要用更复杂的表达式或子查询去模拟“跳过空值”的窗口计算，现在可以直接用语义明确的标准写法。  
  
提交说明还特别强调了性能问题：当启用 `IGNORE NULLS` 时，窗口函数可能要反复回看同一批行去寻找非空值，因此实现里加入了专门的缓存信息来减少重复求值。换句话说，这不是只加语法、不管代价，而是考虑到了真正进入生产分析场景后的执行成本。  
  
### 3. `INSERT ... ON CONFLICT DO SELECT` 扩展了 upsert 场景  
提交 `88327092` 新增 `INSERT ... ON CONFLICT DO SELECT [FOR UPDATE/SHARE]`。  
这是一个很有 PostgreSQL 风格的增强：不是简单把 `ON CONFLICT` 再多塞一种行为，而是把“冲突时直接返回已存在行”变成了原生能力。  
  
它解决的核心问题是：应用在做幂等写入时，经常希望“如果插入成功就返回新行，如果冲突了就直接拿到旧行”。以前通常需要两步甚至多步实现，还要自己处理并发条件。现在 commit message 给出的方向很明确：冲突时可以直接返回预存行，甚至还能在返回前加锁。  
这对高并发业务、去重写入、资源唯一化创建、任务幂等提交都非常实用。  
  
### 4. `COPY FROM` 新增 `ON_ERROR SET_NULL`  
提交 `2a525cc9` 为 `COPY FROM` 增加 `ON_ERROR SET_NULL`。  
这项特性对数据导入链路尤其重要。它的语义不是“所有错误都忽略”，而是更克制、更实用的设计：当字段类型转换失败时，把该列置为 `NULL`，但依然继续遵守 `NOT NULL` 约束和 domain 的 `NOT NULL` 规则。也就是说，它增强的是导入韧性，不是放弃数据约束。  
  
这很适合现实世界的半脏数据场景。很多数据接入任务真正需要的不是“严格失败”或“粗暴跳过整行”，而是“把坏字段变成缺失值，让整批数据先导入，再做后续治理”。从 commit description 来看，这项能力的定位非常明确，就是把 PostgreSQL 的 `COPY` 推向更实用的 ETL 工具。  
  
### 5. 分区表终于可以直接 `COPY TO`  
提交 `4bea91f2` 增加了对分区表直接执行 `COPY TO` 的支持。  
以前用户要导出分区表，常常需要写成 `COPY (SELECT ...) TO ...`，不仅更啰嗦，还会引入额外的查询处理开销。这个提交明确指出，新实现既提升可用性，也提升性能，因为它避免了通过查询层再把结果交回 `COPY TO` 的那条路径。  
  
对于已经大量使用分区表的业务系统，这个变化很朴素，但非常实用。它说明 PostgreSQL 19 继续在把“分区表像普通表一样好用”这件事做实。  
  
### 6. `UPDATE/DELETE FOR PORTION OF`：面向时态数据的原生 DML  
提交 `8e72d914` 是这一批里最有“数据库产品方向感”的特性之一。它为 `UPDATE` 和 `DELETE` 引入了 `FOR PORTION OF`，允许针对 range 或 multirange 列表示的时间区间，只修改其中一段有效期。  
  
它的关键不是“更新范围类型列”，而是：数据库会自动识别哪些记录和目标时间段重叠，只修改那段历史，并在需要时自动生成剩余时间段对应的“leftover”记录。  
这实际上是在把应用层经常自己手写的时态拆分逻辑，往数据库内核下沉。  
  
对于合约有效期、价格版本、雇佣关系、资费规则、主数据历史版本这样的模型，这会非常有价值。它意味着 PostgreSQL 在时态数据处理上又往前走了一步，不再只是“支持 range/multirange 类型”，而是开始提供直接面向业务语义的更新删除能力。  
  
### 7. `jsonpath` 新增更多字符串方法  
提交 `bd4f879a` 为 `jsonpath` 增加了 `trim`、`lower`、`upper`、`initcap`、`replace`、`split_part`` 等方法。  
这项增强的意义在于：JSON 查询表达式内部可以做更多就地文本处理，减少退出 `jsonpath` 再回到 SQL 函数层处理的需要。  
  
对 heavily JSON 的应用来说，这种增强虽然不像新命令那样显眼，但很能提升表达一致性。尤其当过滤逻辑已经在 `jsonpath` 中时，多出这些字符串方法会让条件表达式更完整、更易维护。  
  
## 二、逻辑复制补齐了 sequence 这块长期短板  
  
### 8. publication 支持 `FOR ALL SEQUENCES`  
提交 `96b37849` 为 publication 增加 `FOR ALL SEQUENCES`。  
这件事的重要性在于，逻辑复制过去主要围绕表，而 sequence 经常是“半脱节”的。提交说明直接点明它对升级很有用，这非常现实：很多升级、迁移、切换场景里，sequence 一致性才是最后容易出事故的地方。  
  
除了语法本身，这个提交还补了配套可见性，例如 `psql` 展示和 `pg_publication_sequences` 视图。说明它不是一个孤立开关，而是在往“可配置、可查看、可运维”这个方向推进。  
  
### 9. subscription 新增 `REFRESH SEQUENCES`  
提交 `f0b3573c` 引入 `ALTER SUBSCRIPTION ... REFRESH SEQUENCES`。  
它的意义是把 sequence 映射刷新从隐式副作用，变成 DBA 可以显式触发的动作。并且 commit message 还说明，其他与 publication 变更相关的订阅命令也会自动刷新 sequence 映射，保证订阅端跟发布端状态对齐。  
  
这表明 PostgreSQL 19 对 sequence 复制的思路不是单点修补，而是在做一整套闭环：publication 端能声明、subscription 端能刷新、后续还有专门同步 worker。  
  
### 10. 逻辑复制正式支持 sequence synchronization  
提交 `5509055d` 是上面这条线里最关键的落地。  
commit message 很完整地描述了 sequence 同步机制：sequence 有 `INIT` 和 `READY` 两种状态；有专门的 `sequencesync worker` 批量拉取发布端的 sequence 值与 page LSN；同步成功后标记为 `READY`；它会在 `CREATE SUBSCRIPTION`、`REFRESH PUBLICATION`、`REFRESH SEQUENCES` 等场景被触发。  
  
这说明 PostgreSQL 19 正在真正把 sequence 纳入逻辑复制的一等公民。  
对 DBA 来说，这项能力的价值非常直接：逻辑复制链路更完整，升级、双写切换、迁移演练时少了一个需要额外补丁和人工校准的环节。  
  
## 三、分区管理终于开始走向“内建重组”  
  
### 11. `ALTER TABLE ... MERGE PARTITIONS`  
提交 `f2e4cc42` 增加分区合并。  
### 12. `ALTER TABLE ... SPLIT PARTITION`  
提交 `4b3d1736` 增加分区拆分。  
  
这两个特性应该放在一起看。它们本质上是在告诉用户：PostgreSQL 的分区管理开始提供真正面向生命周期治理的内建命令。  
以前很多分区重划分操作要靠新建分区、插搬数据、切换路由、删旧分区等多步流程手工完成。现在至少已经有了原生命令入口。  
  
当然，commit message 也非常坦率：当前实现比较朴素，单进程执行，而且会在整个操作期间持有父表的 `ACCESS EXCLUSIVE LOCK`，因此并不推荐在高负载的大型分区表上直接使用。  
这点很关键，因为它说明 PostgreSQL 19 在这个方向上更像是“建立官方能力模型”，而不是一步到位解决所有在线场景。即便如此，这仍然是非常值得期待的特性，因为一旦命令语义进入核心，后续优化锁粒度、并行化、在线化就有了清晰基础。  
  
## 四、在线运维能力的升级非常明显  
  
### 13. `WAIT FOR`：standby 一致性等待终于有了原生命令  
提交 `447aae13` 实现 `WAIT FOR`。  
它的目标场景非常明确：用户在 primary 上做完数据变更后，希望在 standby 上读取时能确定这些变化已经 replay。这个需求在读写分离、只读副本读一致性、切换前校验、自动化脚本中都非常常见。  
  
commit message 的价值不只在“新增命令”，还在解释了为什么它必须是 utility command，而不能简单做成函数。因为等待时不能持有 snapshot，否则 snapshot 可能阻碍 WAL 回放，形成一种自我死锁。  
这说明 PostgreSQL 团队不是在表层做个接口，而是认真处理了 recovery 机制与 SQL 执行模型之间的边界条件。  
  
对 DBA 来说，这是非常实用的一步。它让很多过去需要轮询函数、猜状态、靠脚本拼接的逻辑，开始拥有原生、可预期的等待语义。  
  
### 14. 引入 `REPACK`，统一 `VACUUM FULL` 和 `CLUSTER` 的语义入口  
提交 `ac58465e` 引入 `REPACK`。  
这是 PostgreSQL 19 很有产品味道的一项变化。commit message 的核心观点很清楚：`VACUUM FULL` 和 `CLUSTER` 都是“重写表”的手段，但语义入口分裂，而且 `CLUSTER` 这个词本身在业界语义过载严重。于是 PostgreSQL 选择新引入 `REPACK`，把这两类能力统一到一个更容易理解的命令之下。  
  
这其实不只是换个名字，而是在重组用户心智模型。  
对于 DBA 而言，表重组、物理整理、按索引重写、空间回收，本来就是一类事情。`REPACK` 把这些事收敛到同一个入口，为后续继续增加模式打下了基础。  
  
### 15. `REPACK CONCURRENTLY` 把在线重组推向生产可用方向  
提交 `28d534e2` 为 `REPACK` 增加 `CONCURRENTLY`。  
这是这批 commit 里最能体现“PostgreSQL 正在认真面对生产环境”的一项能力。它的基本思路是：在生成新表副本阶段不拿 `ACCESS EXCLUSIVE LOCK`，而是先用较弱锁级别和 MVCC snapshot 建立初始副本，同时通过逻辑解码记录并回放期间发生的并发变更，最后在切换 relfilenode 的瞬间才短暂拿强锁。  
  
这套设计非常像成熟在线 schema/data maintenance 工具的思路。提交说明也很坦率地列出了当前限制：  
只支持消耗一个 replication slot；全系统同一时刻只能跑一个 concurrent repack；最终锁升级仍可能有死锁风险。  
但即便如此，这已经不是概念验证级别，而是明显朝着“可在生产上逐步采用”的方向走。  
  
如果 PostgreSQL 19 最终把这条线打磨好，它对大表维护的影响会非常大。  
  
### 16. autovacuum 开始支持 parallel vacuum workers  
提交 `1ff3180c` 允许 autovacuum 使用并行 vacuum worker。  
这项能力非常务实。过去即使手工 `VACUUM` 已经能利用 parallel vacuum，autovacuum 依然禁用了这条路径，导致系统最关键的后台维护机制在大索引、多索引表上性能受限。现在 PostgreSQL 19 允许 autovacuum 也走同样的并行基础设施。  
  
更重要的是，这不是“默认激进打开”，而是给了清晰的控制面：  
有实例级 GUC `autovacuum_max_parallel_workers`，也有表级存储参数 `autovacuum_parallel_workers`。  
这种设计对 DBA 很友好，因为它允许先按业务表逐步试点，而不是一刀切。  
  
### 17. `pg_stat_autovacuum_scores` 让 autovacuum 调优更可见  
提交 `87f61f0c` 新增系统视图 `pg_stat_autovacuum_scores`。  
它为当前数据库中的每张表显示 autovacuum score，并说明 autovacuum 当前会不会 vacuum / analyze 这张表。  
  
这项特性的价值不在“多了个视图”，而在于它把 autovacuum 的内部决策更直接地暴露了出来。  
长期以来，autovacuum 是否命中、为什么还没跑、哪个表更接近阈值，很多时候需要 DBA 根据统计字段和阈值自己推算。现在 PostgreSQL 把这个决策过程显式可视化了，这会明显降低 autovacuum 调优门槛。  
  
## 五、可观测性继续强化，而且更贴近实际排障  
  
### 18. `pg_stat_recovery`：standby 恢复状态终于有了统一视图  
提交 `01d485b1` 增加系统视图 `pg_stat_recovery`。  
这个视图的设计非常有 DBA 导向：把 recovery 过程中的多个关键状态一次性暴露出来，包括最近成功回放的 WAL 记录边界、当前正在回放的位置、当前 WAL chunk 时间、promotion trigger 状态、最新 commit/abort 时间戳、recovery pause 状态等。  
  
commit message 里有一个很重要的点：虽然其中一部分信息原来能通过不同函数拿到，但新的视图可以通过一次共享内存读取拿到一组一致性的状态快照。这对诊断 standby 延迟、恢复卡顿、是否暂停、何时可提升为主库，都非常重要。  
  
### 19. `EXPLAIN (IO)` 开始进入 PostgreSQL  
提交 `681daed9` 增加 `EXPLAIN (IO)` 基础设施，并率先支持 `BitmapHeapScan`。  
如果说 PostgreSQL 最近几年一直在完善 `EXPLAIN (ANALYZE, BUFFERS, WAL)`，那么这个提交意味着它开始进一步走向 I/O 路径级别的可观测性，尤其关注 AIO / prefetch / I/O requests。  
  
commit message 明确说明目前统计信息还比较克制，只在 `EXPLAIN ANALYZE` 且显式加 `IO` 选项时才采集，并且先从 `ReadStream` 驱动的扫描节点入手。这很符合 PostgreSQL 的一贯风格：先把模型做对、接口立住，再慢慢扩展覆盖面。  
  
对于性能分析人员来说，这很值得期待。因为很多查询瓶颈不只是“读了多少 buffer”，而是“预取是否有效、I/O 请求行为如何、扫描器与底层读流的协同怎样”。`EXPLAIN (IO)` 显然在往这个方向铺路。  
  
### 20. `log_autoanalyze_min_duration` 让 autovacuum 日志更精细  
提交 `dd3ae378` 引入 `log_autoanalyze_min_duration`。  
它看上去不如前面几项“酷”，但对实际运维很有价值。原因是之前 `log_autovacuum_min_duration` 同时作用于 autovacuum 的 `VACUUM` 和 `ANALYZE`，两者无法分开设阈值。现实里 DBA 往往更关心某一类行为，而不是把两者混在一起看。  
  
把 autoanalyze 单独拆出来，本质上是让自动统计信息维护的观测更可控、更精确。对于分析 planner 问题、统计信息更新频率、系统后台维护噪音，这都是很实在的改进。  
  
## 六、怎么理解 PostgreSQL 19 这一波变化  
  
如果把这些 commit message 放在一起看，PostgreSQL 19 的方向其实非常一致：  
  
第一，它在继续增强 SQL 语言本身。`GROUP BY ALL`、窗口函数 `IGNORE NULLS`、`ON CONFLICT DO SELECT`、`FOR PORTION OF`、`jsonpath` 字符串方法，这些都不是只给专家用户准备的底层能力，而是直接影响普通应用开发者写 SQL 的方式。  
  
第二，它在补齐逻辑复制的完整性。sequence 一直是逻辑复制中相对麻烦的部分，而 PG19 这波提交从 publication、subscription、refresh 命令到专门 worker，已经形成了一条完整链路。这对升级、双活切换、低停机迁移都有现实价值。  
  
第三，它在认真改善 DBA 的“生产操作体验”。`WAIT FOR`、`REPACK`、`REPACK CONCURRENTLY`、parallel autovacuum、`pg_stat_recovery`、`pg_stat_autovacuum_scores`、`EXPLAIN (IO)`，这些都直指生产环境里最常见的几个问题：如何安全等待副本、如何在线整理大表、如何让 autovacuum 更快更可控、如何更快看懂恢复和 I/O 行为。  
  
第四，它也保持了 PostgreSQL 一贯的工程取向：很多 commit message 都会明确写出现阶段限制，不把特性包装成“万能”。比如分区 merge/split 目前锁很重，`REPACK CONCURRENTLY` 目前还有 replication slot 和并发限制，`EXPLAIN (IO)` 先只覆盖部分 scan 节点。这种克制反而说明这些特性更可信，因为它们是可用的第一步，而不是夸张宣传。  
  
## 结语  
  
如果要从这批提交里提炼一句话来概括 PostgreSQL 19，那就是：  
  
**它不是只在“再加几个 SQL 小语法”，而是在把 PostgreSQL 往“更强的应用数据库 + 更成熟的生产数据库”两个方向同时推进。**  
  
对开发者来说，最值得期待的是更自然的 SQL 表达方式；对 DBA 来说，最值得期待的是 sequence 逻辑复制、在线 `REPACK`、并行 autovacuum、恢复与 I/O 观测这几条主线。如果这些特性能以当前 commit message 所描述的形态稳定进入正式版，那么 PostgreSQL 19 会是一版非常有实战价值的升级。
