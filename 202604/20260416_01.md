## 先说结论  
  
ORC 的核心价值不是“把数据按列放起来”这么简单，而是把 **类型、编码、压缩、统计信息、跳读能力** 一起固化进文件格式。  
  
如果说普通列存解决的是“只读需要的列”，那么 ORC 进一步解决的是：  
  
- 只读需要的列  
- 只读可能命中条件的 stripe / row group  
- 针对不同类型选择不同编码  
- 用文件自描述信息降低外部元数据依赖  
- 在 Hive/Spark/Presto/Trino 等大数据生态里作为高效分析格式流转  
  
一句话： **ORC 是面向大规模分析扫描的“带类型语义、带索引统计、带压缩编码策略”的列式文件格式。**  
  
它适合 OLAP、数据湖、归档分析、批量扫描、按时间/租户/状态过滤的大表查询；不适合高频单行 OLTP、小事务随机更新、每秒百万级事务这种场景。  
  
---  
  
## 一、药引子：为什么只有行存不够？  
  
应用开发者经常问一个问题：数据库不就是把数据存进去、查出来吗？为什么还要关心行存、列存、Parquet、ORC、Arrow、Lance、Vortex？  
  
因为“查出来”这三个字背后，成本完全不同。  
  
假设一张日志表有 200 个字段，你的查询只关心：  
  
```sql  
SELECT app_id, count(*), avg(latency)  
FROM logs  
WHERE log_time >= now() - interval '1 day'  
  AND status = 500  
GROUP BY app_id;  
```  
  
行存的天然问题是：一行里的 200 个字段往往一起被读进来。即使查询只需要 4 个字段，也可能被迫搬运大量无关数据。  
  
列存的第一性原理很简单：  
  
> 查询只需要哪些列，就只读取哪些列；过滤能在存储层提前判断，就不要把无关数据搬到计算层。  
  
ORC 正是沿着这个方向做到了比较彻底：列裁剪、统计信息、谓词下推、编码压缩、批量向量化读取，组合起来才是它的价值。  
  
---  
  
## 二、ORC 是什么？  
  
ORC，全称 Optimized Row Columnar，是 Apache 生态里的列式文件格式，最早来自 Hive 场景。它的目标不是替代数据库表，而是在 Hadoop / 数据湖 / 大数据分析生态中，高效存储和扫描结构化数据。  
  
ORC 文件是 self-describing 的：文件内包含类型、编码、统计信息等元数据，不完全依赖 Hive Metastore 才能解释内容。  
  
从官方规格看，ORC 的几个关键点是：  
  
- 保留表定义里的类型信息，不像早期 RCFile 那样把列当二进制 blob  
- 使用类型相关编码，例如 dictionary encoding、bit packing、delta encoding、run length encoding  
- 在轻量编码之上还能叠加通用压缩，例如 zlib、Snappy、LZO、LZ4、ZSTD  
- 支持列投影，只读当前查询需要的列  
- 内置轻量索引和 min/max 统计，用于跳过无关 row group / stripe  
  
真正要理解 ORC，不要只记“列存”。要记住这条链路：  
  
```text  
类型语义 -> 编码策略 -> 压缩块 -> stripe/row group 统计 -> 谓词下推 -> 少读 IO -> 少解压 -> 少计算  
```  
  
---  
  
## 三、ORC 文件结构：Header、Body、Tail  
  
ORC 文件可以粗略理解成三段：  
  
```text  
ORC File  
├── Header  
│   └── magic: "ORC"  
├── Body  
│   ├── Stripe 0  
│   │   ├── Index streams  
│   │   ├── Data streams  
│   │   └── Stripe footer  
│   ├── Stripe 1  
│   └── ...  
└── Tail  
    ├── File metadata / stripe statistics  
    ├── Footer  
    ├── Postscript  
    └── psLen  
```  
  
为什么 Tail 在文件末尾？  
  
因为 HDFS 这类文件系统偏向 write-once，写完主体数据后，再把整体元数据写到文件尾部。读取时，ORC reader 会从文件末尾反向读取 Postscript 和 Footer，拿到压缩方式、Footer 长度、stripe 列表、schema、行数、列统计等信息。  
  
Footer 很关键，它包含：  
  
- stripes 列表  
- schema 类型树  
- 总行数  
- 文件级列统计信息  
- row index stride  
- writer 信息  
- 加密相关信息  
  
这也是 ORC 能做“先判断要不要读，再决定读哪里”的基础。  
  
---  
  
## 四、Stripe：ORC 的大块组织单位  
  
ORC 的 Body 被切成多个 stripe。每个 stripe 是相对自包含的读取单元，包含完整行，不会让一行跨 stripe。  
  
一个 stripe 内部又分成：  
  
```text  
Stripe  
├── Index section  
│   └── 各列 row group index / bloom filter 等  
├── Data section  
│   └── 各列的数据流 streams  
└── Stripe footer  
    └── stream 位置、编码信息等  
```  
  
这个设计的好处是：  
  
- 查询只需要某些列时，只读这些列对应的 streams  
- 查询带过滤条件时，先看 stripe / row group 统计信息  
- 不命中的 stripe 或 row group 直接跳过  
- 每个 stripe 相对独立，适合大规模并行扫描  
  
从数据库视角看，stripe 类似“列式大块 + 元数据索引”的组合。它不是 B-tree 那种精准定位索引，而是 OLAP 里更常用的 **metascan / skip scan** 思路：用便宜的元数据过滤掉大量不可能命中的数据块。  
  
这和德哥早年在 `数据库metascan - 存储级、块级、batch级过滤与数据编排` 里讲的思路一致：不一定给所有列建重索引，但可以用块级、batch 级统计信息减少扫描量。  
  
---  
  
## 五、三级索引：file、stripe、row group  
  
ORC 的索引不是传统数据库里的二级索引，而是嵌在文件里的统计和位置元数据。  
  
官方文档把 ORC 的索引分成三层：  
  
| 层级 | 粒度 | 作用 |  
|------|------|------|  
| file level | 整个文件 | 判断整个文件是否可能命中 |  
| stripe level | 每个 stripe | 判断某个 stripe 是否需要读 |  
| row level | stripe 内每 10000 行一组 | 进一步跳过 row group |  
  
列统计通常包含：  
  
- value count  
- 是否有 NULL  
- min / max  
- numeric sum  
- 部分场景下 bloom filter  
  
例如查询：  
  
```sql  
WHERE log_time >= '2026-04-01'  
  AND log_time <  '2026-04-02'  
```  
  
如果某个 stripe 的 `log_time` 最大值都小于 `2026-04-01`，这个 stripe 就没有读取价值。这个判断只需要读元数据，不需要解压数据页。  
  
所以 ORC 的加速不是玄学，落到机制就是：  
  
```text  
过滤条件 -> SARG/Search Argument -> 对比统计信息 -> 跳过文件/stripe/row group -> 减少 IO 和解压  
```  
  
但是这里有一个前提：数据要有局部相关性。  
  
如果 `log_time` 是按时间写入的，min/max 非常有效；如果某列完全随机分布，每个 row group 的 min/max 都覆盖全域，skip index 基本失效。这也是列存调优里经常被忽视的一点： **数据写入顺序就是一种隐形索引。**  
  
---  
  
## 六、编码与压缩：为什么 ORC 文件可以小？  
  
ORC 不是简单把一列值连续写到磁盘，而是根据类型和数据分布选择编码。  
  
常见手段包括：  
  
| 编码 | 适合数据 | 价值 |  
|------|----------|------|  
| Dictionary encoding | 低基数字符串、枚举值 | 用字典 ID 替代重复字符串 |  
| Bit packing | 小范围整数 | 用更少 bit 表示一个值 |  
| Delta encoding | 单调递增/递减序列 | 存差值而不是存完整值 |  
| Run length encoding | 连续重复值 | 存值 + 重复次数 |  
| Varint / ZigZag | 小整数、有符号整数 | 减少整数编码空间 |  
  
在这些轻量编码之上，ORC 还可以叠加通用压缩。官方规格列出的 CompressionKind 包括 NONE、ZLIB、SNAPPY、LZO、LZ4、ZSTD。  
  
这里要注意一个工程取舍：  
  
- 压缩块越大，压缩率通常越好  
- 压缩块越大，随机跳读和解压开销可能越高  
- 压缩块越小，过滤和随机读取更灵活，但元数据和压缩率可能受影响  
  
所以列存格式不是“压缩率越高越好”，而是要看查询路径：你到底是在全表批量扫，还是经常小范围过滤、交互式查询、随机抽样？  
  
---  
  
## 七、类型系统：ORC 为什么强调 type-aware？  
  
ORC 文件包含类型和编码信息，支持丰富的标量和复合类型：  
  
- boolean、tinyint、smallint、int、bigint  
- float、double  
- string、char、varchar  
- binary  
- decimal  
- timestamp、timestamp with local time zone、date  
- struct、list、map、union  
  
这件事对数据库人很重要。  
  
如果格式不知道类型，只把列当 bytes，就很难做正确的 min/max、sum、谓词下推、时间语义、decimal 精度处理、嵌套结构裁剪。ORC 的 type-aware 设计，让 writer 可以根据类型选择编码，也让 reader 可以根据类型解释统计信息。  
  
例如 timestamp 就不是一个简单字符串。ORC 文档区分了不带时区 timestamp 和 timestamp with local time zone。跨时区分析系统如果把时间语义搞错，压缩和扫描再快也没用，因为结果不可信。  
  
---  
  
## 八、ACID：ORC 支持事务，但不是 OLTP  
  
ORC 文档里有 ACID support，但要看清边界。  
  
Hive 场景下，ORC 是 write-once 文件格式。更新和删除不是在原文件里原地修改，而是通过 base files + delta files 表达 insert/update/delete，再通过 compaction 合并。  
  
大概机制是：  
  
```text  
base file  
+ delta files  
+ transaction list  
+ row id: originalTransaction, bucket, rowId  
+ compaction  
```  
  
这套机制适合数据湖里“批量写入、批量更新、分钟级流式提交、查询看到事务一致性”的需求，但官方也明确说它不是为 OLTP 设计的。它可以支持一个事务更新大量行，但不是为了每小时百万级小事务。  
  
这点非常关键：  
  
> ORC 的 ACID 是数据湖/分析系统语境下的 ACID，不是 PostgreSQL heap + WAL + MVCC 那种高并发 OLTP 路线。  
  
如果你要的是订单系统、账户余额、库存扣减，别因为看到 ACID 就把 ORC 当 OLTP 表存储。

## 九、PostgreSQL 里的 ORC 影子：cstore_fdw  
  
PostgreSQL 生态里有一个很有代表性的列存插件：`cstore_fdw`。  
  
需要先纠正一个常见误解：`cstore_fdw` 官方说法是它的数据布局 **inspired by ORC**，并不是直接使用标准 ORC 文件格式。因此不能简单理解为“PostgreSQL 直接用 ORC 文件当表”。  
  
`cstore_fdw` 的价值在于：它把 ORC 类似的列存思想带进 PostgreSQL FDW 框架：  
  
- 列投影：只读查询需要的列  
- 压缩：减少内存和磁盘体积  
- skip indexes：为 row group / block 存 min/max，用于跳过无关数据  
- 使用 PostgreSQL FDW API 和类型表示  
- 支持 PostgreSQL optimizer 统计信息  
- 使用外部表方式加载、查询  
  
典型使用方式：  
  
```sql  
-- postgresql.conf  
shared_preload_libraries = 'cstore_fdw'  
  
-- SQL  
CREATE EXTENSION cstore_fdw;  
  
CREATE SERVER cstore_server  
FOREIGN DATA WRAPPER cstore_fdw;  
  
CREATE FOREIGN TABLE customer_reviews  
(  
    customer_id text,  
    review_date date,  
    review_rating integer,  
    product_id text,  
    product_title text  
)  
SERVER cstore_server  
OPTIONS (  
    compression 'pglz',  
    stripe_row_count '150000',  
    block_row_count '10000'  
);  
  
COPY customer_reviews FROM '/data/customer_reviews.csv' WITH CSV;  
ANALYZE customer_reviews;  
```  
  
几个关键参数：  
  
| 参数 | 含义 | 取舍 |  
|------|------|------|  
| `filename` | 数据文件位置 | 不指定则放在 `$PGDATA/cstore_fdw` |  
| `compression` | `none` 或 `pglz` | 压缩省空间，但消耗 CPU |  
| `stripe_row_count` | 每个 stripe 行数，默认 150000 | 小了省内存，大了扫描更顺 |  
| `block_row_count` | 每个列块行数，默认 10000 | 大了压缩好，小了跳读更准 |  
  
限制也要看清：  
  
- 支持 `COPY`  
- 支持 `INSERT INTO cstore_table SELECT ...`  
- 不支持 `DELETE`  
- 不支持 `UPDATE`  
- 不支持单行 insert  
- 官方 README 已提示现代化列存能力已进入 Citus columnar table access method，`cstore_fdw` 更像历史路线和学习样本  
  
所以在今天做 PostgreSQL 列存选型，不建议新项目优先押 `cstore_fdw`。学习 ORC 思想可以看它；生产新系统更应该评估 Citus columnar、DuckDB/Parquet/ORC、Greenplum AOCO、或者专门的数据湖表格式。

## 十、ORC vs Parquet：怎么理解差异？  
  
这篇不是 Parquet 专题，但 ORC 经常和 Parquet 一起被问到。  
  
可以先用一个简化判断：  
  
| 维度 | ORC | Parquet |  
|------|-----|---------|  
| 起源生态 | Hive / Hadoop | Dremel / Hadoop 生态 |  
| 强项 | Hive 场景、内置统计索引、复杂类型、压缩编码 | 跨生态普及度、Spark/DuckDB/Arrow 工具链 |  
| 文件自描述 | 强 | 强 |  
| 列裁剪 | 支持 | 支持 |  
| 谓词下推 | 支持 | 支持 |  
| 常见用途 | Hive 数仓、数据湖分析 | 数据湖、跨引擎交换、湖仓事实标准之一 |  
  
不要抽象地问“ORC 和 Parquet 谁更快”。正确问题是：  
  
```text  
我的数据类型是什么？  
过滤列是否有局部相关性？  
查询是全表扫描还是选择性过滤？  
引擎是谁？Hive、Spark、Trino、DuckDB、Presto？  
压缩算法是什么？  
文件大小、stripe/row group/page 大小怎么设置？  
对象存储还是本地 SSD？  
```  
  
离开这些条件谈性能，基本都是口水仗。

## 十一、什么时候适合 ORC？  
  
适合：  
  
| 场景 | 原因 |  
|------|------|  
| Hive/Spark/Trino 等大数据分析 | 生态支持成熟 |  
| 大宽表，只查少量列 | 列裁剪收益明显 |  
| 时间序列日志、行为数据、账单流水 | 按时间写入，min/max 跳读有效 |  
| 冷热分层、归档分析 | 压缩和批量扫描友好 |  
| 数据湖批处理 | 文件格式开放，可跨引擎读取 |  
  
不适合：  
  
| 场景 | 原因 |  
|------|------|  
| 高频 OLTP | 文件格式不是事务数据库存储引擎 |  
| 单行高频更新/删除 | write-once + delta/compaction 模型不合适 |  
| 小文件高频写入 | 元数据和调度成本会放大 |  
| 过滤列分布随机 | min/max 跳读收益差 |  
| 低延迟点查 | B-tree/LSM/主键索引路线更合适 |

## 十二、怎么验证 ORC 是否适合你？  
  
不要只看格式介绍，拿自己的数据验证。  
  
建议至少做 5 组实验：  
  
### 1、列裁剪收益  
  
```sql  
-- 读所有列  
SELECT count(*) FROM read_orc('logs.orc');  
  
-- 只读少量列  
SELECT app_id, count(*)  
FROM read_orc('logs.orc')  
GROUP BY app_id;  
```  
  
观察 IO bytes、CPU time、wall time。  
  
### 2、过滤下推收益  
  
```sql  
SELECT count(*)  
FROM read_orc('logs.orc')  
WHERE log_time >= '2026-04-01'  
  AND log_time <  '2026-04-02';  
```  
  
如果数据按 `log_time` 写入，应该能看到明显跳读收益。  
  
### 3、随机列过滤失效场景  
  
```sql  
SELECT count(*)  
FROM read_orc('logs.orc')  
WHERE random_key = 'abc';  
```  
  
如果 `random_key` 在每个 stripe / row group 都分布很散，min/max 基本没法帮你。  
  
### 4、压缩算法对比  
  
同一份数据分别写成 Snappy、ZSTD、ZLIB 等配置，比较：  
  
- 文件大小  
- 写入耗时  
- 扫描耗时  
- CPU 使用率  
- 对象存储读取量  
  
### 5、文件粒度测试  
  
不要制造大量小 ORC 文件。小文件会让调度、元数据读取、对象存储请求成本变成瓶颈。大数据系统里，小文件问题经常比格式本身更致命。

## 十三、给 DBA / 架构师的选型建议  
  
如果你在做 PostgreSQL / PolarDB 应用系统，ORC 通常不是主表存储，而是分析链路的一环：  
  
```text  
PostgreSQL/PolarDB  
  -> CDC / ETL  
  -> 数据湖 ORC/Parquet  
  -> Spark/Trino/DuckDB/Presto  
  -> 报表 / 训练 / 归档 / 审计  
```  
  
如果你想在 PostgreSQL 内部体验列存：  
  
- 学习历史机制：看 `cstore_fdw`  
- 新项目列存表：优先看 Citus columnar access method 或其他仍活跃维护的方案  
- 本地分析：DuckDB 读 ORC/Parquet 很方便  
- 大规模湖仓：结合 Iceberg/Delta/Hudi 表格式，而不是只选一个文件格式  
  
真正的判断标准不是“这个格式先进不先进”，而是：  
  
```text  
数据如何写入？  
查询如何过滤？  
哪些列被投影？  
哪些统计能跳读？  
压缩节省的 IO 是否大于解压 CPU？  
引擎是否能把谓词真正下推到文件 reader？  
```  
  
把这几个问题答清楚，ORC 的价值就很清楚了。

## 小结  
  
ORC 值得学，不是因为它是“又一个列存格式”，而是它把列存的几个底层变量串起来了：  
  
- 列裁剪减少无关 IO  
- 类型语义支持正确编码和统计  
- stripe / row group 统计支持 skip scan  
- 编码 + 压缩降低存储和传输成本  
- Footer / Postscript 让文件自描述、可跳读  
- ACID 支持服务于数据湖批量变更，而不是 OLTP 小事务  
  
数据库筑基课学 ORC，重点不是背格式字段，而是理解一个生产判断：  
  
> 对 OLAP 来说，性能常常不是来自“算得更快”，而是来自“根本不读、不解压、不计算那些不可能命中的数据”。  
  
这也是列存、BRIN、cstore_fdw skip index、Parquet row group、ORC stripe 这些技术背后共同的第一性原理。
