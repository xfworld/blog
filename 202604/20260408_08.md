## 背景 

本文深入解读 PostgreSQL DB93032a7cbd30199429167e649e22fab89327ed 这个 commit，该 commit 解决了备库晋升（Promotion）过程中，逻辑复制槽同步（Logical Replication Slot Synchronization）worker 可能阻塞晋升进度的严重问题。

  

**背景痛点**：在 PostgreSQL 流复制环境中，备库晋升为主库时，需要确保所有复制槽（Replication Slot）同步工作已停止。以往实现中，startup 进程通过发送 SIGUSR1 信号通知 slotsync worker 退出，但在 worker 正阻塞等待主库响应（如网络故障场景）时，该信号无法中断其等待状态，导致整个晋升过程被无限期阻塞。

**价值与意义**：该 commit 引入专用进程信号机制（PROCSIG_SLOTSYNC_MESSAGE），确保在晋升触发时，slotsync worker 能立即从任何等待状态中脱离并退出，从而保证晋升操作能在合理时间内完成，提升了主备切换的可靠性。

---

## 原理介绍

### 1. 复制槽同步机制回顾

PostgreSQL 逻辑复制槽同步功能（Slot Sync）是 PostgreSQL 17 引入的特性，允许备库从主库同步逻辑复制槽定义。其核心组件包括：

- **Slot Sync Worker**：后台 worker，负责定期同步复制槽
- **pg_sync_replication_slots() 函数**：也可通过 SQL 函数触发同步
- **SlotSyncCtx**：共享内存结构，用于控制同步状态

关键共享变量：
- `syncing`：标识当前是否有进程正在同步
- `stopSignaled`：晋升触发标志
- `pid`：正在同步的进程 PID

### 2. 问题根源分析

**原实现逻辑**（修复前）：

```c
// ShutDownSlotSync() 函数中
if (sync_process_pid != InvalidPid)
    kill(sync_process_pid, SIGUSR1);  // 仅发送 SIGUSR1
```

**问题所在**：

slotsync worker 在等待主库响应时（如调用 `WaitLatch` 或阻塞在网络 I/O），SIGUSR1 信号默认行为是：
- 如果进程正在阻塞于 `sleep()`、`wait()` 等系统调用，信号可中断
- 如果进程正在阻塞于 `recv()`、管道 I/O 等，且未设置 SA_RESTART，则信号可中断
- 但如果 worker 正在某些内部等待循环中，SIGUSR1 不会立即唤醒它

在 `slotsync.c` 的 `SyncReplicationSlots()` 函数中，worker 可能阻塞在：

```c
// 等待主库响应
rc = WaitLatch(MyLatch, WL_LATCH_SET | WL_TIMEOUT, timeout, ...);
```

当网络故障时，这种等待可能持续较长时间，而 SIGUSR1 无法保证能立即中断。

### 3. 修复方案详解

**核心改动**：

1. **新增专用信号类型** `PROCSIG_SLOTSYNC_MESSAGE`（procsignal.h:39）

2. **新增中断处理函数**（slotsync.c）：
   - `HandleSlotSyncMessageInterrupt()`：在 procsignal handler 中调用，设置 `InterruptPending = true` 和 `SlotSyncShutdownPending = true`
   - `ProcessSlotSyncMessage()`：在 `ProcessInterrupts()` 中调用，执行实际的退出/报错逻辑

3. **修改信号发送方式**（slotsync.c:1825-1827）：
```c
// 修复后
SendProcSignal(sync_process_pid, PROCSIG_SLOTSYNC_MESSAGE, INVALID_PROC_NUMBER);
```

**工作流程**：

```
Promotion 触发
    │
    ▼
ShutDownSlotSync() 被调用
    │
    ├─► 设置 SlotSyncCtx->stopSignaled = true
    │
    └─► 发送 PROCSIG_SLOTSYNC_MESSAGE 给 sync_process_pid
              │
              ▼
procsignal_sigusr1_handler() 收到信号
              │
              ▼
HandleSlotSyncMessageInterrupt() 执行
    ├─► InterruptPending = true
    └─► SlotSyncShutdownPending = true
              │
              ▼
下次 CHECK_FOR_INTERRUPTS() 执行
              │
              ▼
ProcessInterrupts() 检测到 SlotSyncShutdownPending
              │
              ▼
ProcessSlotSyncMessage() 执行
    │
    ├─► slotsync worker → proc_exit(0) 正常退出
    └─► pg_sync_replication_slots() → ereport(ERROR) 报错退出
```

### 4. 关键代码逻辑

**HandleSlotSyncMessageInterrupt()**（slotsync.c）：
```c
void HandleSlotSyncMessageInterrupt(void)
{
    InterruptPending = true;
    SlotSyncShutdownPending = true;
    /* latch will be set by procsignal_sigusr1_handler */
}
```

此函数仅设置标志位，实际处理在 `ProcessInterrupts()` 中进行，这确保了信号处理的安全上下文。

**ProcessSlotSyncMessage()**（slotsync.c）：
```c
void ProcessSlotSyncMessage(void)
{
    SlotSyncShutdownPending = false;

    if (AmLogicalSlotSyncWorkerProcess())
    {
        ereport(LOG, ...);
        proc_exit(0);  // Worker 正常退出
    }
    else
    {
        if (!IsSyncingReplicationSlots())
            return;  // 同步已完成，无需中断
        ereport(ERROR, ...);  // SQL 函数报错退出
    }
}
```

## 应用场景及最佳实践

### 场景一：网络故障期间的 Standby 晋升

**背景**：主库故障，需要将 Standby 晋升为新主库，但此时 Standby 与原主库之间存在网络问题。

**修复前行为**：

```
时间轴：
T0: 故障检测，触发 promotion
T1: startup 发送 SIGUSR1 给 slotsync worker
T2: slotsync worker 正等待主库响应（网络阻塞）
T3: SIGUSR1 未能中断等待，startup 持续等待...
T10: 网络超时（假设30秒），等待结束
T11: promotion 完成
总耗时：远超正常情况，可能长达数分钟
```

**修复后行为**：

```
时间轴：
T0: 故障检测，触发 promotion
T1: startup 发送 PROCSIG_SLOTSYNC_MESSAGE
T2: procsignal handler 设置中断标志
T3: 下次 CHECK_FOR_INTERRUPTS() 立即处理
T4: slotsync worker 立即退出
T5: promotion 完成
总耗时：几乎与正常情况相同
```

**关键改进**：即使在网络故障场景下，晋升也能快速完成，不再被 slotsync worker 阻塞。

### 场景二：slot 同步正在进行时的晋升

**背景**：备库正常运行中，slotsync worker 正在同步复制槽，此时管理员手动触发晋升。

**修复前后的对比**：

| 阶段 | 修复前 | 修复后 |
|------|--------|--------|
| 信号发送 | SIGUSR1 | PROCSIG_SLOTSYNC_MESSAGE |
| worker 响应 | 依赖下次检查 stopSignaled | 通过 CHECK_FOR_INTERRUPTS 立即响应 |
| 退出方式 | 等待到下一次检查点 | 立即 proc_exit(0) |
| pg_sync_replication_slots() | 可能长时间阻塞 | 立即 ereport(ERROR) |
| promotion 延迟 | 存在不确定性 | 可预测的短延迟 |

### 最佳实践建议

1. **监控 slotsync worker 状态**
   ```sql
   SELECT * FROM pg_stat_get_activity(pid) 
   WHERE application_name = 'logical slot sync worker';
   ```

2. **晋升前检查同步状态**
   ```sql
   -- 确认没有进行中的同步
   SELECT * FROM pg_stat_replication 
   WHERE sync_state IN ('sync', 'potential');
   ```

3. **设置合理的网络超时**
   - `wal_retrieve_retry_interval`：控制 standby 等待 WAL 的时间
   `hot_standby_feedback`：影响反馈间隔

4. **晋升超时监控**
   ```sql
   -- 监控 promotion 是否超时
   SELECT NOW() - pg_postmaster_start_time() AS uptime,
          pg_is_in_recovery() AS is_standby;
   ```

## 小结和思考

### 核心要点

1. **问题本质**：slotsync worker 在网络异常时的等待不可被 SIGUSR1 中断，导致 promotion 流程被阻塞

2. **解决方案**：引入专用进程信号 `PROCSIG_SLOTSYNC_MESSAGE`，配合 `CHECK_FOR_INTERRUPTS()` 机制实现可靠中断

3. **设计亮点**：
   - 区分了 `HandleSlotSyncMessageInterrupt()`（信号处理上下文）和 `ProcessSlotSyncMessage()`（安全的中断处理上下文）
   - 对 worker 和 SQL 函数采用不同的退出策略（exit vs error）
   - 保持了 `stopSignaled` 标志用于防止 promotion 后 worker 重启

### 对 DBA 的实际价值

- **可靠性提升**：主备切换时间变得可预测，不再受网络状况影响
- **运维简化**：晋升操作不会再因为 slotsync worker 而超时失败
- **故障恢复加快**：关键业务能在更短时间内恢复

### 延伸思考

1. **信号机制设计**：PostgreSQL 巧妙地通过 `CHECK_FOR_INTERRUPTS()` 统一处理各种中断，兼顾了性能和安全性

2. **进程间通信**：除 SIGUSR1 外，PostgreSQL 还支持 `SendProcSignal` 等多种进程通信方式，适用于不同场景

3. **优雅退出**：数据库进程应支持多种退出路径，确保资源正确释放的同时避免阻塞
