## 背景 

PostgreSQL 的 autovacuum 是保证数据库健康运行的核心机制，但长期以来 autovacuum 在处理大表时存在严重的性能瓶颈——它只能使用单线程处理索引清理工作。对于拥有数十个索引的巨型表，一次 autovacuum 可能需要数小时才能完成，这不仅导致表膨胀、查询性能下降，还可能引发表膨胀导致的存储危机。

本篇文章深入解读 PostgreSQL 提交 `1ff3180`（Allow autovacuum to use parallel vacuum workers），该提交首次实现了 autovacuum 的并行 vacuum 能力，允许 autovacuum worker 像手动 VACUUM 一样启动多个并行 worker 来并行处理索引的 vacuum 和 cleanup 工作。

  

**背景痛点**: 在此之前，autovacuum 始终以单线程模式运行，无论表上有多少索引或配置如何。对于大型表（如几十GB的日志表、历史分区表），索引清理成为瓶颈，导致 vacuum 时间过长、存储膨胀、查询性能劣化。

**核心价值**: 本次提交让 autovacuum 获得了与手动 VACUUM 相同的并行 vacuum 能力，可显著缩短大表的维护时间，改善数据库整体健康度。

## 一、原理介绍

### 1.1 原有机制的问题

在原有实现中，`autovacuum` 被设计为始终禁用并行 vacuum。即使用户手动执行 `VACUUM (PARALLEL 4) my_table` 可以并行处理索引，`autovacuum` 触发时仍然只能用单线程处理。这对于大表来说是严重的性能瓶颈。

**注意**: 在 `vacuumparallel.c` 的 `parallel_vacuum_main` 函数中，至今仍存在一个未更新的注释：
```c
/*
 * A parallel vacuum worker must have only PROC_IN_VACUUM flag since we
 * don't support parallel vacuum for autovacuum as of now.
 */
```
这个注释已经过时（本次提交已实现 autovacuum 并行 vacuum），但未被更新。这是注释陈旧，不影响实际行为——实际行为已由 `shared->is_autovacuum` 路径和 `autovacuum_max_parallel_workers` 生效逻辑决定（见 `vacuumparallel.c` 第 457 行和第 742 行）。

### 1.2 并行 vacuum 的基础设施

PostgreSQL 早在之前就为手动 VACUUM 实现了并行 vacuum 功能，代码位于 `vacuumparallel.c`。其核心架构如下：

```
┌─────────────────────────────────────────────────────────┐
│                    Leader Worker                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │              PVShared (DSM Segment)             │   │
│  │  ┌──────────────────────────────────────────┐  │   │
│  │  │ relid, elevel, queryid                   │  │   │
│  │  │ maintenance_work_mem_worker, ring_nbuffers│  │   │
│  │  │ cost_balance (pg_atomic), idx (pg_atomic) │  │   │
│  │  │ is_autovacuum (bool)                      │  │   │
│  │  ├──────────────────────────────────────────┤  │   │
│  │  │ PVSharedCostParams cost_params;          │  │   │
│  │  │   - generation (pg_atomic)               │  │   │
│  │  │   - cost_delay, cost_limit, ...          │  │   │
│  │  └──────────────────────────────────────────┘  │   │
│  └─────────────────────────────────────────────────┘   │
│         ▲                              ▲               │
│         │         DSM                  │               │
│  ┌──────┴──────┐              ┌───────┴───────┐        │
│  │ Worker 1    │              │ Worker N      │        │
│  │ Index 1     │    ...       │ Index M       │        │
│  └─────────────┘              └───────────────┘        │
└─────────────────────────────────────────────────────────┘
```

**关键组件说明**:

1. **PVShared**: 存储在 DSM 中的共享信息结构体，包含：
   - 目标表 relid、日志级别、queryid
   - maintenance_work_mem_worker、ring_nbuffers
   - pg_atomic 的 cost_balance 和 idx 计数器
   - **is_autovacuum**: 标志是否为 autovacuum worker
   - **cost_params**: autovacuum 专用的 cost-based delay 参数（手动 VACUUM PARALLEL 不使用此字段）
2. **DSM (Dynamic Shared Memory) Segment**: 允许 leader 和 worker 进程共享 PVShared 数据
3. **PVSharedCostParams**: 存储 cost-based vacuum delay 参数，使用 generation counter 实现 worker 间同步

### 1.3 新增的 cost 参数传播机制

autovacuum 与手动 VACUUM 的一个重要区别在于：autovacuum 的 cost-based delay 参数可能在运行时通过 SIGHUP 信号重新加载（如执行 `ALTER SYSTEM SET autovacuum_vacuum_cost_delay = 5; SELECT pg_reload_conf();`）。

为解决 autovacuum leader 和其并行 worker 之间参数同步的问题，本提交引入了 **generation counter 机制**：

```c
typedef struct PVSharedCostParams
{
    pg_atomic_uint32 generation;  // 原子操作的代际计数器
    slock_t mutex;                 // 保护以下字段的锁

    /* 需要同步给 worker 的参数 */
    double   cost_delay;           // vacuum_cost_delay
    int      cost_limit;           // vacuum_cost_limit
    int      cost_page_dirty;      // VacuumCostPageDirty
    int      cost_page_hit;         // VacuumCostPageHit
    int      cost_page_miss;        // VacuumCostPageMiss
} PVSharedCostParams;
```

**同步流程**（仅适用于 autovacuum workers，手动 VACUUM PARALLEL 无需此机制）:

```
Leader (autovacuum worker) 进程:
1. vacuum_delay_point() 检测到 ConfigReloadPending
2. ProcessConfigFile(PGC_SIGHUP) 重新加载配置
3. VacuumUpdateCosts() 更新本地 cost 参数
4. parallel_vacuum_propagate_shared_delay_params() 写入 DSM
5. pg_atomic_fetch_add_u32(&pv_shared_cost_params->generation, 1) 递增代际

Worker (并行 vacuum worker) 进程:
1. 在 vacuum_delay_point() 中调用 parallel_vacuum_update_shared_delay_params()
2. 比较本地 generation (shared_params_generation_local) 与共享 generation
3. 如有变化，从 DSM 读取新参数（cost_delay, cost_limit 等）并调用 VacuumUpdateCosts()
```

### 1.4 新的配置选项

本提交引入两个新的配置维度：

#### 1.4.1 全局 GUC: `autovacuum_max_parallel_workers`

```sql
-- postgresql.conf
autovacuum_max_parallel_workers = 4  -- 默认值 0（禁用）
```

**作用**: 设置单个 autovacuum worker 可以启动的最大并行 worker 数量。

**约束**: 除参数上限外，还受系统当前可用 parallel worker 资源约束。

#### 1.4.2 表级存储参数: `autovacuum_parallel_workers`

```sql
-- 在 CREATE TABLE 或 ALTER TABLE 时设置
CREATE TABLE my_large_table (
    id BIGINT,
    data TEXT
) WITH (autovacuum_parallel_workers = 4);

-- 或
ALTER TABLE my_large_table SET (autovacuum_parallel_workers = 4);
```

**语义**:

| 值 | 含义 |
|----|------|
| `-1`（默认）| 自动决定并行度（按可并行索引数推导），结果再受 `autovacuum_max_parallel_workers` 等上限约束 |
| `0` | 禁用该表的并行 vacuum |
| `正整数` | 使用指定值（同时受 GUC 上限约束） |

### 1.5 worker 数量计算逻辑

`parallel_vacuum_compute_workers()` 函数决定了实际启动的 worker 数量：

```c
static int
parallel_vacuum_compute_workers(Relation *indrels, int nindexes, int nrequested,
                                 bool *will_parallel_vacuum)
{
    int max_workers;

    // 关键变更：根据是否 autovacuum 选择不同的上限
    max_workers = AmAutoVacuumWorkerProcess() ?
        autovacuum_max_parallel_workers :
        max_parallel_maintenance_workers;

    // ...计算可并行处理的索引数...

    // 最终取 Min(请求数, 可并行索引数, 上限)
    parallel_workers = Min(parallel_workers, max_workers);

    return parallel_workers;
}
```

**实际并行条件**:
1. 索引大小（ RelationGetNumberOfBlocks ）必须不小于 `min_parallel_index_scan_size`（即不能太小）
2. 表至少有 2 个满足并行条件的索引（因为 leader 本身会处理 1 个索引）
3. 索引的 `amparallelvacuumoptions` 不能等于 `VACUUM_OPTION_NO_PARALLEL`（这意味着索引完全不参与并行）；如果索引只支持 `VACUUM_OPTION_PARALLEL_CLEANUP` 而不支持 `VACUUM_OPTION_PARALLEL_BULKDEL`（如 BRIN 索引），则只能参与 cleanup 阶段的并行

## 二、应用场景及最佳实践

### 2.1 场景一：超大型分区表的定期维护

**背景**: 某电商公司的订单历史表 `orders_history` 每月一个分区，每个分区约 80GB，包含 12 个索引（主键、买家ID、卖家ID、订单时间、状态等）。由于订单数据量大，autovacuum 经常无法在合理时间内完成清理。

**优化前**:

```sql
-- 查看 vacuum 状态
SELECT schemaname, relname, n_dead_tup, n_live_tup,
       last_autovacuum, autovacuum_count
FROM pg_stat_user_tables
WHERE relname LIKE 'orders_history%';
```

**结果**: `last_autovacuum` 时间戳经常是数天前，`n_dead_tup` 持续保持在数百万级别。

**优化后配置**:

```sql
-- 为各子分区设置并行 worker 数（分区父表的 reloption 不会自动继承到现有子分区）
ALTER TABLE orders_history_2025_01 SET (
    autovacuum_parallel_workers = 4
);
ALTER TABLE orders_history_2025_02 SET (
    autovacuum_parallel_workers = 4
);
-- ...

-- postgresql.conf
autovacuum_max_parallel_workers = 4
max_parallel_workers = 8
min_parallel_index_scan_size = 0  -- 如需让更小索引参与并行，可设为 0
```

**效果预估**:
- 原来单线程清理 12 个索引需要约 45 分钟
- 启用 4 个并行 worker 后，索引清理阶段缩短至约 12 分钟
- 总体 vacuum 时间从 60+ 分钟缩短至 30 分钟左右

### 2.2 场景二：高并发 OLTP 系统的表膨胀控制

**背景**: 某金融系统的交易流水表 `transactions` 24小时不间断写入，每天产生约 500 万条记录，20 个索引。由于是高并发系统，需要严格控制 vacuum 对业务的影响（cost-based delay 设置较高）。

**痛点**: 原来 autovacuum 单线程处理，每次 vacuum 期间累积的 dead tuples 导致表持续膨胀到 2x 正常大小。

**优化前配置**:

```sql
-- postgresql.conf
autovacuum_vacuum_cost_delay = 20      -- 较高延迟避免影响业务
autovacuum_vacuum_cost_limit = 200
autovacuum_max_workers = 3
```

**优化后**:

```sql
-- 启用并行 vacuum
ALTER TABLE transactions SET (
    autovacuum_parallel_workers = 3   -- 允许最多 3 个并行 worker
);

-- 调整全局配置
-- postgresql.conf
autovacuum_max_parallel_workers = 3
```

**核心价值**: 并行化缩短了索引阶段的总时长，多个 worker 可以同时处理不同索引。需要注意的是，vacuum 的 cost balance 机制仍是共享的/受全局约束的，并非每个 worker 独立叠加。

**配置后效果**:
- 表膨胀从 2x 降低到 1.2x 以内
- dead tuple 清理延迟从 3-4 天降低到 1 天以内
- 业务高峰期的 vacuum 延迟影响更小（多 worker 分散负载）

### 2.3 场景三：抑制特定表的并行 vacuum

**场景**: 某表 `critical_config` 是极其重要的配置表，虽然有多个索引，但：
1. 表很小（只有几百行）
2. 对响应时间极其敏感

**处理方式**: 显式禁用该表的并行 vacuum

```sql
ALTER TABLE critical_config SET (
    autovacuum_parallel_workers = 0   -- 禁用并行
);
```

**原因**: 对于小表，并行 overhead（进程创建、DSM 分配）可能反而比收益更大。

### 2.4 对比总结

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 索引清理方式 | 单线程顺序处理 | 多 worker 并行处理 |
| 大表 vacuum 时间 | 数小时 | 显著缩短 |
| 表膨胀风险 | 高（清理不及时） | 低（清理速度更快） |
| cost 参数传播 | N/A | Leader→Worker 实时同步 |
| 配置灵活性 | 仅 GUC | GUC + per-table |
| 默认行为 | 并行被禁用 | 需显式启用（向后兼容） |

## 三、监控与故障排查

### 3.1 验证并行 vacuum 是否生效

```sql
-- 1. 检查 autovacuum 是否使用了并行 worker
-- 在 postgresql.conf 中启用详细日志
-- log_autovacuum_min_duration = 0

-- 日志中会显示类似:
-- "parallel workers: index vacuum: 4 planned, 4 launched in total"
-- "parallel workers: index cleanup: 4 planned, 4 launched"

-- 2. 通过 pg_stat_activity 观察
SELECT pid, state, query, query_start
FROM pg_stat_activity
WHERE backend_type IN ('autovacuum launcher', 'autovacuum worker')
  AND state = 'active';
```

### 3.2 参数传播验证

当 autovacuum 在运行期间 GUC 发生变更时，worker 应该能感知到：

```sql
-- 在 vacuum 运行过程中执行
ALTER SYSTEM SET autovacuum_vacuum_cost_limit = 500;
SELECT pg_reload_conf();

-- 如果启用 DEBUG2 日志级别，将看到:
-- "parallel autovacuum worker updated cost params: cost_limit=500, ..."
-- 注意：此消息为 DEBUG2 级别，默认不显示，需设置 log_min_messages = DEBUG2
```

### 3.3 常见问题

**Q1: 设置了 `autovacuum_max_parallel_workers = 4` 但没看到并行？**

可能原因：
1. 表的索引数不足 2 个
2. 索引太小（小于 `min_parallel_index_scan_size`）
3. `max_parallel_workers` 全局限制
4. 并行 worker 资源不足，例如 `max_parallel_workers`/`max_worker_processes` 或当前可用 parallel worker 不足

**Q2: 并行 vacuum 会消耗更多内存吗？**

并行 vacuum 会把可用的 `vac_work_mem`/`maintenance_work_mem` 分摊给 worker，而不是每个 worker 都独占一整份 `maintenance_work_mem`。应关注并发 autovacuum 总量与 `autovacuum_work_mem`/`maintenance_work_mem` 的整体预算，但不要按简单乘法估算单表并行 vacuum 内存。

**Q3: 如何在不影响业务的情况下启用并行 vacuum？**

调整 cost 参数以控制 vacuum 的 I/O 速率：
```sql
ALTER SYSTEM SET autovacuum_vacuum_cost_delay = 10;  -- 增大延迟，降低侵扰
ALTER SYSTEM SET autovacuum_vacuum_cost_limit = 1000; -- 适当提高 limit
```

## 四、代码层面的关键变更

### 4.1 vacuumparallel.c 的核心变更

**PVShared 结构新增字段**（本次提交新增，第 160-170 行）:

```c
typedef struct PVShared
{
    // ... 原有字段 ...
    
    /*
     * 如果为 true，表示运行的是并行 autovacuum。
     * 否则表示运行的是并行手动 VACUUM。
     */
    bool        is_autovacuum;

    /*
     * autovacuum leader 和其并行 workers 之间共享的
     * cost-based vacuum delay 参数。
     * 手动 VACUUM PARALLEL 不使用此字段（手动 vacuum
     * 的 cost 参数在执行期间不会改变）。
     */
    PVSharedCostParams cost_params;
} PVShared;
```

**PVSharedCostParams 结构**（新增定义，第 67-88 行）:

```c
typedef struct PVSharedCostParams
{
    pg_atomic_uint32 generation;  // 原子操作的代际计数器，初始值为 1
    slock_t mutex;               // 保护以下字段的锁

    /* 需要同步给 worker 的参数 */
    double   cost_delay;           // vacuum_cost_delay
    int      cost_limit;           // vacuum_cost_limit
    int      cost_page_dirty;      // VacuumCostPageDirty
    int      cost_page_hit;        // VacuumCostPageHit
    int      cost_page_miss;       // VacuumCostPageMiss
} PVSharedCostParams;
```

**并行 worker 的 cost 参数初始化**（第 1260 行前后）:

```c
static void
parallel_vacuum_main(dsm_segment *seg, shm_toc *toc)
{
    // ... 初始化代码 ...
    
    /* 设置 cost-based vacuum delay */
    if (shared->is_autovacuum)
    {
        /*
         * 从 leader 的共享状态初始化 cost 参数，
         * 而不是使用 GUC 默认值，因为 leader 可能
         * 已经应用了 per-table 或 autovacuum 特定的覆盖。
         */
        pv_shared_cost_params = &(shared->cost_params);
        parallel_vacuum_update_shared_delay_params();
    }
    else
        VacuumUpdateCosts();
}
```

### 4.2 autovacuum.c 的核心变更

**从硬编码禁用改为基于配置计算并行度**（第 2940 行前后）:

```c
// 优化前: autovacuum 始终禁用并行 vacuum（无论表有多少索引）

// 优化后: 根据 per-table reloption 和 GUC 计算 nworkers
tab->at_params.nworkers = 0;
if (avopts)
{
    if (avopts->autovacuum_parallel_workers == 0)
    {
        // reloption 显式设置为 0，禁用该表的并行 vacuum
        tab->at_params.nworkers = -1;
    }
    else if (avopts->autovacuum_parallel_workers > 0)
        tab->at_params.nworkers = avopts->autovacuum_parallel_workers;
    // -1 的情况保持 nworkers=0，后续由 parallel_vacuum_compute_workers()
    // 根据索引数量和 GUC (autovacuum_max_parallel_workers) 计算实际并行度
}
```

**语义澄清**: 当 `nworkers = 0` 时，表示"让系统根据索引数量自动决定"；`nworkers = -1` 表示"禁用并行"；正整数表示"使用指定值（但仍受 GUC 上限约束）"。

### 4.3 vacuum.c 的 cost 传播逻辑

此传播机制**仅适用于 autovacuum workers**（手动 VACUUM PARALLEL 不需要此机制，因为其参数在执行期间不会改变）。

```c
void
vacuum_delay_point(bool is_analyze)
{
    // ... 中断检查 ...

    if (IsParallelWorker())
    {
        /*
         * 更新 cost-based vacuum delay 参数（仅适用于并行 autovacuum worker）。
         * 需要在 VacuumCostActive 检查之前调用，因为它可能启用 cost-based delay。
         */
        parallel_vacuum_update_shared_delay_params();
    }

    if (!VacuumCostActive && !ConfigReloadPending)
        return;

    /*
     * 以下代码路径仅适用于 autovacuum worker（手动 VACUUM 不会有 ConfigReloadPending）。
     * 当 autovacuum 运行期间收到 SIGHUP 并重新加载配置时，
     * 新的 cost 参数需要传播给所有并行 worker。
     */
    if (ConfigReloadPending && AmAutoVacuumWorkerProcess())
    {
        ConfigReloadPending = false;
        ProcessConfigFile(PGC_SIGHUP);
        VacuumUpdateCosts();

        /*
         * 将更新后的 cost 参数传播到共享内存，
         * 以便并行 workers 能够感知变化。
         */
        parallel_vacuum_propagate_shared_delay_params();
    }
    // ... rest of function ...
}
```

## 五、小结和思考

### 5.1 核心改进总结

本次提交 `1ff3180` 为 PostgreSQL 的 autovacuum 系统带来了期待已久的并行 vacuum 能力，其核心价值体现在：

1. **性能提升**: 通过并行处理索引的 vacuum 和 cleanup，可将大表的维护时间缩短数倍
2. **配置灵活性**: 同时提供全局 GUC 和 per-table reloption 两个维度的配置
3. **向后兼容**: 默认值设为 0（禁用），现有系统不受影响
4. **运行时参数同步**: 引入 generation counter 机制确保 cost 参数在 leader 和 worker 间正确同步
5. **代码复用**: 复用了已有的 `vacuumparallel.c` 基础设施，减少了维护负担

### 5.2 最佳实践建议

| 场景 | 建议配置 |
|------|----------|
| 大型表（>10GB，多索引） | `autovacuum_max_parallel_workers = 4`，`autovacuum_parallel_workers = 4` |
| 高并发业务表 | 通常配合较高的 `autovacuum_vacuum_cost_delay` 或较保守的 `cost_limit` 使用并行 |
| 小表 (<100MB) | 保持默认值或显式禁用并行 |
| 内存受限环境 | 降低 `maintenance_work_mem` 或 `autovacuum_max_parallel_workers` |

### 5.3 未来展望

1. **自适应并行度**: 根据系统负载动态调整并行 worker 数量
2. **索引类型支持扩展**: 目前某些索引类型（如 BRIN）仅支持 `VACUUM_OPTION_PARALLEL_CLEANUP`，不支持 `VACUUM_OPTION_PARALLEL_BULKDEL`，未来可能完善支持
3. **并行 vacuum 与 eager freeze 集成**: 进一步优化大表的 freeze 策略

### 5.4 升级路径

对于正在使用 PostgreSQL 的 DBA：

```sql
-- 1. 升级后默认不启用，先观察
-- 2. 为需要优化的表单独启用
ALTER TABLE my_large_table SET (autovacuum_parallel_workers = 2);

-- 3. 观察效果后，再考虑全局启用
-- postgresql.conf
-- autovacuum_max_parallel_workers = 2
```

### 5.5 与手动 VACUUM 的对比

| 特性 | 手动 VACUUM PARALLEL | Autovacuum Parallel |
|------|---------------------|---------------------|
| 并行 worker 上限 | `max_parallel_maintenance_workers` | `autovacuum_max_parallel_workers` |
| Per-table 控制 | `PARALLEL N` 选项 | `autovacuum_parallel_workers` reloption |
| Cost 参数同步 | N/A（单次执行） | Leader→Worker 实时同步 |
| 触发方式 | 手动执行 | 自动触发 |
