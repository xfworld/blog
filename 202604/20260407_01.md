## 背景 

Autovacuum 是 PostgreSQL 保持数据库健康的核心后台进程，但长期以来 DBA 只能看到表级别的静态配置（如 `autovacuum_vacuum_threshold`），而无法直接观测到 autovacuum 内部使用的评分机制——这个评分决定了当多个表同时需要清理时，autovacuum 优先处理哪个表。

本篇文章深入解读 PostgreSQL 提交 `87f61f0`（Add pg_stat_autovacuum_scores system view），该提交新增了 `pg_stat_autovacuum_scores` 系统视图，让 DBA 可以直接查询每个表的 autovacuum 评分、各组件分数以及是否即将触发 vacuum 或 analyze 操作。

  

**背景痛点**: 在此之前，DBA 只能通过 `pg_stat_user_tables` 查看死元组数量和最后清理时间，无法直接了解 autovacuum 的内部评分机制。当多个表同时需要清理时，哪些表会被优先处理？表的 freeze 风险有多高？这些问题只能靠经验推断，缺乏透明性。

**核心价值**: `pg_stat_autovacuum_scores` 将 autovacuum 的评分机制透明化，让 DBA 能够：
- 实时监控每个表的 autovacuum 评分
- 提前识别需要优先清理的表
- 理解各组件分数（XID、MXID、vacuum、insert、analyze）对总分的贡献
- 更好地调优 autovacuum 参数

## 一、原理介绍

### 1.1 Autovacuum 的评分机制

当 autovacuum worker 从 pg_class 中获取需要处理的表列表时，它会为每个表计算一个"分数"（score）。这个分数的本质是**各维度需求程度的比率**：实际值与阈值的比值。比值越大，说明表越"迫切"需要处理。

autovacuum 使用**最大组件分数**作为表的最终分数：

```c
scores->max = Max(scores->xid, scores->mxid);
scores->max = Max(scores->max, scores->vac);
scores->max = Max(scores->max, scores->vac_ins);
scores->max = Max(scores->max, scores->anl);
```

Autovacuum 按 `scores->max` 从高到低排序，优先处理分数最高的表。

需要注意的是，分数影响同库候选表之间的相对优先级，但实际执行还受 launcher 调度周期、worker 可用槽位、锁冲突、autovacuum_enabled 参数以及预防性 vacuum（wraparound 强制）等因素的综合影响，不应将其理解为全局唯一的决定因素。

### 1.2 五大评分组件详解

#### 1.2.1 XID Score（事务ID年龄分数）

```c
xid_age = recentXid - relfrozenxid;  // 当前XID - 表的最后冻结XID
scores->xid = (double) xid_age / freeze_max_age;
scores->xid *= autovacuum_freeze_score_weight;
```

**含义**: 表的 XID 年龄相对于有效的 `freeze_max_age`（表级 reloption 或经全局 GUC 裁剪后的值）的比例。

**阈值判定**: 当 `xid_age >= effective_xid_failsafe_age` 时，分数会被幂次放大（`scores->xid = pow(scores->xid, Max(1.0, xid_age / 100000000))`），确保持久化风险表排在最前。注意：如果 `relfrozenxid` 不是正常事务 ID（即 `TransactionIdIsNormal(relfrozenxid)` 返回 false，例如 `relfrozenxid` 为 `InvalidTransactionId`=0），则 `xid_age` 为 0；代码上按 0 处理，不应直接推断为已发生环绕。

#### 1.2.2 MXID Score（多事务ID年龄分数）

```c
mxid_age = recentMulti - relminmxid;  // 当前MXID - 表的最后冻结MXID
scores->mxid = (double) mxid_age / multixact_freeze_max_age;
scores->mxid *= autovacuum_multixact_freeze_score_weight;
```

**含义**: 类似于 XID Score，但针对多事务ID，用于防止多事务ID环绕（wraparound）。注意：如果 `relminmxid` 不是有效的多事务 ID（即 `MultiXactIdIsValid(relminmxid)` 返回 false，例如 `relminmxid` 为 `InvalidMultiXactId`=0），则 `mxid_age` 为 0；代码上按 0 处理，不应直接推断为已发生异常或环绕。

#### 1.2.3 Vacuum Score（清理分数）

```c
vacthresh = vac_base_thresh + vac_scale_factor * reltuples;
scores->vac = (double) vactuples / Max(vacthresh, 1);
scores->vac *= autovacuum_vacuum_score_weight;
```

**含义**: 实际死元组数量与触发 vacuum 的阈值的比值。

**阈值计算公式**:
- `vac_base_thresh`: 基础阈值（默认 50）
- `vac_scale_factor`: 比例因子（默认 0.2）
- `reltuples`: 表的行数估计值
- 最终阈值 = `vac_base_thresh + vac_scale_factor * reltuples`，如果计算结果超过 `vac_max_thresh` 且 `vac_max_thresh >= 0`，则使用 `vac_max_thresh` 作为上限

#### 1.2.4 Vacuum Insert Score（插入清理分数）

```c
vacinsthresh = vac_ins_base_thresh + vac_ins_scale_factor * reltuples * pcnt_unfrozen;
scores->vac_ins = (double) instuples / Max(vacinsthresh, 1);
scores->vac_ins *= autovacuum_vacuum_insert_score_weight;
```

**含义**: 自上次 vacuum 以来的插入元组数量（`ins_since_vacuum` 字段）与阈值的比值。`pcnt_unfrozen` 是表"活跃部分"（非冻结页）的比例。

#### 1.2.5 Analyze Score（分析分数）

```c
anlthresh = anl_base_thresh + anl_scale_factor * reltuples;
scores->anl = (double) anltuples / Max(anlthresh, 1);
scores->anl *= autovacuum_analyze_score_weight;
```

**含义**: 自上次 analyze 以来修改的元组数量与阈值的比值。

### 1.3 分数权重参数

各组件分数还可以通过权重参数调节影响力度：

| 参数 | 默认值 | 范围 | 作用 |
|------|--------|------|------|
| `autovacuum_freeze_score_weight` | 1.0 | 0.0-10.0 | XID 分数权重 |
| `autovacuum_multixact_freeze_score_weight` | 1.0 | 0.0-10.0 | MXID 分数权重 |
| `autovacuum_vacuum_score_weight` | 1.0 | 0.0-10.0 | Vacuum 分数权重 |
| `autovacuum_vacuum_insert_score_weight` | 1.0 | 0.0-10.0 | Insert vacuum 分数权重 |
| `autovacuum_analyze_score_weight` | 1.0 | 0.0-10.0 | Analyze 分数权重 |

**实战技巧**: 如果你希望优先处理表膨胀（dead tuples），可以提高 `autovacuum_vacuum_score_weight`；如果更关注统计信息新鲜度（analyze），可以提高 `autovacuum_analyze_score_weight`。

### 1.4 代码实现：pg_stat_get_autovacuum_scores 函数

```c
Datum
pg_stat_get_autovacuum_scores(PG_FUNCTION_ARGS)
{
    // 初始化
    effective_multixact_freeze_max_age = MultiXactMemberFreezeThreshold();
    recentXid = ReadNextTransactionId();
    recentMulti = ReadNextMultiXactId();

    // 扫描 pg_class
    rel = table_open(RelationRelationId, AccessShareLock);
    scan = table_beginscan_catalog(rel, 0, NULL);

    while ((tup = heap_getnext(scan, ForwardScanDirection)) != NULL)
    {
        Form_pg_class form = (Form_pg_class) GETSTRUCT(tup);

        // 跳过非表类型（包括索引等），但包含 TOAST 表
        if (form->relkind != RELKIND_RELATION &&
            form->relkind != RELKIND_MATVIEW &&
            form->relkind != RELKIND_TOASTVALUE)
            continue;
        if (form->relpersistence == RELPERSISTENCE_TEMP)
            continue;

        // 提取 reloptions 并计算分数
        avopts = extract_autovac_opts(tup, RelationGetDescr(rel));
        relation_needs_vacanalyze(form->oid, avopts, form, ...);

        // 将结果写入 tuplestore
        vals[0] = ObjectIdGetDatum(form->oid);
        vals[1] = Float8GetDatum(scores.max);
        // ... 其他字段 ...
        tuplestore_putvalues(...);
    }
}
```

**视图定义**（system_views.sql）：

```sql
CREATE VIEW pg_stat_autovacuum_scores AS
    SELECT
        s.oid AS relid,
        n.nspname AS schemaname,
        c.relname AS relname,
        s.score,           -- 最大分数
        s.xid_score,       -- XID 组件分数
        s.mxid_score,      -- MXID 组件分数
        s.vacuum_score,    -- Vacuum 组件分数
        s.vacuum_insert_score,  -- Insert vacuum 组件分数
        s.analyze_score,    -- Analyze 组件分数
        s.do_vacuum,       -- 是否需要 vacuum
        s.do_analyze,      -- 是否需要 analyze
        s.for_wraparound   -- 是否为 wraparound 预防
    FROM pg_stat_get_autovacuum_scores() s
    JOIN pg_class c on c.oid = s.oid
    LEFT JOIN pg_namespace n ON n.oid = c.relnamespace;
```

## 二、应用场景及最佳实践

### 2.1 场景一：识别即将触发 Wraparound 的表

**背景**: XID wraparound 是 PostgreSQL 最严重的安全问题之一。一旦 XID 耗尽，数据库将无法写入新数据。Autovacuum 通过定期冻结旧 XID 来防止这个问题。

**使用 pg_stat_autovacuum_scores**：

```sql
-- 查找 XID 年龄最高的表（可能有 wraparound 风险）
SELECT
    relid,
    schemaname,
    relname,
    xid_score,
    mxid_score,
    score,
    for_wraparound
FROM pg_stat_autovacuum_scores
ORDER BY xid_score DESC
LIMIT 10;

-- 结果示例：
--  relid  | schemaname |      relname      | xid_score | mxid_score | score  | for_wraparound
-- --------+------------+-------------------+-----------+------------+--------+----------------
--  184567 | public     | big_transaction_log |  0.85    |    0.12    |  0.85  | f
--  184568 | public     | order_history_2024  |  0.82    |    0.15    |  0.82  | f
```

**优化前**（无此视图时）：
- 只能通过 `SELECT datname, age(datfrozenxid) FROM pg_database` 查看全局状态
- 无法精确定位哪个表最需要优先处理

**优化后**：
- 可精确定位高风险表
- 可设置告警（如 `xid_score >= autovacuum_freeze_score_weight` 时告警，默认为 1.0；或关注 `for_wraparound = true` 的表；0.7 仅作为自定义预警阈值示例）

### 2.2 场景二：理解多表竞争时的处理顺序

**背景**: 当 autovacuum worker 可用 slots 有限，但多个表都需要清理时，autovacuum 会按分数排序处理。理解这个机制有助于调优。

**示例**：

```sql
-- 查看所有即将被 vacuum 的表及其分数分解
SELECT
    relname,
    score,
    ROUND(xid_score::numeric, 3) AS xid,
    ROUND(mxid_score::numeric, 3) AS mxid,
    ROUND(vacuum_score::numeric, 3) AS vac,
    ROUND(vacuum_insert_score::numeric, 3) AS vac_ins,
    ROUND(analyze_score::numeric, 3) AS anl,
    do_vacuum,
    do_analyze
FROM pg_stat_autovacuum_scores
WHERE do_vacuum = true OR do_analyze = true
ORDER BY score DESC;
```

**结果示例**：

```
   relname    | score |  xid  | mxid |  vac  | vac_ins |  anl  | do_vacuum | do_analyze
--------------+-------+-------+------+--------+----------+-------+------------+------------
 orders       |  1.85 | 0.00  | 0.00 |  1.85 |    0.00  |  0.00 | t         | f
 users        |  0.95 | 0.00  | 0.00 |  0.95 |    0.00  |  0.00 | t         | f
 user_sessions|  0.85 | 0.85  | 0.00 |  0.00 |    0.00  |  0.00 | t         | f
```

**分析**：
- `orders` 表的 vacuum_score 最高（1.85），所以会最先被处理
- `user_sessions` 的 xid_score 为 0.85，总分也是 0.85，会排后

### 2.3 场景三：调优 autovacuum 参数

**背景**: DBA 希望调整权重参数，让 autovacuum 优先处理表膨胀而非 XID 年龄。

**调优前问题**：
```sql
-- 假设某表持续有大量死元组但 XID 年龄较低
-- 默认权重下，XID 和 vacuum 权重相同
```

**调优思路**：

```sql
-- 提高 vacuum 权重，降低 freeze 权重
ALTER SYSTEM SET autovacuum_vacuum_score_weight = 2.0;
ALTER SYSTEM SET autovacuum_freeze_score_weight = 0.5;
SELECT pg_reload_conf();
```

**验证效果**：

```sql
-- 调优前后对比
SELECT relname, score, vacuum_score, xid_score
FROM pg_stat_autovacuum_scores
WHERE relname = 'orders';

-- 调优后，vacuum_score 的实际值会因权重变化而改变，分数计算会重新调整
```

### 2.4 场景四：监控特定表的维护需求

**场景**: 某关键业务表 `critical_orders` 需要确保及时 vacuum。

**监控查询**：

```sql
-- 创建监控视图（包含关键表）
CREATE VIEW critical_tables_autovacuum_status AS
SELECT
    schemaname,
    relname,
    score,
    vacuum_score,
    vacuum_insert_score,
    do_vacuum,
    do_analyze,
    CASE
        WHEN score > 1.0 THEN '🔥 紧急'
        WHEN score > 0.5 THEN '⚠️ 关注'
        ELSE '✅ 正常'
    END AS status
FROM pg_stat_autovacuum_scores
WHERE relname IN ('critical_orders', 'orders', 'customers');

-- 查询
SELECT * FROM critical_tables_autovacuum_status;
```

### 2.5 场景五：识别被忽略的表

**背景**: 某些表虽然有大量死元组，但由于 `autovacuum_enabled = false` 或其他原因，autovacuum 不会处理。

**识别问题**：

```sql
-- 查看 do_vacuum = false 但 vacuum_score 仍然较高的表
SELECT
    relname,
    vacuum_score,
    do_vacuum,
    do_analyze
FROM pg_stat_autovacuum_scores
WHERE vacuum_score > 0.5
  AND do_vacuum = false;

-- 可能原因：
-- 1. autovacuum_enabled = false（该表被禁用 autovacuum），即使分数高也不会执行常规 vacuum
--    （但如果 XID/MXID 超阈值，do_vacuum 仍可能为 true，因为预防性 vacuum 不受 autovacuum_enabled 控制）
-- 2. 该表没有统计信息（tabentry 为 NULL），视图仍会显示该表，但 vacuum/analyze 相关分数可能为 0
-- 3. 表是 TOAST 表（relkind = RELKIND_TOASTVALUE），不会执行 analyze，但会执行 vacuum
```

## 三、视图字段详解

### 3.1 字段完整列表

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `relid` | oid | 表的 OID |
| `schemaname` | name | 模式名 |
| `relname` | name | 表名 |
| `score` | double precision | 最大组件分数（排序依据） |
| `xid_score` | double precision | XID 年龄组件分数 |
| `mxid_score` | double precision | MXID 年龄组件分数 |
| `vacuum_score` | double precision | 死元组清理组件分数 |
| `vacuum_insert_score` | double precision | 插入清理组件分数 |
| `analyze_score` | double precision | 统计信息更新组件分数 |
| `do_vacuum` | bool | 是否会执行 vacuum |
| `do_analyze` | bool | 是否会执行 analyze |
| `for_wraparound` | bool | 是否为 wraparound 预防 vacuum |

### 3.2 分数阈值判定规则

| 条件 | `do_vacuum` | `do_analyze` | `for_wraparound` |
|------|-------------|--------------|------------------|
| 死元组数 > vacuum 阈值 | true | - | - |
| 插入元组数 > insert vacuum 阈值 | true | - | - |
| XID 年龄 > freeze_max_age | true | - | - |
| MXID 年龄 > multixact_freeze_max_age | true | - | - |
| 修改元组数 > analyze 阈值 | - | true | - |
| 达到 wraparound 预防阈值 | true | - | true |

### 3.3 重要提示

> **注意**: 此视图计算分数时使用的是当前的源信息，而 autovacuum worker 实际处理时可能会有所不同。因此，此视图不能完全准确地预测 autovacuum 会处理哪些表以及处理顺序。

这意味着：
- 视图显示的数据是"实时快照"
- Autovacuum worker 获取表列表和实际处理之间可能有时间差
- 分数可能在实际处理前发生变化

> **关于 `autovacuum_enabled = false` 的说明**：当表的 `autovacuum_enabled` reloption 设置为 false 时，autovacuum **不会**对该表执行常规的 vacuum 和 analyze 操作（因为 `av_enabled` 为 false）。但是，如果该表的 XID 或 MXID 年龄超过了 `freeze_max_age` 或 `multixact_freeze_max_age` 阈值（导致 `force_vacuum = true`），autovacuum **仍会**执行预防性 vacuum 以防止 wraparound。此时 `do_vacuum` 为 true；`autovacuum_enabled=false` 会禁用常规 vacuum/analyze，但不会阻止防 wraparound vacuum。

## 四、实战查询模板

### 4.1 日常巡检

```sql
-- 1. 查看需要处理的表（按紧急程度排序）
SELECT
    schemaname,
    relname,
    score,
    do_vacuum,
    do_analyze,
    for_wraparound
FROM pg_stat_autovacuum_scores
WHERE do_vacuum OR do_analyze OR for_wraparound
ORDER BY score DESC;

-- 2. 查看 XID 风险最高的表
SELECT * FROM pg_stat_autovacuum_scores
ORDER BY xid_score DESC
LIMIT 5;

-- 3. 查看表膨胀最严重的表
SELECT * FROM pg_stat_autovacuum_scores
ORDER BY vacuum_score DESC
LIMIT 5;
```

### 4.2 性能调优

```sql
-- 分析各组件分数的分布，理解当前负载特征
SELECT
    COUNT(*) AS total_tables,
    SUM(CASE WHEN vacuum_score > 1.0 THEN 1 ELSE 0 END) AS high_vac_tables,
    SUM(CASE WHEN xid_score > 0.5 THEN 1 ELSE 0 END) AS high_xid_tables,
    SUM(CASE WHEN analyze_score > 1.0 THEN 1 ELSE 0 END) AS high_anl_tables
FROM pg_stat_autovacuum_scores;
```

### 4.3 告警配置参考

```sql
-- 高优先级告警：XID 分数过高
CREATE OR REPLACE FUNCTION check_autovacuum_xid_risk()
RETURNS void AS $$
DECLARE
    high_xid_count INTEGER;
BEGIN
    SELECT COUNT(*) INTO high_xid_count
    FROM pg_stat_autovacuum_scores
    WHERE xid_score > 0.8 OR for_wraparound = true;

    IF high_xid_count > 0 THEN
        RAISE WARNING 'Autovacuum XID risk: % tables need immediate attention', high_xid_count;
    END IF;
END;
$$ LANGUAGE plpgsql;
```

## 五、小结和思考

### 5.1 核心价值总结

`pg_stat_autovacuum_scores` 的推出让 autovacuum 的决策过程第一次对 DBA 透明可见：

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 优先级排序依据 | 黑箱 | 完全透明 |
| XID 风险识别 | 只能看全局 `pg_database.age()` | 可定位到具体表 |
| 参数调优 | 靠经验猜测 | 可量化分析各组件影响 |
| 问题排查 | 事后补救 | 可提前预警 |

### 5.2 最佳实践建议

1. **纳入日常巡检**: 将高分数表的查询纳入数据库巡检流程
2. **配置合理告警**: 通用告警更适合关注 `for_wraparound = true` 的表，或使用 `xid_score >= autovacuum_freeze_score_weight`（默认 1.0）作为阈值；0.7 仅作为自定义预警阈值示例，不作为通用建议
3. **理解权重参数**: 根据业务特征调整各组件的权重
4. **结合其他视图**: 配合 `pg_stat_user_tables` 和 `pg_stat_bgwriter` 全面了解数据库状态

### 5.3 与其他视图的对比

| 视图 | 用途 | 特点 |
|------|------|------|
| `pg_stat_user_tables` | 表级统计信息 | 显示历史累计数据（最后一次 vacuum/analyze 时间、死元组数等） |
| `pg_stat_bgwriter` | BGWriter 统计 | 显示后台写入器活动 |
| `pg_stat_activity` | 活动会话 | 显示当前运行的查询，包括 autovacuum |
| `pg_stat_autovacuum_scores` | Autovacuum 评分 | 显示即时的、预测性的清理需求评分 |

### 5.4 未来展望

1. **历史趋势追踪**: 未来可能新增系统表来存储分数历史，支持趋势分析
2. **自动化参数调优**: 基于分数分布自动推荐权重参数
3. **增强的可视化**: 在 pgAdmin 等工具中集成分数热力图
