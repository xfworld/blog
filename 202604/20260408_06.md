## 背景 

`pg_stash_advice` 是 PostgreSQL 的一个扩展模块，用于存储和自动应用执行计划建议。该扩展与 `pg_plan_advice` 配合工作，当查询规划器规划某个查询时，如果该查询的 ID 在建议存储（stash）中存在对应的建议字符串，规划器将自动应用该建议来指导生成更优的执行计划。

**背景痛点**：在引入持久化机制之前，`pg_stash_advice` 的所有数据存储在动态共享内存（DSA）中，这意味着数据库重启后，所有已存储的建议会全部丢失。DBA 需要在每次数据库启动后重新配置建议，这大大限制了该扩展在生产环境中的实用性。
  

本 commit（c10edb102ada607eb054bc9e23690109d86849ef）为 `pg_stash_advice` 引入了磁盘持久化能力，新增了 `pg_stash_advice.persist` 和 `pg_stash_advice.persist_interval` 两个 GUC 参数，并通过后台工作进程（Background Worker）将数据定期写入数据目录下的 `pg_stash_advice.tsv` 文件。数据库重启后，后台工作进程会自动从该文件加载之前存储的建议，实现了"一次配置，永久生效"的能力。

## 1. 原理介绍

### 1.1 整体架构

`pg_stash_advice` 的核心是一个基于 queryId → advice_string 的键值存储映射。系统架构涉及以下几个关键组件：

```
┌─────────────────────────────────────────────────────────────────┐
│                      PostgreSQL Server                          │
│  ┌──────────────┐    ┌─────────────────┐    ┌──────────────┐ │
│  │ pg_plan_advice│    │ pg_stash_advice │    │   Planner    │ │
│  │   ( Advisor)  │◄───│   ( Storage )   │◄───│  ( Consumer) │ │
│  └──────────────┘    └─────────────────┘    └──────────────┘ │
│                              │                                  │
│                              ▼                                  │
│                    ┌─────────────────┐                          │
│                    │  Shared Memory  │                          │
│                    │  (DSA + Hash)   │                          │
│                    └─────────────────┘                          │
│                              │                                  │
│                              ▼                                  │
│                    ┌─────────────────┐                          │
│                    │Background Worker│                          │
│                    │  ( Persist )    │                          │
│                    └─────────────────┘                          │
│                              │                                  │
│                              ▼                                  │
│                    ┌─────────────────┐                          │
│                    │pg_stash_advice.tsv│                        │
│                    │   ( Disk File )  │                        │
│                    └─────────────────┘                          │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 核心数据结构

从 `pg_stash_advice.h` 中的结构体定义可以看出共享内存的组织方式：

```c
typedef struct pgsa_shared_state
{
    LWLock       lock;              // 保护整个结构的轻量锁
    int          dsa_tranche;        // DSA 内存池 tranche ID
    int          stash_tranche;      // stash 哈希表 tranche ID
    int          entry_tranche;      // entry 哈希表 tranche ID
    uint64       next_stash_id;      // 下一个可用的 stash ID
    dsa_handle   area;               // DSA 内存区域句柄
    dshash_table_handle stash_hash;  // stash 名称→ID 映射表句柄
    dshash_table_handle entry_hash;  // entry (stash_id + queryId)→advice 映射表句柄
    pid_t        bgworker_pid;      // 后台工作进程 PID
    pg_atomic_flag stashes_ready;    // 标志：stash 数据是否已从磁盘加载完成
    pg_atomic_uint64 change_count;   // 原子计数器：追踪数据变更次数
} pgsa_shared_state;
```

**关键字段说明**：

- `stashes_ready` 是一个原子标志，用于在数据库启动时实现锁住机制。当值为"未就绪"时，所有修改 stash 的操作都会被拒绝，直到后台工作进程完成从磁盘加载数据并将标志设置为"就绪"。
- `change_count` 是原子递增计数器，每当创建 stash、删除 stash、设置或清除 advice 字符串时都会递增。后台工作进程通过比较 `change_count` 与上次写入时的值来判断是否有新变化需要持久化到磁盘。

### 1.3 持久化工作流程

持久化机制通过后台工作进程实现，分为**启动加载**、**运行期监测**和**关闭保存**三个阶段：

#### 阶段一：启动加载

当 `pg_stash_advice.persist=true` 时，系统启动流程如下：

```
1. _PG_init() 被调用
   └─► pgsa_start_worker()  // 注册并启动后台工作进程

2. 后台工作进程 pg_stash_advice_worker_main() 执行
   └─► pgsa_read_from_disk()  // 读取 pg_stash_advice.tsv
       └─► pgsa_reset_all_stashes()  // 先清空现有共享内存
       └─► 解析 TSV 文件中的 stash 和 entry
       └─► pgsa_restore_stashes()  // 恢复 stash 结构
       └─► pgsa_restore_entries()  // 恢复 entry 数据

3. 加载完成后
   └─► pg_atomic_test_set_flag(&pgsa_state->stashes_ready)  // 解除锁定
```

关键代码位于 `pgsa_init_shared_state()` 函数：

```c
/*
 * If this module was loaded via shared_preload_libraries, then
 * pg_stash_advice_persist is a GUC variable. If it's true, that means
 * that we should lock out manual stash modifications until the dump file
 * has been successfully loaded. If it's false, there's nothing to load,
 * so we set stashes_ready immediately.
 */
if (!pg_stash_advice_persist)
    pg_atomic_test_set_flag(&state->stashes_ready);
```

这意味着如果禁用了持久化（`persist=false`），系统会立即设置 `stashes_ready` 标志，允许所有操作。如果启用了持久化，则需要等待后台工作进程从磁盘加载完成后才能解除锁定。

#### 阶段二：运行期监测

后台工作进程的主循环通过 `change_count` 原子计数器检测变更：

```c
while (!ShutdownRequestPending)
{
    if (pg_stash_advice_persist_interval <= 0)
    {
        // persist_interval=0: 只在关闭时写入，等待 latch 信号
        (void) WaitLatch(MyLatch, WL_LATCH_SET | WL_EXIT_ON_PM_DEATH, -1L, ...);
    }
    else
    {
        // 计算距离下次写入的时间
        next_write_time = TimestampTzPlusMilliseconds(last_write_time,
                                    pg_stash_advice_persist_interval * 1000);
        delay_in_ms = TimestampDifferenceMilliseconds(GetCurrentTimestamp(),
                                                      next_write_time);

        if (delay_in_ms <= 0)  // 到达写入时间点
        {
            current_change_count = pg_atomic_read_u64(&pgsa_state->change_count);
            if (current_change_count != last_change_count)  // 有变更
            {
                pgsa_write_to_disk();  // 写入磁盘
                last_change_count = current_change_count;
            }
            last_write_time = GetCurrentTimestamp();
        }

        // 睡眠等待
        (void) WaitLatch(MyLatch, WL_LATCH_SET | WL_TIMEOUT | WL_EXIT_ON_PM_DEATH,
                         delay_in_ms, ...);
    }
    ResetLatch(MyLatch);
}
```

**工作原理**：后台工作进程每隔 `persist_interval` 秒（默认 30 秒）醒来一次，检查 `change_count` 是否发生变化。如果自上次写入以来有新的增删改操作，就将当前所有 stash 数据写入磁盘。这种设计避免了每次变更都立即写盘的开销，同时保证了最多 `persist_interval` 秒的数据丢失窗口。

#### 阶段三：关闭保存

```c
/* 收到关闭信号后，执行最后一次写入 */
pgsa_write_to_disk();
```

在数据库正常关闭时，后台工作进程会收到 `SIGTERM` 信号，在退出前执行最后一次磁盘写入，确保数据不丢失。

### 1.4 文件格式

持久化文件采用 TSV（制表符分隔）格式，保存在数据目录下的 `pg_stash_advice.tsv`：

```
stash	production_advice
stash	testing_advice
entry	production_advice	1234567890	SeqScan(t)
entry	production_advice	9876543210	IndexScan(ti)
entry	testing_advice		5555555555	HashJoin(a,b)
```

**格式说明**：
- 以 `stash\t` 开头的行表示一个 stash 的创建
- 以 `entry\t` 开头的行表示一个 advice 条目
- advice 字符串中特殊字符（`\t`, `\n`, `\r`, `\`）使用反斜杠转义

这种纯文本格式的好处是可以人工检查和编辑，便于排查问题。

### 1.5 变更追踪机制

所有的写操作（创建/删除 stash、设置/清除 advice）都会触发 `change_count` 的原子递增：

```c
void pgsa_create_stash(char *stash_name)
{
    // ... 创建 stash 逻辑 ...
    /* Bump change count. */
    pg_atomic_add_fetch_u64(&pgsa_state->change_count, 1);
}

void pgsa_drop_stash(char *stash_name)
{
    // ... 删除 stash 逻辑 ...
    /* Bump change count. */
    pg_atomic_add_fetch_u64(&pgsa_state->change_count, 1);
}

void pgsa_set_advice_string(char *stash_name, int64 queryId, char *advice_string)
{
    // ... 设置 advice 逻辑 ...
    /* Bump change count. */
    pg_atomic_add_fetch_u64(&pgsa_state->change_count, 1);
}
```

这种设计将变更检测的时间复杂度降为 O(1)，后台工作进程只需简单比较两个整数即可判断是否有变更发生。

## 2. 应用场景及最佳实践

### 2.1 典型应用场景

**场景描述**：在某电商平台的数据库中，一条复杂的统计查询 `SELECT count(*), date_trunc('day', created_at) FROM orders GROUP BY 1 ORDER BY 2` 由于统计信息不准确，查询规划器错误地选择了嵌套循环连接（Nested Loop）而非哈希连接（Hash Join），导致执行时间从预期的 2 秒变成了 45 秒。DBA 通过分析后人工设置了 `HashJoin` 建议，查询恢复正常。

**痛点问题**：在没有持久化机制之前，每次数据库重启都需要重新设置这个建议。对于一个 7x24 运行的电商系统，这意味着每次计划外的数据库重启（如系统更新、故障恢复）后的 45 秒内，所有用户都会经历极慢的查询响应。

### 2.2 Commit 前后的对比

####  Commit 前（无持久化）

```
数据库重启前：
┌──────────────────────────────────────────┐
│ Shared Memory (DSA)                      │
│  ┌────────────────────────────────────┐  │
│  │ production_advice                   │  │
│  │   queryId=1234567890 → "SeqScan(t)" │  │
│  │   queryId=9876543210 → "HashJoin"   │  │
│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
        │  数据库重启  │
        ▼             ▼
数据库重启后：
┌──────────────────────────────────────────┐
│ Shared Memory (DSA) - EMPTY              │
│  ┌────────────────────────────────────┐  │
│  │ (no stashes)                       │  │
│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
        ❌ 建议全部丢失，需要手动恢复
```

```
数据库重启前：

│  │ production_advice                   │  │
│  │   queryId=1234567890 → "SeqScan(t)" │  │

│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
        │  persist_interval=30s  │
        ▼                        ▼
┌──────────────────────────────────────────┐
│ pg_stash_advice.tsv (disk)               │

┌──────────────────────────────────────────┐

│  ┌────────────────────────────────────┐  │
│  │ production_advice                   │  │

```

### 2.3 配置示例

#### postgresql.conf 配置示例

```conf
# 必须在 shared_preload_libraries 中加载
shared_preload_libraries = 'pg_plan_advice, pg_stash_advice'

# 启用持久化（默认开启）
pg_stash_advice.persist = true

# 持久化间隔（秒），0 表示只在重启时写入
pg_stash_advice.persist_interval = 30

# 当前使用的 stash 名称
pg_stash_advice.stash_name = 'production_advice'
```

#### 使用流程

```sql
-- 1. 创建 stash
SELECT pg_create_advice_stash('production_advice');

-- 2. 设置特定查询的建议
SELECT pg_set_stashed_advice('production_advice', 1234567890, 'SeqScan(t)');
SELECT pg_set_stashed_advice('production_advice', 9876543210, 'HashJoin(a,b)');

-- 3. 查看当前 stash 内容
SELECT * FROM pg_get_advice_stashes();
SELECT * FROM pg_get_advice_stash_contents('production_advice');

-- 4. 数据库重启后，数据自动恢复
```

### 2.4 最佳实践

**1. persist_interval 的选择**

| 场景 | 推荐值 | 理由 |
|------|--------|------|
| 频繁变更的测试环境 | 0 | 测试时变更频繁，重启时一次性写入即可 |
| 生产环境 | 30-60 | 平衡磁盘 IO 开销与数据丢失窗口 |
| 对数据丢失零容忍 | 考虑结合应用层定期备份 | 即使设置为 0，也存在进程异常终止时的丢失风险 |

**2. 监控建议**

可以通过以下方式监控持久化状态：

```sql
-- 查看后台工作进程是否运行
SELECT * FROM pg_stat_activity WHERE application_name = 'pg_stash_advice worker';

-- 检查日志中是否有加载/保存记录
-- 搜索 "loaded X advice stashes and Y entries" 和 "saved X advice stashes and Y entries"
```

**3. 异常处理**

如果数据库异常终止（如 kill -9），TSV 文件可能处于不完整状态。后台工作进程在启动时会先调用 `pgsa_reset_all_stashes()` 清空共享内存，然后重新加载文件，这样即使文件损坏，也只会丢失数据而不会导致系统不可用。

**4. 手动触发持久化**

如果需要立即保存当前状态（不等待下次持久化间隔），可以通过以下方式：

```sql
-- 设置 persist_interval=0 后，下次重启时自动保存
-- 或者直接调用 pg_start_stash_advice_worker() 启动后台工作进程
```

## 3. 小结和思考

### 核心价值总结

本 commit 为 `pg_stash_advice` 扩展引入了磁盘持久化机制，解决了该扩展长期以来的核心痛点——数据库重启后数据丢失的问题。通过后台工作进程和原子计数器的精巧设计，系统实现了低开销的变更检测和定期批量写入的能力。

**技术亮点**：

1. **原子状态管理**：`stashes_ready` 标志使用 `pg_atomic_flag` 实现无锁状态查询，在数据加载完成前有效地阻止了所有写操作
2. **变更检测 O(1)**：通过 `change_count` 原子计数器，后台工作进程可以在常数时间内判断是否有新数据需要写入
3. **原子性保证**：TSV 文件写入采用"先写临时文件，再原子重命名"（`durable_rename`）的方式，确保文件一致性
4. **后台工作进程**利用 PostgreSQL 内置的 `BackgroundWorker` 机制，实现了与主进程的安全隔离

### 对 DBA 的实际意义

1. **减少重复劳动**：一次设置，永久生效，无需每次重启后手动恢复建议
2. **提升生产稳定性**：对于包含人工调整执行计划的数据库，重启不再是噩梦
3. **简化运维流程**：建议的备份和迁移可以通过简单的文件复制实现

### 可能的改进方向

1. **压缩支持**：TSV 文件可能随时间增长变得很大，未来可考虑压缩存储
2. **多文件支持**：当前所有 stash 保存在单一文件中，未来可考虑按 stash 分离文件以提升并发性能
3. **增量同步**：当前是全量写入，未来可考虑基于 WAL 的增量同步方案
