## 背景  
使用 PG 16 的 pg_upgrade 升级 PG 11.22 — 可行性与注意事项  
  
### 可行性  
  
**完全可行。** 根据源码，`pg_upgrade` 支持从 9.2.X 及以上版本升级到当前版本。  
  
版本检查逻辑明确要求旧集群 >= 9.2，新集群必须是当前版本的 pg_upgrade 所对应的版本：  
  
  
  
### 针对 PG 11 升级的特有检查（会导致升级失败）  
  
以下检查在从 PG 11（版本号 `1100`）升级时会被触发，若检查不通过则**直接报错退出**：  
  
#### 1. `WITH OIDS` 表（PG 12 起不再支持）  
  
PG 11 允许 `CREATE TABLE ... WITH OIDS`，PG 12 起已移除。升级前必须处理：  
  
```sql  
ALTER TABLE ... SET WITHOUT OIDS;  
```  
  
#### 2. 用户自定义后缀运算符（PG 14 起不再支持）  
  
PG 11 允许用户定义 postfix operator，PG 14 起已移除。需在升级前 `DROP` 这些运算符：  
  
#### 3. 不兼容的多态函数（PG 14 起签名变更）  
  
引用了 `anyarray`/`anyelement` 类型参数的用户自定义对象，需在升级前 DROP，升级后改用 `anycompatiblearray`/`anycompatible` 重建：  
  
#### 4. 用户自定义编码转换函数（PG 14 起函数签名变更）  
  
PG 14 修改了编码转换函数的参数签名，旧版本的用户自定义编码转换无法自动升级，需先删除：  
  
  
  
### 通用注意事项  
  
#### 二进制兼容性要求  
  
新旧集群的以下编译参数必须一致，否则升级失败：  
- 32/64 位架构  
- `block_size`、`wal_block_size`、`wal_segment_size`  
- `max_identifier_length`、`max_index_keys`  
- `date_is_int`（日期/时间存储格式）  
- **数据校验和（data checksum）设置必须一致**  
  
#### 外部扩展（Extension）  
  
pg_upgrade 无法自动检查外部模块的二进制兼容性，需手动确保新集群中安装了对应版本的 `.so` 文件：  
  
#### `reg*` OID 引用类型  
  
用户表中若有列使用以下类型，**无法升级**：`regconfig`、`regdictionary`、`regnamespace`、`regoper`、`regoperator`、`regproc`、`regprocedure`（`regclass`、`regrole`、`regtype` 可以升级）：  
  
#### 逻辑复制槽（Logical Replication Slots）  
  
从 PG 11 升级时，逻辑复制槽**不会被迁移**（该功能从 PG 17 起才支持）：  
  
#### 未提交的 Prepared Transactions  
  
升级前旧集群中不能有未完成的 prepared transactions：  
  
  
  
### 升级后必须执行的操作  
  
升级完成后，优化器统计信息不会完整迁移，需运行：  
  
```bash  
vacuumdb --all --analyze-in-stages --missing-stats-only  
vacuumdb --all --analyze-only  
```  
  
  
  
### 推荐操作流程  
  
1. **先用 `--check` 模式预检**，不修改任何数据：  
   ```bash  
   pg_upgrade --check -b /old/bin -B /new/bin -d /old/data -D /new/data  
   ```  
2. 根据检查报告，在旧集群中处理 `WITH OIDS` 表、postfix 运算符、多态函数、编码转换等问题。  
3. 确保新集群已通过 `initdb` 初始化，且编译参数与旧集群兼容。  
4. 在新集群中安装所有扩展的共享库文件。  
5. 执行正式升级，**始终使用新版本（PG 16）的 `pg_upgrade` 二进制**。  
6. 升级后运行 `vacuumdb` 重建统计信息。
