## 背景 


本文档解读 Heikki Linnakangas 提交的 Lock Manager 内存管理改进，包括 `max_locks_per_transaction` 默认值从 64 增加到 128，以及锁哈希表分配策略的优化。

### 相关 Commit

| Commit | 描述 |
|--------|------|
| `79534f90657c` | Change default of max_locks_per_transaction to 128 |
| `e1ad034809a46` | Make the lock hash tables fixed-sized |
| `3e854d2ff1a6e` | Remove 10% safety margin from lock manager hash table estimates |
| `3fd057772827` | Refactor PredicateLockShmemInit to not reuse var for different things |
| `feb03dfecdc9` | Remove bogus "safety margin" from predicate.c shmem estimates |

### 核心价值

- **默认值翻倍**：从 64 增加到 128
- **内存可预测**：锁哈希表现在是固定大小，不再动态增长
- **更精确估算**：移除了过度的安全边际

## 原理详解

### 1. 核心数据结构

#### 1.1 NLOCKENTS 宏 (`src/backend/storage/lmgr/lock.c` line 59-60)

```c
/*
 * 计算锁条目的总数
 * = max_locks_per_xact * (MaxBackends + max_prepared_xacts)
 */
#define NLOCKENTS() \
    mul_size(max_locks_per_xact, add_size(MaxBackends, max_prepared_xacts))
```

这个宏计算的是每个后端能够持有的锁数量乘以总后端数（加上prepared事务数）。

#### 1.2 锁哈希表请求 (`LWLockShmemRequest()` line 416-435)

```c
LWLockShmemRequest(void *arg)
{
    int64   max_table_size;

    /*
     * 计算锁哈希表大小
     */
    max_table_size = NLOCKENTS();

    /*
     * 请求 LOCK 哈希表空间
     * 使用新的 ShmemRequestHash API
     */
    ShmemRequestHash(.name = "LOCK hash",
                     .nelems = max_table_size,
                     .ptr = &LockMethodLockHash,
                     .hash_info.keysize = sizeof(LOCKTAG),
                     .hash_info.entrysize = sizeof(LOCK),
                     .hash_info.num_partitions = NUM_LOCK_PARTITIONS,
                     .hash_flags = HASH_ELEM | HASH_BLOBS | HASH_PARTITION,
    );

    /* PROCLOCK 数量是 LOCK 数量的两倍 */
    max_table_size *= 2;

    ShmemRequestHash(.name = "PROCLOCK hash",
                     .nelems = max_table_size,
                     .ptr = &LockMethodProcLockHash,
                     .hash_info.keysize = sizeof(PROCLOCKTAG),
                     .hash_info.entrysize = sizeof(PROCLOCK),
                     .hash_info.hash = proclock_hash,
                     .hash_info.num_partitions = NUM_LOCK_PARTITIONS,
                     .hash_flags = HASH_ELEM | HASH_FUNCTION | HASH_PARTITION,
    );
}
```

### 2. 固定大小哈希表的实现

#### 2.1 `HASH_FIXED_SIZE` 标志

新的 `ShmemRequestHash` API 内部使用 `HASH_FIXED_SIZE` 标志，确保哈希表不会动态增长：

```c
/* 在 shmem_hash.c 中 */
if (flags & HASH_FIXED_SIZE)
{
    /* 确保哈希表不会动态增长 */
    info->dsize = -1;  /* 固定大小 */
    info->numentries = max_table_size;
}
```

#### 2.2 关键变化：使用新的 ShmemRequestHash API

**之前**（使用 `ShmemInitHash`）：

```c
ShmemInitHash("LockManager",
              init_table_size,    /* 初始大小 = max/2 */
              max_table_size,     /* 最大大小 */
              HASH_ELEM | HASH_BLOBS | HASH_PARTITION,
              &lockManageState);
```

**之后**（使用新的 `ShmemRequestHash`）：

```c
ShmemRequestHash(.name = "LOCK hash",
                 .nelems = max_table_size,
                 .ptr = &LockMethodLockHash,
                 ...);
```

### 3. 安全边际的移除

#### 3.1 移除的代码 (`src/backend/storage/lmgr/lock.c`)

**之前**（有 10% 安全边际）：

```c
/*
 * Since NLOCKENTS is only an estimate, add 10% safety margin.
 */
size = add_size(size, size / 10);
```

**之后**（移除安全边际）：

```c
/* 直接使用估算值，不再添加安全边际 */
```

#### 3.2 移除的原因

1. `hash_estimate_size()` 已经提供准确的大小估算
2. 如果需要更多空间，应该调整 `max_locks_per_transaction` 参数
3. 移除过度保守的边际可以节省内存

### 4. 默认值变化

#### 4.1 GUC 定义 (`src/backend/utils/misc/guc_parameters.dat`)

```
max_locks_per_transaction
    boot_val = '128'  # 之前是 '64'
```

#### 4.2 为什么需要增加默认值

| 因素 | 影响 |
|------|------|
| 移除安全边际 | 减少约 10% 可用空间 |
| 固定大小哈希表 | 不再有动态增长的空间 |
| **净效果** | 有效锁容量减少 |
| **补偿** | 默认值从 64 增加到 128 |

```
之前：64 * (1 + 0.1 safety margin) ≈ 70 实际可用
之后：128 * 1.0 = 128 实际可用

实际上增加了约 83% 的有效容量
```

### 5. FastPathLockGroups 的计算 (`src/backend/utils/init/postinit.c` line 596)

```c
/*
 * 计算 FastPath 锁组的数量
 * = max_locks_per_xact / 平均每个事务的锁数
 */
FastPathLockGroups = max_locks_per_xact / 16;  /* 平均每个事务约需 16 个锁 */
```

## 使用实践

### 1. 查看当前配置

```sql
-- 查看当前值
SHOW max_locks_per_transaction;

-- 查看实际使用的锁空间
SELECT * FROM pg_shmem_allocations WHERE name LIKE '%Lock%';
```

### 2. 如果之前手动设置了该值

**重要**：如果之前在 `postgresql.conf` 中设置了 `max_locks_per_transaction`，新版本的默认值调整**不会**影响你：

```
This only changes the default, so if you had changed
max_locks_per_transaction in postgresql.conf, you will still have
fewer locks available than before for the same setting value.
```

**建议**：如果之前设置为 64，可能需要增加到 128 以获得相同的有效容量。

### 3. 监控锁使用情况

```sql
-- 查看当前锁使用
SELECT
    locktype,
    database,
    relation::regclass,
    mode,
    granted,
    count(*)
FROM pg_locks
GROUP BY
    locktype,
    database,
    relation,
    mode,
    granted
ORDER BY count(*) DESC;

-- 查看每个后端的锁数量
SELECT
    pid,
    granted,
    count(*)
FROM pg_locks
GROUP BY pid, granted
ORDER BY count(*) DESC;
```

### 4. 计算推荐值

```sql
-- 计算推荐值：当前值 * 2
-- 如果之前用 64，现在建议用 128

-- 查看最大并发后端数
SHOW max_connections;

-- 查看 prepared 事务数
SHOW max_prepared_transactions;

-- 计算公式
-- NLOCKENTS = max_locks_per_xact * (MaxBackends + max_prepared_xacts)
```

### 5. 调整该参数

#### 5.1 临时调整（会话级）

```sql
SET max_locks_per_transaction = 256;
```

#### 5.2 永久调整（postgresql.conf）

```conf
# 增加以支持更多并发锁
max_locks_per_transaction = 256
```

### 6. 常见场景的推荐值

| 场景 | max_connections | max_locks_per_transaction |
|------|----------------|---------------------------|
| 小型应用 | 100 | 128（默认）|
| 中型应用 | 500 | 256 |
| 大型应用 | 1000 | 512 |
| 存储过程密集型 | 200 | 512（需要更多锁）|

### 7. 诊断锁相关问题

#### 7.1 锁等待超时

```sql
-- 如果遇到 "FATAL: out of shared memory"
-- 说明 max_locks_per_transaction 不足

-- 查看当前锁数量与最大值的比例
SELECT
    count(DISTINCT lock) as total_locks,
    count(DISTINCT pid) as num_holders
FROM pg_locks;
```

#### 7.2 长时间锁等待

```sql
-- 查看等待中的锁
SELECT
    a.pid,
    a.usename,
    a.query,
    l.locktype,
    l.mode,
    l.relation::regclass,
    l.granted
FROM pg_stat_activity a
JOIN pg_locks l ON a.pid = l.pid
WHERE NOT l.granted
ORDER BY a.query_start;
```

### 8. 与其他参数的关系

| 参数 | 作用 | 与 max_locks_per_transaction 的关系 |
|------|------|-------------------------------------|
| `max_connections` | 最大后端数 | 乘数关系 |
| `max_prepared_transactions` | prepared 事务数 | 影响总锁容量 |
| `max_locks_per_xact` | 每个事务的锁数 | 同义词 |
| `shared_memory_size` | 共享内存大小 | 锁哈希表占用其中一部分 |

## 总结

Lock Manager 的这次调整是 PostgreSQL 内存管理的优化，主要变化：

**关键要点**：

1. **默认值翻倍**：从 64 增加到 128，提供更多锁空间
2. **固定大小哈希表**：不再动态增长，内存使用更可预测
3. **移除安全边际**：不再有过度的保守估算
4. **向后兼容**：如果在 postgresql.conf 中设置了值，不受影响
5. **建议**：如果之前手动设为 64，考虑增加到 128 以获得相同有效容量

**性能影响**：

- 内存占用略有增加（因为默认哈希表更大）
- 但避免了运行时动态调整的开销
- 锁管理更加可预测
