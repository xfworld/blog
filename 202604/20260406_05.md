## 背景 
  
本文深入解读了 PostgreSQL 近期对锁管理器（Lock Manager）共享内存分配机制的五项关键优化。这些优化统一围绕一个核心目标：**消除锁表大小估算中的"安全 margin"，并将 LOCK 与 PROCLOCK 两个哈希表改为固定大小分配，从而使锁管理器的内存使用行为更加确定、可预测**。

  

**背景痛点**：在旧的实现中，PostgreSQL 的锁管理器通过 `max_locks_per_transaction` 参数估算所需共享内存，并额外预留 10% 作为"安全 margin"。更关键的是，LOCK 表和 PROCLOCK 表之间可以相互"借用"空间——这导致了一个诡异的现象：当一个表耗尽内存时，重启系统后另一个表反而可能有空间可用。这种**非确定性行为**让 DBA 难以准确预估和配置锁资源，也给问题诊断带来了极大困扰。

**核心价值**：这组优化通过（1）移除虚假的 10% 安全 margin，（2）引入 `HASH_FIXED_SIZE` 标志使两表大小固定隔离，（3）将默认 `max_locks_per_transaction` 从 64 提升至 128，在显著减少锁管理器内存浪费的同时，实现了内存分配的完全确定性。代价是同等配置下可用锁数量略有减少，但通过提高默认值得以补偿。

## 原理介绍

### 1. PostgreSQL 锁管理器概述

PostgreSQL 的锁管理器负责管理数据库内部的并发控制，涉及两种核心数据结构：

- **LOCK**：表示被锁定的对象（如表、页面、元组），按 `LOCKTAG` 唯一标识
- **PROCLOCK**：表示"进程 × 锁"的组合，即某个后端进程在某个锁上的具体持有状态

两者是一对多关系——一个 LOCK 可以被多个 PROCLOCK 持有。

这两个结构都存储在共享内存哈希表中，大小由 `max_locks_per_transaction` 参数决定。计算公式为：

```c
NLOCKENTS() = max_locks_per_transaction × (MaxBackends + max_prepared_xacts)
```

### 2. 旧实现的问题：动态借用与安全 margin

在优化前的实现中（参考 commit e1ad034809a46 修改前的代码），LOCK 和 PROCLOCK 哈希表使用 `ShmemInitHash` 创建，初始大小（`init_table_size`）设为估算值的 1/2，最大大小（`max_table_size`）为估算值。由于没有 `HASH_FIXED_SIZE` 标志，哈希表在达到初始大小后会动态扩展，**直到碰触最大大小为止**——但两个表的最大大小是分别计算的，共享内存池却是共用的。

这造成了以下问题：

```
共享内存池（两部分共用）
├── LOCK 表估算区（max_locks_per_xact × nbackends）
└── PROCLOCK 表估算区（LOCK 表估算值 × 2）
```

旧代码中，`init_table_size = max_table_size / 2`，两个表都可以在共享池中扩展，**LOCK 表可能"吃掉"本应属于 PROCLOCK 表的空间，反之亦然**。重启后分配又可能不同——这就是"非确定性"的根源。

### 3. 核心优化原理

#### 优化一：移除虚假的安全 margin（commit 3e854d2f 和 feb03dfe）

`hash_estimate_size()` 函数本身已经能较准确地根据元素数量和单元素大小估算所需内存桶数。旧代码额外加了 10% 的 "safety margin"，但这个 margin 实际上从未被真正使用（因为后来引入了 `HASH_FIXED_SIZE` 标志后哈希表无法超限分配）。这 10% 的内存被白白预留了。

优化后直接移除了 `size = add_size(size, size / 10)` 这行代码，节省了这部分共享内存。

#### 优化二：固定哈希表大小（commit e1ad034809a46）

通过在 `ShmemRequestHash` 调用中添加 `HASH_FIXED_SIZE` 标志，使 LOCK 和 PROCLOCK 两个哈希表**严格锁定在预设大小**，不允许动态扩展：

```c
ShmemRequestHash(.name = "LOCK hash",
                 .nelems = max_table_size,
                 .ptr = &LockMethodLockHash,
                 .hash_info.keysize = sizeof(LOCKTAG),
                 .hash_info.entrysize = sizeof(LOCK),
                 .hash_info.num_partitions = NUM_LOCK_PARTITIONS,
                 .hash_flags = HASH_ELEM | HASH_BLOBS | HASH_PARTITION | HASH_FIXED_SIZE,
    );
```

同样 PROCLOCK 表也加上 `HASH_FIXED_SIZE`。这样两个表**各自严格占用各自计算好的那部分空间，互不侵犯**。

#### 优化三：调高默认值以补偿（commit 79534f90657c）

移除安全 margin 和固定两表大小后，相同 `max_locks_per_transaction` 配置下可用锁数量会有所减少。为补偿这一变化，将默认值从 **64 提升到 128**——约等于翻倍，使得两表的实际空间与优化前"大致持平"。

> 规则：如果用户之前设置了自定义值，想恢复同等能力，应将 `max_locks_per_transaction` **翻倍**。

#### 优化四：predicate.c 的同步清理（commit 3fd057772827 和 feb03dfe）

predicate.c（谓词锁/可序列化事务相关）同样有类似的 10% safety margin 代码，且其哈希表早已使用 `HASH_FIXED_SIZE`（commit 7c797e7194 引入），意味着那个 margin 从未生效。这次一并清理，并做了代码重构——将重复使用的 `max_table_size` 变量拆分为语义更明确的 `max_predicate_lock_targets`、`max_predicate_locks`、`max_serializable_xacts` 和 `max_rw_conflicts`，避免混淆。

### 4. 共享内存初始化流程

```
postmaster 启动
  └── LockManagerShmemRequest()     ← 估算并请求共享内存
        ├── LOCK hash 大小 = NLOCKENTS()
        ├── PROCLOCK hash 大小 = NLOCKENTS() × 2
        └── （两者都带 HASH_FIXED_SIZE，不允许扩展）

postgres 后端启动
  └── InitLockManagerAccess()       ← 创建本地 LOCALLOCK 哈希表（非共享）
```

## 应用场景及最佳实践

### 场景一：大型批量导入业务

**优化前（旧版默认 max_locks_per_transaction=64）**：

```
假设场景：200 个并发后端，每个事务需要锁定 50 个不同表对象

LOCK 表最大容量 ≈ 64 × 200 = 12,800 个锁对象
PROCLOCK 表最大容量 ≈ 12,800 × 2 = 25,600 个进程-锁组合

实际运行时：
- 大批量导入期间，LOCK 表快速填满
- 由于两表共享内存且无 HASH_FIXED_SIZE，LOCK 表"借用"了 PROCLOCK 的空间
- 后续事务需要新的 PROCLOCK 时发现空间不足
- 但如果重启数据库，重分配后可能 PROCLOCK 有空间而 LOCK 不足——行为不确定
```

**优化后（max_locks_per_transaction=128）**：

```
LOCK 表最大容量 ≈ 128 × 200 = 25,600 个锁对象
PROCLOCK 表最大容量 ≈ 25,600 × 2 = 51,200 个进程-锁组合

- 两表大小固定，互不侵犯
- 行为完全可预测：容量由配置值决定
- 默认值翻倍后，总容量与优化前基本持平
- 重启前后行为一致，不再出现"重启后就好了"的现象
```

**DBA 操作建议**：
```bash
# 检查当前锁表使用情况（优化前难以准确预估）
SELECT locktype, database, relation, mode, count(*)
FROM pg_locks
GROUP BY locktype, database, relation, mode
ORDER BY count DESC LIMIT 20;

# 如果经常触达上限，估算所需值
# 经验公式：max_locks_per_transaction ≈ 单事务最大锁对象数 × 1.5
# 例如：单事务最多锁 80 个对象 → 设置为 120
```

### 场景二：可序列化事务的谓词锁

**优化前 predicate.c 的问题**：

predicate.c 中对可序列化事务维护了多张表：
- `PREDICATELOCKTARGET` 表：记录被锁定的目标
- `PREDICATELOCK` 表：记录每个事务-目标组合
- `SERIALIZABLEXID` 表：记录可序列化事务
- `RWConflictPool`：读写冲突池

旧代码中：
```c
/* 这 10% 的 safety margin 从未被使用，因为 HASH_FIXED_SIZE 已启用 */
size = add_size(size, size / 10);  /* ← 删除这行 */
```

删除这个无效的 margin 后，主锁管理器反而能获得更多内存。

**DBA 可观察到的变化**：

```bash
# 查看共享内存中各子系统的实际占用
SELECT name, setting, unit FROM pg_settings
WHERE name LIKE '%locks_per_transaction%';

# 监控锁等待情况
SELECT wait_event_type, wait_event, count(*)
FROM pg_stat_activity
WHERE state = 'active'
GROUP BY wait_event_type, wait_event;
```

### 场景三：配置迁移指南

**对于已有自定义配置的用户**：

| 优化前配置值 | 优化后等效配置值 | 说明 |
|------------|---------------|------|
| 64（默认） | 128（自动补偿） | 无需修改 |
| 100 | 200 | 如需维持同等能力 |
| 32 | 64 | 如负载不高，可维持现状 |

**升级检查清单**：
```sql
-- 1. 检查当前值
SHOW max_locks_per_transaction;
SHOW max_predicate_locks_per_transaction;

-- 2. 评估峰值使用（历史经验值）
SELECT count(*), locktype FROM pg_locks GROUP BY locktype;

-- 3. 预估新需求
--    新容量 = (新max_locks / 旧max_locks) × 旧容量
--    如从64升到128，容量翻倍
```

## 小结和思考

### 核心要点回顾

1. **移除虚假 10% 安全 margin**：这 10% 的额外内存预留实际上从未被使用（因为 `HASH_FIXED_SIZE` 启用后哈希表无法超限分配），删除它只是"还债"，不影射实际行为

2. **固定 LOCK/PROCLOCK 哈希表大小**：引入 `HASH_FIXED_SIZE` 后，两个表的内存占用严格固定，消除了"重启前 LOCK 不够但 PROCLOCK 够，重启后反过来"的诡异现象

3. **默认值翻倍补偿**：将 `max_locks_per_transaction` 从 64 提升到 128，保持整体容量与优化前基本持平

4. **语义重构提升可维护性**：predicate.c 的重构将模糊的 `max_table_size` 变量替换为语义明确的四个独立变量，降低了未来维护出错的风险

### 对 DBA 的实际影响

- **升级后**：如果保持默认配置，锁容量基本不变；如果使用自定义配置，需要评估是否翻倍以维持同等能力
- **可预测性提升**：锁表大小的行为现在完全由 `max_locks_per_transaction` 决定，不再有"玄学"般的重启后差异
- **内存效率改善**：释放了从未真正使用的 10% margin 和被虚假借用的空间

### 延伸思考

这组优化体现了 PostgreSQL 在工程化方向上的持续精进——不是追求功能的新颖，而是打磨系统的确定性和可预测性。对于大型数据库运维而言，**"行为可预测"往往比"平均性能更高"更重要**，因为瓶颈往往出现在尾部延迟而非平均值上。锁管理器作为 PostgreSQL 并发控制的核心组件，这次的优化虽不张扬，却实实在在提升了生产环境的可管理性。
