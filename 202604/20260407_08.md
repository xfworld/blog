## 背景 

本文深入解读 PostgreSQL 提交 `9f3755ea07aa8858999bdac3d33151e93d2643e2` 对 pg_trgm 扩展排序和去重操作的性能优化。该优化通过模板化 qsort 使编译器更容易内联比较函数，以及为去重操作设计专门的相等性判断函数，减少了比较回调/间接调用相关开销，预期可提升 trigram 生成场景下的执行效率，幅度需以 benchmark 为准。

  

**背景痛点**: pg_trgm 是 PostgreSQL 模糊匹配和相似度搜索的核心扩展，广泛应用于 LIKE 模式匹配、相似性搜索（% 操作符）。在 pg_trgm 相关查询中，查询串/模式串的 trigram 生成、排序、去重是高频路径之一，原有实现使用间接函数指针调用，编译器无法内联比较函数，可能导致分支预测效率降低和缓存效率下降。注意：正则表达式 `~/~*` 在 GIN/GiST 查询提取阶段直接走 `createTrgmNFA()`（`trgm_gin.c:117-119`、`trgm_gist.c:261-263`），不经过本次 commit 改动的排序/去重路径。

**核心价值**: 该优化将排序比较函数内联化，并简化去重判断逻辑，在高频文本相似度计算场景（模糊搜索预处理、模式串 trigram 提取、LIKE/ILIKE 查询预处理）下可减少间接调用开销、提升编译器优化空间。本次优化凡走 `generate_trgm()` / `generate_wildcard_trgm()` 的路径均会受益，属于直接受益的操作符/场景包括：`%`（similarity）、`=`、LIKE/ILIKE，以及 GiST KNN 距离相关的 `<->` / `<->>` / `<->>>` 查询串预处理或距离支持函数缓存路径（`gtrgm_distance()` 对查询串调用 `generate_trgm()`，见 `trgm_gist.c:453`）；regex 例外，走 `createTrgmNFA()` 不经此路径。SQL 层 `<->` 距离操作符经 `similarity()` 间接调用 `generate_trgm()`（`trgm_op.c:1347-1348`），同样直接受益。`show_trgm()` 等非索引路径亦然。

word_similarity（`<%` / `%>`）、strict_word_similarity（`<<%` / `%>>`）的 SQL 层计算本体不直接受益——二者核心计算路径本身调用 `calc_word_similarity()`，内部通过 `generate_trgm_only()` 生成 trigram 后对 `pos_trgm`（positional trigram）执行 `qsort(ptrg, ..., comp_ptrgm)`（`trgm_op.c:885`），不走本次 commit 改动的 `generate_trgm()` / `generate_wildcard_trgm()` 排序去重路径。但 GiST KNN 距离操作符 `<->>` / `<->>>` 在 `gtrgm_distance()`（`trgm_gist.c:469`）中对查询串调用 `generate_trgm()`，相关索引查询路径中的查询串 trigram 生成仍会受益。

## 1. 原理介绍

### 1.1 pg_trgm 与 Trigram 机制

pg_trgm 是 PostgreSQL 的扩展模块，通过将字符串中的每个单词拆解为带前后空格的 trigram 集合来实现高效的模糊匹配。pg_trgm 按 locale/encoding 感知的字母数字字符划分单词（`ISWORDCHR` 内部调用 `t_isalnum_with_len()`），默认构建下比较是大小写不敏感的（`IGNORECASE` 会让 `generate_trgm`/`generate_wildcard_trgm` 在提取前做 case-folding）。每个单词在提取 trigram 时会被视为带有 LPADDING=2（两个前缀空格）和 RPADDING=1（一个后缀空格）。具体过程如下：

```
字符串 "post" 提取的 trigram（提取阶段示意，generate_trgm 在返回前会排序去重）:
  "  p"  (2 前导空格 + p)
  " po"  (空格 + p + o)
  "pos"  (p + o + s)
  "ost"  (o + s + t)
  "st "  (s + t + 1 后缀空格)

字符串 "foo|bar" 会按 | 分隔为两个单词分别提取:
  "  f", " fo", "foo", "oo "   -- foo 部分
  "  b", " ba", "bar", "ar "   -- bar 部分
```

> **注意**：以上示例均针对单字节字符场景。内部 `trgm` 的类型为 `typedef char trgm[3]`——即每个 trigram 在内存中始终用 3 个字节表示。`compact_trigram()`（trgm_op.c:374）处理的是"三个可能为多字节的字符"，当这三个字符合起来恰好为 3 字节时（如拉丁字母）直接存储这 3 个原始字节；否则（如多字节 UTF-8 字符，总字节数大于 3）将其压缩为由 CRC32 派生的 3 字节表示——具体为先计算 `pg_crc32`，再取其 3 字节（见 `trgm_op.c:382-391`）。因此，"概念上的三个连续字符"与"内部的 3 字节表示"并不总是一一对应。

`generate_trgm()` 自身完成的语义是：为字符串生成 trigram 数组 → 排序 → 去重，最终返回已排序、去重后的数组。这一步在 `show_trgm()`（第 1180 行）、`similarity()`（第 1347-1348 行）等非索引路径中同样会被调用。

**在索引查询场景中**，上述已排序去重的 trigram 数组会继续参与以下流程：
1. 通过 GIN/GiST 索引查找匹配的候选行（查询串预处理阶段）
2. 对候选行进行谓词过滤与重检查（堆级 recheck），以及是否会再次执行 SQL 层相似度/模式匹配计算，取决于访问方法和策略：GiST 上 SimilarityStrategyNumber（`%`）为 exact，recheck = false（`trgm_gist.c:316-319`）；GIN 上则统一 inexact，recheck = true（`trgm_gin.c:187-188`）

### 1.2 原有实现的性能瓶颈

优化前的代码使用了传统的函数指针调用方式：

```c
// 原始实现 (commit 前)
static int
comp_trgm(const void *a, const void *b)
{
    return CMPTRGM(a, b);  // 通过函数指针调用
}

// 调用方式
qsort(GETARR(trg), len, sizeof(trgm), comp_trgm);
```

问题在于 `CMPTRGM` 是一个**函数指针**（见 `trgm.h:48`），而 `comp_trgm` 作为 `qsort()` 的比较器回调，会导致以下两层开销：

```c
extern int (*CMPTRGM) (const void *a, const void *b);
```

1. **通用 qsort 回调不可内联**：PostgreSQL 的 `qsort` 接口（通过 `port.h:495` 的宏映射到底层的 `pg_qsort`）需要通过比较器回调执行比较逻辑，编译器无法将 `comp_trgm` 的比较逻辑内联到排序代码路径中——这是通用排序框架的比较器回调阻止内联的固有代价
2. **CMPTRGM 的运行时选择/间接调用**：首次调用时通过 `CMPTRGM_CHOOSE` 动态选择 signed 或 unsigned 版本，之后通过函数指针间接调用

这两层因素叠加，使得编译器难以对比较逻辑进行内联优化，并可能导致分支预测效率和缓存效率下降。

```c
// CMPTRGM_CHOOSE 的运行时选择逻辑
static int
CMPTRGM_CHOOSE(const void *a, const void *b)
{
    if (GetDefaultCharSignedness())
        CMPTRGM = CMPTRGM_SIGNED;  // 替换函数指针
    else
        CMPTRGM = CMPTRGM_UNSIGNED;
    return CMPTRGM(a, b);
}
```

提交说明写得很直接：**Use templated qsort() so that the comparison function can be inlined**——问题的本质是通用排序框架的比较器回调阻止了内联，而不仅仅是函数指针间接调用本身。

### 1.3 模板化 qsort 实现比较函数内联

sort_template.h 是 PostgreSQL 提供的一个巧妙机制：它通过 C 预处理的模板展开，在编译期生成专门化的排序函数。核心思想是**用宏参数控制代码生成**。

在 `trgm_op.c` 中的使用方式：

```c
// 第一步：定义 signed 版本的排序函数
#define ST_SORT trigram_qsort_signed
#define ST_ELEMENT_TYPE_VOID        // 元素类型为 void，需要额外传入 element_size
#define ST_COMPARE(a, b) CMPTRGM_SIGNED(a, b)  // 内联这个比较表达式
#define ST_SCOPE static
#define ST_DEFINE                  // 生成函数定义
#define ST_DECLARE                 // 生成函数声明
#include "lib/sort_template.h"     // 触发模板展开

// 第二步：定义 unsigned 版本的排序函数
// 注意：sort_template.h 在每次 #include 后会 #undef 掉所有 ST_* 宏，
// 所以所有宏都必须重新定义，不能只写 ST_SORT 和 ST_COMPARE
#define ST_SORT trigram_qsort_unsigned
#define ST_ELEMENT_TYPE_VOID
#define ST_COMPARE(a, b) CMPTRGM_UNSIGNED(a, b)
#define ST_SCOPE static
#define ST_DEFINE
#define ST_DECLARE
#include "lib/sort_template.h"

// 第三步：包装函数，根据运行时环境选择
static void
trigram_qsort(trgm *array, size_t n)
{
    if (GetDefaultCharSignedness())
        trigram_qsort_signed(array, n, sizeof(trgm));
    else
        trigram_qsort_unsigned(array, n, sizeof(trgm));
}
```

**关键区别**：在 sort_template.h 展开后，`CMPTRGM_SIGNED(a, b)` 这个宏会被直接内联到模板生成的专用排序实现中，使编译器更容易内联比较逻辑，而不是通过函数指针间接调用。

### 1.4 专门的相等性判断函数

`qunique()` 函数的语义是：**输入已排序的数组，移除相邻的重复元素**。它只需要判断**相等性**，不需要判断大小关系。

原有实现复用排序的比较函数：

```c
// 原始实现 - 使用完整的排序比较函数
qunique(array, n, sizeof(trgm), comp_trgm);  // comp_trgm 返回 -1, 0, 1
```

优化后使用专门的相等性判断函数：

```c
// 优化后 - 只判断是否相等
static inline int
CMPTRGM_EQ(const void *a, const void *b)
{
    char *aa = (char *) a;
    char *bb = (char *) b;

    // 相等返回 0，不等返回 1
    // 不需要返回值 -1/1 来表示大小关系
    return aa[0] != bb[0] || aa[1] != bb[1] || aa[2] != bb[2] ? 1 : 0;
}
```

**优化点**：
- `CMPTRGM_EQ` 为 `static inline`，且只做相等判断（不等返回 1，相等返回 0），比排序比较函数更简单
- **短路求值**：一旦找到不相等的字符立即返回
- **简化返回值**：只需区分 0（相等）和非 0（不等），无需返回 -1 或 1
- **潜在收益**：为编译器进一步优化提供机会，例如常量传播、死代码消除；但可观测收益需以编译结果和 benchmark 为准——`qunique()` 仍是函数指针接口，能否完全内联取决于编译器对 `qunique()` 及其比较器参数的整体优化情况

查看 qunique 的实现（`src/include/lib/qunique.h`）：

```c
static inline size_t
qunique(void *array, size_t elements, size_t width,
        int (*compare) (const void *, const void *))
{
    char *bytes = (char *) array;
    size_t i, j;

    if (elements <= 1)
        return elements;

    for (i = 1, j = 0; i < elements; ++i)
    {
        // 比较函数只需返回 0（相等）或非 0（不等）
        if (compare(bytes + i * width, bytes + j * width) != 0 &&
            ++j != i)
            memcpy(bytes + j * width, bytes + i * width, width);
    }

    return j + 1;
}
```

### 1.5 优化后的调用点

修改涉及两处调用点，均在 `trgm_op.c` 中：

**`generate_trgm` 函数中的排序去重调用**：
```c
// 优化前
qsort(GETARR(trg), len, sizeof(trgm), comp_trgm);
len = qunique(GETARR(trg), len, sizeof(trgm), comp_trgm);

// 优化后
trigram_qsort(GETARR(trg), len);
len = trigram_qunique(GETARR(trg), len);
```

**`generate_wildcard_trgm` 函数中的排序去重调用**：
```c
// 优化前
qsort(GETARR(trg), len, sizeof(trgm), comp_trgm);
len = qunique(GETARR(trg), len, sizeof(trgm), comp_trgm);

// 优化后
trigram_qsort(GETARR(trg), len);
len = trigram_qunique(GETARR(trg), len);
```

## 2. 应用场景及最佳实践

### 2.1 场景一：大规模模糊搜索

> **说明**：本节所说"排序优化"是指 trigram 数组内部排序，不是 SQL Sort 节点，也不是 GiST KNN 排序能力本身。

**场景描述**：电商平台的商品搜索功能，用户输入 "iphoe"（拼写错误）期望找到 "iPhone" 相关商品。

```sql
-- 创建 pg_trgm GIN 索引
CREATE INDEX idx_products_name_trgm ON products USING gin(name gin_trgm_ops);

-- 用户搜索 'iphoe'
SELECT name, similarity(name, 'iphoe') as sim
FROM products
WHERE name % 'iphoe'
ORDER BY sim DESC
LIMIT 10;
```

> **补充说明**：此查询中 GIN 索引只加速 `%` 谓词过滤，`ORDER BY sim DESC` 仍是普通排序；若要利用索引做 Top-N 相似结果排序，应改用 GiST + 距离操作符，如 `WHERE name % 'iphoe' ORDER BY name <-> 'iphoe' LIMIT 10`。注意：`ORDER BY name <-> 'iphoe' LIMIT 10`（不加 `%` 过滤）是"全表取最近的 10 个"的语义，与原查询"先用 `%` 阈值过滤、再按 similarity 排序"不同，这是另一种 Top-N 最近邻语义的查询，不是原查询的等价改写。

**优化效果**：当输入字符串较长、需要频繁重复生成 trigram，或整体执行时间中预处理占比较高时，更可能观察到收益。候选集较大通常意味着索引扫描、recheck、similarity() 计算等后续成本更重，不应作为正向条件。实际收益取决于输入长度、字符分布和编译器/CPU 环境，建议以 benchmark 为准。

### 2.2 场景二：LIKE/ILIKE 模式匹配加速

**场景描述**：新闻系统的标签检索，假设 tags 为普通 text 列，支持 LIKE/ILIKE 模式匹配加速。

```sql
-- 创建带 pg_trgm 索引的 text 列
CREATE INDEX idx_articles_tags_trgm ON articles USING gin(tags gin_trgm_ops);

-- 使用 ILIKE 模式匹配（大小写不敏感），GIN 索引可利用 trigram 提取条件加速扫描
SELECT id, title FROM articles WHERE tags ILIKE '%database%';
```

**优化效果**：LIKE/ILIKE 匹配时，查询模式串会经过 `generate_wildcard_trgm` 提取 wildcard 模式中所有匹配结果必然包含的 trigrams，再进行排序去重，这两步均受益于模板化实现。实际收益取决于模式复杂度、字符分布和硬件环境，建议以 EXPLAIN ANALYZE 为准。

> **补充说明**：正则表达式 `~` / `~*` 走不同的内部路径。GIN/GiST 在查询提取阶段调用 `createTrgmNFA()`（实现细节见 `trgm_gin.c:117-119`、`trgm_gist.c:261-263`），由 `trgm_regexp.c` 完成正则 trigram 提取和 NFA 图构建；对正则提取可确定性的 trigram 并据此做索引过滤；文档层面说明正则会提取 trigrams 并据此做索引搜索（见 `doc/src/sgml/pgtrgm.sgml:538-543`）。

### 2.3 场景三：高频相似度计算预处理

**场景描述**：在需要对大量字符串进行相似度计算或模糊匹配的系统中，`generate_trgm()` / `generate_wildcard_trgm()` 的排序去重是预处理阶段的高频路径。

> **边界说明**：本次优化仅覆盖 `generate_trgm()` / `generate_wildcard_trgm()` 内部的 trigram 排序去重。`word_similarity()` / `strict_word_similarity()` 后续还有 `pos_trgm`（positional trigram）的 `qsort(ptrg, ..., comp_ptrgm)` 排序（`trgm_op.c:885` 附近），这部分仍走旧路径，未被本次 commit 优化。

以下为 pg_trgm 内部执行流程的伪代码说明（`generate_trgm` / `generate_wildcard_trgm` 在 `contrib/pg_trgm/trgm.h` 中声明、在 `contrib/pg_trgm/trgm_op.c` 中实现的内部 C 函数，不对 SQL 层暴露）：

> **注意**：`generate_trgm` 的真实签名为 `TRGM *generate_trgm(char *str, int slen)`（见 `contrib/pg_trgm/trgm.h:122`），返回封装后的 `TRGM *` 指针。`generate_trgm` 内部已经完成生成、排序、去重，返回时数组已唯一有序，调用者不需要再次排序去重。

```c
// 示意性伪代码（不可直接编译）：查询串 "iphoe" 的 trigram 生成流程
char buf[] = "iphoe";                      // 需放入可写缓冲区
TRGM *trg = generate_trgm(buf, 5);         // 内部完成生成、排序、去重
size_t len = ARRNELEM(trg);                // 获取实际 trigram 数量（已唯一有序）
// 后续通过 GIN/GiST 索引进行谓词过滤或 similarity() 计算
```

**列 vs 常量/参数的实际 SQL 示例**：

```sql
-- 查找与给定关键词相似度最高的文章
SELECT id, title, similarity(content, $1) AS sim
FROM articles
WHERE content % $1
ORDER BY sim DESC
LIMIT 10;
```


**优化效果**：排序去重路径走模板专用化实现，减少比较回调开销，具体收益取决于输入长度、字符分布和编译环境。

### 2.4 最佳实践建议

> **说明**：本节所说"排序优化"是指 trigram 数组内部排序，不是 SQL Sort 节点，也不是 GiST KNN 排序能力本身。

1. **索引创建时机**：对于需要频繁进行模糊查询的表，优先创建 pg_trgm 索引。基于当前目录这版 PostgreSQL/pg_trgm 代码，GIN 与 GiST 均支持 %、%>、%>>、LIKE/ILIKE、正则表达式和 = 过滤（= 过滤是 pg_trgm 1.6 才补进 opfamily 的较新能力）；其中只有 GiST 支持基于距离操作符的 KNN 排序（如 `ORDER BY col <-> $1 LIMIT n`）。需注意：虽然 = 已纳入 opfamily（`pg_trgm--1.5--1.6.sql`），但 GIN/GiST 上的 = 仍是 inexact/recheck 路径（`trgm_gin.c:187`、`trgm_gist.c:350-352`），需要堆上重检查；精确等值查询的首选索引通常仍是 B-tree；pg_trgm 的优势主要在模糊匹配、相似度、LIKE/regex 等场景。

   GiST opclass 中 FOR ORDER BY 注册的距离操作符为 `<->`（相似度距离）、`<->>`（word similarity 距离）、`<->>>`（strict word similarity 距离）。SQL 层可写的 `<<->` 和 `<<<->` 是它们在 SQL 语义上的 commutator（`<<->` commutator 为 `<->>`，`<<<->` commutator 为 `<->>>`）。GiST KNN 常见写法既可以是列在左侧的 `col <-> query`，也可以是其 commutator 形式（如常量在左侧的 `'word' <<-> col`）；官方文档（`pgtrgm.sgml:507-521`）明确将后者列为 GiST 可高效实现的典型查询写法，为避免读者混淆，全文可统一采用一种写法，但不要暗示另一种写法不能有效使用 GiST KNN。

   ```sql
   -- GIN 索引（通用场景）
   CREATE INDEX idx_table_col_trgm ON table_name USING gin(col gin_trgm_ops);
   -- GiST 索引（需要 KNN 距离排序时）
   CREATE INDEX idx_table_col_trgm_gist ON table_name USING gist(col gist_trgm_ops);
   ```

2. **选择合适的操作符**：

   **布尔谓词**（返回 true/false，用于 WHERE 条件）：
   - `%` — 相似度谓词（需要 similarity_threshold GUC）
   - `<%` / `%>` — word similarity 谓词（互为 commutator，需要 word_similarity_threshold GUC）
   - `<<%` / `%>>` — strict word similarity 谓词（互为 commutator，需要 strict_word_similarity_threshold GUC）

   **距离/排序操作符**（返回 float4，用于 ORDER BY）：
   - `<->` — 相似度距离
   - `<<->` / `<->>` — word similarity 距离
   - `<<<->` / `<->>>` — strict word similarity 距离

3. **监控查询计划**：使用 `EXPLAIN ANALYZE` 确认索引被正确使用：

   ```sql
   EXPLAIN ANALYZE SELECT * FROM foo WHERE col % 'bar';
   ```

4. **调整相似度阈值**：根据业务需求调整阈值：

   ```sql
   SET pg_trgm.similarity_threshold = 0.3;  -- 默认 0.3
   ```

## 3. 小结和思考

### 3.1 技术总结

本次优化涉及 53 行新增、10 行删除（净增 43 行），在 pg_trgm 中应用现有排序模板与去重接口的优化：

1. **模板化代码生成**：通过 sort_template.h 实现编译期多态，减少间接调用相关开销，为编译器内联/优化创造条件

2. **内联优化**：比较函数内联后，编译器更容易进行常量传播、死代码消除、循环展开等高级优化——但能否实际触发取决于编译器的优化决策

3. **专业化分工**：排序比较（需要 -1/0/1）vs 相等性判断（只需 0/非0），不同的语义使用不同的实现

4. **性能收益来源**：
   - 减少了排序路径中的比较器回调/间接调用成本，并用专门的相等比较器优化了去重路径
   - 比较函数内联后，编译器可能获得更好的常量传播、死代码消除机会
   - **可能带来的潜在收益**：CPU 分支预测效率改善、指令缓存命中率提升、循环展开等编译器优化——但这些是否出现以及幅度多大，需要 benchmark / perf / 反汇编验证
   - **收益边界**：本次优化仅覆盖 `generate_trgm()` / `generate_wildcard_trgm()` 内部的 trigram 排序去重；`word_similarity()` / `strict_word_similarity()` 后续的 positional trigram 排序（`qsort(ptrg, ..., comp_ptrgm)`）仍走旧路径，未被优化

### 3.2 深层思考

**从函数指针到模板展开的范式转换**：这是一个典型的"用编译期计算换运行时效率"的案例。PostgreSQL 的 sort_template.h 机制在不改变外部行为/调用语义的前提下，通过预处理生成专门化代码。

**DBA 视角的启示**：虽然我们不需要理解底层汇编，但理解优化动机有助于：
- 判断某个扩展的性能特性
- 在评估新技术时关注高频路径的优化
- 理解 EXPLAIN ANALYZE 输出中的成本构成

**未来方向**：sort_template.h 的思路（通过模板专用化避免运行时函数指针开销）可作为参考，适用于部分高频、可专用化的比较路径。但是否可推广到 btree 比较函数、hash 聚合或 hashjoin 的哈希函数，需结合各模块的具体接口约束、排序支持能力和调用形态来判断。
