## 背景 


本文档解读 Daniel Gustafsson 提交的在线启用/禁用数据校验和功能，该功能允许在运行中的集群上启用或禁用数据校验和，无需限制对集群的访问。

### 核心价值

- **零停机**：在线启用/禁用校验和，无需停止数据库
- **数据完整性**：生产环境可启用校验和保护
- **操作简便**：通过 SQL 函数或 pg_checksums 工具即可操作

### 3. 状态转换机制

#### 3.1 ProcSignal Barrier 实现

校验和状态转换使用 ProcSignal Barrier 机制来协调所有后端：

**发射并等待屏障** (`src/backend/postmaster/datachecksum_state.c` line 370-394)：
```c
void
EmitAndWaitDataChecksumsBarrier(uint32 state)
{
    uint64 barrier;

    switch (state)
    {
        case PG_DATA_CHECKSUM_INPROGRESS_ON:
            barrier = EmitProcSignalBarrier(PROCSIGNAL_BARRIER_CHECKSUM_INPROGRESS_ON);
            WaitForProcSignalBarrier(barrier);
            break;

        case PG_DATA_CHECKSUM_INPROGRESS_OFF:
            barrier = EmitProcSignalBarrier(PROCSIGNAL_BARRIER_CHECKSUM_INPROGRESS_OFF);
            WaitForProcSignalBarrier(barrier);
            break;

        case PG_DATA_CHECKSUM_VERSION:
            barrier = EmitProcSignalBarrier(PROCSIGNAL_BARRIER_CHECKSUM_ON);
            WaitForProcSignalBarrier(barrier);
            break;

        case PG_DATA_CHECKSUM_OFF:
            barrier = EmitProcSignalBarrier(PROCSIGNAL_BARRIER_CHECKSUM_OFF);
            WaitForProcSignalBarrier(barrier);
            break;
    }
}
```

**处理屏障** (`src/backend/storage/ipc/procsignal.c`)：
```c
static void
ProcessProcSignalBarrier(SignalReason reason)
{
    switch (reason)
    {
        case PROCSIGNAL_BARRIER_CHECKSUM_INPROGRESS_ON:
            processed = AbsorbDataChecksumsBarrier(PG_DATA_CHECKSUM_INPROGRESS_ON);
            break;
        case PROCSIGNAL_BARRIER_CHECKSUM_ON:
            processed = AbsorbDataChecksumsBarrier(PG_DATA_CHECKSUM_VERSION);
            break;
        // ...
    }
}
```

#### 3.2 启用校验和的状态转换

**`src/backend/access/transam/xlog.c`**

1. **开始启用** - `SetDataChecksumsOnInProgress()` (line 4722-4761)：
```c
void
SetDataChecksumsOnInProgress(void)
{
    /* 发射屏障，通知所有后端切换到 INPROGRESS_ON 状态 */
    EmitProcSignalBarrier(PROCSIGNAL_BARRIER_CHECKSUM_INPROGRESS_ON);
    /* 等待所有后端确认 */
    WaitForProcSignalBarrier(PROCSIGNAL_BARRIER_CHECKSUM_INPROGRESS_ON);
    /* 更新共享状态 */
    SpinLockAcquire(&DataChecksumState->mutex);
    DataChecksumState->state = PG_DATA_CHECKSUM_INPROGRESS_ON;
    SpinLockRelease(&DataChecksumState->mutex);
}
```

2. **完成启用** - `SetDataChecksumsOn()` (line 4786-4842)：
```c
void
SetDataChecksumsOn(void)
{
    /* 发射屏障，通知所有后端切换到 ON 状态 */
    EmitProcSignalBarrier(PROCSIGNAL_BARRIER_CHECKSUM_ON);
    /* 等待所有后端确认 */
    WaitForProcSignalBarrier(PROCSIGNAL_BARRIER_CHECKSUM_ON);
    /* 更新共享状态 */
    SpinLockAcquire(&DataChecksumState->mutex);
    DataChecksumState->state = PG_DATA_CHECKSUM_VERSION;
    SpinLockRelease(&DataChecksumState->mutex);
}
```

static inline bool
DataChecksumsNeedWrite(void)
{
    return DataChecksumState->state == PG_DATA_CHECKSUM_VERSION ||
           DataChecksumState->state == PG_DATA_CHECKSUM_INPROGRESS_ON ||
           DataChecksumState->state == PG_DATA_CHECKSUM_INPROGRESS_OFF;

### 4. 后台工作进程实现

#### 4.1 Launcher 主函数 (`src/backend/postmaster/datachecksum_state.c` line 1013-1161)

```c
static uint32
DataChecksumsWorkerLauncherMain(uint32 slot, Datum main_argument)
{
    // 1. 获取操作类型（启用或禁用）
    DataChecksumsWorkerOperation operation = ...

    // 2. 获取目标数据库列表
    List *databases = GetAllDatabases();

    // 3. 对每个数据库启动 worker
    foreach(cell, databases)
    {
        char *dbname = (char *) lfirst(cell);
        LaunchDataChecksumsWorker(dbname, operation);
    }

    // 4. 等待所有 worker 完成
    WaitForDataChecksumsWorkers();
}
```

#### 4.2 Per-Database Worker 主函数 (line 1453-1636)

```c
static uint32
DataChecksumsWorkerMain(uint32 slot, Datum main_argument)
{
    // 1. 打开数据库连接
    BackgroundWorkerInitializeConnection(dbname, NULL);

    // 2. 获取该数据库中所有关系
    List *relations = GetAllRelationsInDatabase();

    // 3. 处理每个关系
    foreach(cell, relations)
    {
        ProcessSingleRelationByOid(relid);
    }

    // 4. 标记完成
    SpinLockAcquire(&DataChecksumState->mutex);
    DataChecksumState->worker_result = CHECKSUMS_SUCCESSFUL;
    SpinLockRelease(&DataChecksumState->mutex);
}
```

#### 4.3 核心处理函数 (line 628-701)

```c
ProcessSingleRelationFork(Oid relid, ForkNumber forkno)
{
    // 1. 打开关系
    rel = relation_open(relid, AccessShareLock);

    // 2. 遍历所有缓冲区
    for (buf = 0; buf < nbufs; buf++)
    {
        // 3.  pin buffer 并获取页面
        bufHdr = GetBufferDescriptor(buf);
        LockBuffer(bufHdr, BUFFER_LOCK_SHARE);

        // 4. 关键部分：标记缓冲区为脏
        START_CRIT_SECTION();
        MarkBufferDirty(buf);
        log_newpage_buffer(buf, false);
        END_CRIT_SECTION();

        UnlockBuffer(bufHdr, BUFFER_LOCK_SHARE);
    }

    // 5. 释放关系
    relation_close(rel, AccessShareLock);
}
```

### 5. 竞态条件修复 (commit d771b0a907e6)

修复的问题：如果 worker 在 launcher 开始等待之前完成，launcher 会错误地将 `BGWH_STOPPED` 视为错误。

**修复代码** (line 803-810)：
```c
/* 在 ProcessDatabase() 中 */
if (worker_result == BGWH_STOPPED)
{
    /*
     * Worker 可能在 launcher 调用 WaitForProcSignalBarrier() 之前
     * 就完成了。需要检查是否真正成功。
     */
    if (DataChecksumState->worker_result == CHECKSUMS_SUCCESSFUL)
        worker_result = worker_result; /* 保持成功状态 */
}
```

**修复还添加了正确的自旋锁保护** (line 1633-1635)：
```c
SpinLockAcquire(&DataChecksumState->mutex);
DataChecksumState->worker_result = CHECKSUMS_SUCCESSFUL;
SpinLockRelease(&DataChecksumState->mutex);
```

## 使用实践

### 1. 检查当前校验和状态

```sql
-- 检查集群级别的校验和状态
SHOW data_checksums;

-- 或者查看 pg_control
SELECT pg_controldata() LIKE '%checksum%';
```

### 2. 在线启用校验和

```bash
# 方式一：使用 SQL 函数（需要扩展）
SELECT pg_enable_data_checksums();

# 方式二：使用 pg_checksums 工具（PostgreSQL 17+）
pg_checksums --enable -D $PGDATA
```

### 3. 在线禁用校验和

```bash
# 使用 pg_checksums 工具禁用
pg_checksums --disable -D $PGDATA
```

### 4. 监控校验和处理进度

```sql
-- 查看后台工作进程状态
SELECT
    pid,
    application_name,
    state,
    query_start,
    query
FROM pg_stat_activity
WHERE application_name LIKE '%checksum%';

-- 查看已处理的关系数量
SELECT
    datname,
    relid::regclass,
    last_vacuum,
    last_autovacuum
FROM pg_stat_user_tables
ORDER BY last_autovacuum DESC;
```

### 5. 验证校验和是否生效

```sql
-- 强制检查点确保所有脏页被写出
CHECKPOINT;

-- 尝试读取一个页面来触发校验和验证
SELECT count(*) FROM any_table;
```

### 6. 性能影响

启用校验和后，写入操作会有额外开销：

```
校验和计算 = ~1-2% CPU 开销（取决于硬件）
    - AVX2 优化后开销更小
    - ARM CRC 扩展进一步优化

I/O 开销 = 无显著变化
    - 校验和在页面写出时计算
    - 不增加 I/O 次数
```

### 7. 与 pg_basebackup 的兼容性

```bash
# 启用校验和的集群可以使用标准的备份工具
pg_basebackup -D backup -Fp -Xs -P

# 恢复时校验和会自动验证
pg_restore backup
```

### 8. 常见问题处理

#### 8.1 校验和验证失败

```sql
-- 如果发现校验和不匹配，会在错误日志中报告
-- 可以通过 pg_checksums 工具检查
pg_checksums --check -D $PGDATA
```

#### 8.2 后台工作进程失败

```sql

```

## 总结

在线数据校验和功能是 PostgreSQL 数据完整性保护的重要增强。该功能解决了之前必须离线操作的问题，让生产环境也能方便地启用校验和保护。

**关键要点**：

1. **零停机**：后台工作进程在启用/禁用过程中处理所有脏页
2. **状态转换**：使用 ProcSignal Barrier 确保所有后端协调一致
3. **渐进式处理**：每个数据库独立处理，不阻塞正常操作
4. **性能可控**：校验和计算开销约 1-2%，可接受
5. **监控重要**：通过 `pg_stat_activity` 监控工作进程状态
