## PostgreSQL 19 preview - REPACK 不再和逻辑复制抢槽位: `max_repack_replication_slots` 来了

digoal  
  
### 日期

### 标签  
PostgreSQL , max_repack_replication_slots , REPACK , replication slot , 逻辑复制槽隔离 , 并发 REPACK , 复制槽池分区 , CheckSlotRequirements , ReplicationSlotCreate

## 背景 

这个提交 `e76d8c749c3152657711ed733f0aea61c0e36a91` 引入了一个全新的 GUC 参数 `max_repack_replication_slots`，专门为 `REPACK` 命令预留一批独立的复制槽，彻底解决了并发 REPACK 与常规逻辑复制槽之间的资源竞争问题。

  

背景痛点:

- `REPACK CONCURRENTLY` 底层依赖逻辑解码（Logical Decoding），执行时必须占用一个复制槽（replication slot）。
- 在此前，所有复制槽都从 `max_replication_slots` 这个池子里分配，一旦业务上同时运行着多个逻辑复制订阅和并发 REPACK，槽位很容易耗尽，导致 REPACK 报错 "all replication slots are in use"。
- 很多 DBA 的应对方式是把 `max_replication_slots` 设得很大，但这会无端增加 slot 维护开销和 WAL 保留压力，并不优雅。

这个提交带来的价值和意义:

- 复制槽资源按用途隔离：`max_replication_slots` 继续服务逻辑复制订阅，`max_repack_replication_slots` 专门服务 REPACK，两套池子互不干扰。
- 默认值为 5，大多数场景开箱即用，DBA 不必再反复调整 `max_replication_slots`。
- 完全向后兼容：普通逻辑复制行为不变，REPACK 专用槽只在需要时从专属池中分配。

## 原理介绍

### 1. 整体思路: 槽池分区

在理解这次改动之前，需要知道 PostgreSQL 的复制槽在共享内存中是一个连续的数组:

```
ReplicationSlotCtl->replication_slots[0 .. max_replication_slots + max_repack_replication_slots - 1]
```

提交前，这个数组的大小由 `max_replication_slots` 决定，所有槽混在一起，谁都可以申请。

提交后，数组从逻辑上被划分为**两段**:

```
索引范围 [0, max_replication_slots)         -> 普通复制槽（逻辑复制订阅、pg_create_logical_replication_slot 等）
索引范围 [max_replication_slots, max_replication_slots + max_repack_replication_slots) -> REPACK 专用槽
```

申请槽时传入一个 `repack` 布尔参数，落在哪一段由此决定。**两段池子独立计数、独立报错**，不存在跨池争抢。

### 2. 新增 GUC: `max_repack_replication_slots`

在 `src/backend/replication/slot.c` 中定义（默认值 5）:

```c
int max_repack_replication_slots = 5;   /* 默认保留 5 个槽给 REPACK */
```

对应的 GUC 注册（`src/backend/utils/misc/guc_parameters.dat`）:

```
{ name => 'max_repack_replication_slots', type => 'int', context => 'PGC_POSTMASTER',
  short_desc => 'Sets the maximum number of replication slots for use by REPACK.',
  boot_val => '5', min => '0', max => 'MAX_BACKENDS' }
```

这是一个 `PGC_POSTMASTER` 级别参数，必须在服务器启动时设置，不能通过 `ALTER SYSTEM` 动态修改——因为槽数组大小属于共享内存布局的一部分。

### 3. 核心函数签名变化

`ReplicationSlotCreate()` 和 `CheckSlotRequirements()` 的签名都新增了 `bool repack` 参数，调用方需要明确声明自己是不是为 REPACK 服务:

```c
// slot.h
void ReplicationSlotCreate(const char *name, bool db_specific,
                           ReplicationSlotPersistency persistency,
                           bool two_phase, bool repack, bool failover, bool synced);

// logical.h
void CheckLogicalDecodingRequirements(bool repack);
void CheckSlotRequirements(bool repack);
```

### 4. 槽分配的核心逻辑

在 `ReplicationSlotCreate()` 中（`slot.c` 第 440–454 行），槽搜索引入了起止边界:

```c
startpoint = !repack ? 0 : max_replication_slots;
endpoint   = max_replication_slots + (repack ? max_repack_replication_slots : 0);

for (int i = 0; i < max_replication_slots + max_repack_replication_slots; i++)
{
    ReplicationSlot *s = &ReplicationSlotCtl->replication_slots[i];

    if (s->in_use && strcmp(name, NameStr(s->data.name)) == 0)
        ereport(ERROR, ...);   // 名称冲突检测横跨整个数组

    // 只有落在自己池子范围内的空闲槽才能被选中
    if (i >= startpoint && i < endpoint && !s->in_use && slot == NULL)
        slot = s;
}
```

注意名称冲突检查是**横跨整个数组**的（无论是普通槽还是 REPACK 槽，已有的同名槽都会被检测到），但空闲槽位的分配严格限制在对应池子里。

当 REPACK 专用池耗尽时，报错 hint 也会相应变化:

```c
errhint("Free one or increase \"%s\".",
        repack ? "max_repack_replication_slots" : "max_replication_slots")
```

### 5. `CheckSlotRequirements()` 的分支检查

`slot.c` 第 1665–1688 行的 `CheckSlotRequirements()` 根据 `repack` 参数做不同的校验:

```c
if (!repack && max_replication_slots == 0)
    ereport(ERROR, errmsg("replication slots can only be used if "
                           "\"max_replication_slots\" > 0"));

if (repack && max_repack_replication_slots == 0)
    ereport(ERROR, errmsg("REPACK can only be used if "
                           "\"max_repack_replication_slots\" > 0"));
```

这意味着如果 DBA 把 `max_repack_replication_slots` 设为 0，REPACK 就彻底无法使用（而不是降级去抢普通槽）。

### 6. REPACK 工作进程是如何使用这套机制的

在 `repack_worker.c` 的 `repack_setup_logical_decoding()` 中（提交后）:

```c
CheckLogicalDecodingRequirements(true);   // 传入 true，表示 REPACK 用途

ReplicationSlotCreate(NameStr(slotname), true, RS_TEMPORARY,
                      false, true,        // <-- 新增的 repack=true
                      false, false);

ctx = CreateInitDecodingContext(REPL_PLUGIN_NAME, NIL,
                                 true,          // need_full_snapshot
                                 true,          // for_repack
                                 InvalidXLogRecPtr, ...);
```

普通逻辑复制函数（如 `pg_logical_slot_get_changes_guts`、`pg_create_logical_replication_slot`）则传入 `false`，始终从普通槽池中分配。

### 7. 共享内存大小计算的变化

所有遍历槽数组的循环和共享内存初始化都需要把两段池子加在一起计算:

```c
// ReplicationSlotsShmemRequest()
size = add_size(size,
    mul_size(max_replication_slots + max_repack_replication_slots,
             sizeof(ReplicationSlot)));
```

注意 `RestoreSlotFromDisk()` 中有一个特殊处理：它只遍历到 `max_replication_slots` 为止，不加 `max_repack_replication_slots`。注释解释了原因——这是为了在服务器重启后强制执行当前的 GUC 上限，防止曾经 shutdown 时的槽数超过新配置。

## 应用场景及最佳实践

### 场景一: 同时运行多个逻辑复制订阅 + 并发 REPACK

**提交前的问题:**

假设 `max_replication_slots = 10`，系统运行着 8 个逻辑复制订阅槽。此时只剩 2 个空闲槽。

业务在两个大表上同时执行 `REPACK CONCURRENTLY`，REPACK 每次需要 1 个临时槽，两个 REPACK 把剩余 2 个槽用光了。随后应用想再创建一个新的逻辑复制订阅——报错 "all replication slots are in use"，此时 DBA 必须手动增大 `max_replication_slots`，而且增大后所有订阅都要重新对接。

**提交后的改善:**

```ini
# postgresql.conf
max_replication_slots = 10        # 逻辑复制订阅用
max_repack_replication_slots = 5 # REPACK 专用
```

即使 8 个订阅槽全满，REPACK 仍然有独立的 5 个槽可用，两者互不影响。DBA 可以分别调大或调小 `max_repack_replication_slots`，不影响逻辑复制侧的配置。

### 场景二: REPACK 并发数超过可用槽

**提交前:**

只有一个池子，REPACK 和订阅争抢。当 REPACK 并发数过高时，业务方不得不压低 `max_replication_slots`，或者干脆禁用部分订阅的并发 REPACK 功能。

**提交后:**

`max_repack_replication_slots = 5` 默认值允许最多 5 个并发 REPACK。如果需要更多，只需:

```ini
max_repack_replication_slots = 10
```

不需要动 `max_replication_slots`，订阅侧完全不受影响。

### 场景三: 生产库希望彻底禁止 REPACK 使用任何槽

```ini
max_repack_replication_slots = 0
```

此时任何 `REPACK` 命令都会直接报错:

```
REPACK can only be used if "max_repack_replication_slots" > 0
```

这提供了比以往更精细的管控手段。

## 小结和思考

这个提交的核心贡献是**资源隔离**——通过把复制槽数组在逻辑上切分成两段，让 REPACK 拥有了独立于逻辑复制的专属槽池。这是 PostgreSQL 在维护性和运维灵活性上的又一次小步改进。

几个值得 DBA 关注的点:

1. **默认值 5 是经过考量的**: 5 个并发 REPACK 对大多数业务场景已经足够，如果真的遇到不够的情况，说明业务对 REPACK 的依赖程度很高，应该重新评估表的设计。
2. **`max_repack_replication_slots` 是 static 池，不支持动态修改**: 这是 Postmaster 级别参数。如果要调整，必须重启实例。这意味着 DBA 在上线前应该评估好两个池的大小。
3. **错误 hint 的精确化**: 当槽耗尽时，PostgreSQL 现在会明确告诉 DBA 应该调哪个参数（`max_replication_slots` 还是 `max_repack_replication_slots`），这让排障更直接。
4. **不影响物理复制槽**: `max_replication_slots` 同时服务物理复制和逻辑复制，而 `max_repack_replication_slots` 只服务 REPACK 这个特定场景，两者在 WAL 保留方面的行为一致，xmin 计算不受影响。
5. **向后兼容**: 如果 `max_repack_replication_slots` 未在配置文件中显式设置，默认为 5，不影响已有部署的行为。

总体来说，这是一个"让专业工具各司其职"的设计思路。对于有高并发 REPACK 需求的系统，这是一个值得升级到位的功能。
