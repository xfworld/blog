## 背景 

本文深入解读 PostgreSQL 近期引入的 `EXPLAIN (IO)` 基础设施及相关扫描节点的实现（SeqScan、TidRangeScan、BitmapHeapScan），以及 `auto_explain.log_io` 配置参数。这一组 commit 为 PostgreSQL 的**预读（prefetch）**和**异步 I/O（async I/O）**提供了透明的统计信息输出能力，让 DBA 能够清晰看到扫描节点从 ReadStream 中预取了多少数据、实际的 I/O 请求次数、等待次数、平均预读深度等关键指标。

  

**背景痛点**：在 PostgreSQL 中，大表扫描性能高度依赖底层的预读机制和异步 I/O 能力。然而长期以来，DBA 只能通过 `pg_stat_activity`、系统级的 `iostat`/`iotop` 等手段间接推断预读行为是否生效，无法在 SQL 层直接观测到"当前查询的预读距离是多少"、"产生了多少次异步 I/O 请求"、"平均每次 I/O 读取了多少块"等核心指标。这导致 I/O 密集型场景下的性能调优更多依赖经验猜测而非数据驱动。

**核心价值**：这组 commit 首次在 SQL 层（`EXPLAIN ANALYZE IO`）和 `auto_explain` 插件层同时暴露了 ReadStream 的内部 I/O 统计信息，使 DBA 能够量化预读效率、识别预读不足或过度预读的问题，从而做出有据可查的调优决策（如调整 `effective_io_concurrency`、`effective_cache_size` 等参数）。

## 一、原理介绍

### 1.1 从"黑盒 I/O"到"透明可见"

在引入这组 commit 之前，`EXPLAIN ANALYZE` 能够告诉你：
- 扫描了多少行（`actual rows`）
- 循环了多少次（`actual loops`）
- 命中了多少 buffer（`Shared Hit Blocks`、`Shared Read Blocks`）

但它无法告诉你：
- 数据是从预读缓冲区（ReadStream）中直接获取的，还是发生了真正的磁盘 I/O
- 预读引擎的"预读深度"（look-ahead distance）是否合理
- 产生了多少次异步 I/O 请求，每次请求的大小和并发度如何

PostgreSQL 的扫描节点在处理大表时，底层会通过 **ReadStream** 模块使用异步 I/O 进行数据预取。ReadStream 维护一个缓冲区队列，在后台异步地向存储子系统发起预读请求，从而隐藏 I/O 延迟。但这套机制对 DBA 完全不可见——直到现在。

### 1.2 核心数据结构：IOStats

所有 I/O 统计信息的核心载体是 `IOStats` 结构体（定义于 `src/include/executor/instrument_node.h`）：

```c
typedef struct IOStats
{
    /* 预取统计：反映 ReadStream 的 look-ahead 行为 */
    uint64  prefetch_count;    /* 返回给消费者的缓冲区总数 */
    uint64  distance_sum;      /* 每次返回时的 pinned_buffers 累加值 */
    int16   distance_max;      /* 观察到的最大预读深度 */
    int16   distance_capacity; /* 预读队列最大容量（max_pinned_buffers）*/

    /* I/O 请求统计 */
    uint64  wait_count;        /* 因等待 I/O 完成的次数 */
    uint64  io_count;          /* 发起的 I/O 请求总次数 */
    uint64  io_nblocks;         /* 所有 I/O 请求读取的块总数 */
    uint64  io_in_progress;    /* "in-progress I/O 数量"的累加值（用于计算平均值）*/
} IOStats;
```

**关键指标解读**：

| 字段 | 含义 | 调优价值 |
|------|------|----------|
| `prefetch_count` | ReadStream 返回给扫描节点的缓冲区总数 | 反映扫描过程中预取的总数据量 |
| `distance_max` | 实际达到的最大预读深度 | 如果接近 `distance_capacity`，说明预读可能受限 |
| `io_count` | 实际发起的异步 I/O 请求次数 | 结合 `prefetch_count` 可判断预取命中率 |
| `wait_count` | 因 I/O 未完成而等待的次数 | 反映 I/O 瓶颈程度 |
| `io_nblocks / io_count` | 平均每次 I/O 请求读取的块数 | 反映 I/O 合并效率，过小说明预读策略有问题 |

### 1.3 统计信息的收集链路

**1. heapam.c 中的扫描初始化**（`heap_beginscan`）：

```c
/* enable read stream instrumentation */
if ((flags & SO_SCAN_INSTRUMENT) && (scan->rs_read_stream != NULL))
{
    scan->rs_base.rs_instrument = palloc0_object(TableScanInstrumentation);
    read_stream_enable_stats(scan->rs_read_stream,
                             &scan->rs_base.rs_instrument->io);
}
```

当扫描启用 `SO_SCAN_INSTRUMENT` 标志时（对应 `INSTRUMENT_IO` 选项），堆扫描初始化函数会分配 `TableScanInstrumentation` 结构，并通过 `read_stream_enable_stats()` 将其绑定到 ReadStream。此后 ReadStream 在每次预读缓冲区时自动更新统计计数器。

**2. ReadStream 中的统计更新**（`src/backend/storage/aio/read_stream.c`）：

```c
/* 每当从 ReadStream 中取出一个缓冲区时 */
static inline void
read_stream_count_prefetch(ReadStream *stream)
{
    IOStats *stats = stream->stats;
    if (stats == NULL)
        return;

    stats->prefetch_count++;
    stats->distance_sum += stream->pinned_buffers;  /* 当前队列深度 */
    if (stream->pinned_buffers > stats->distance_max)
        stats->distance_max = stream->pinned_buffers;
}

/* 每当发起一次异步 I/O 请求时 */
static inline void
read_stream_count_io(ReadStream *stream, int nblocks, int in_progress)
{
    IOStats *stats = stream->stats;
    if (stats == NULL)
        return;

    stats->io_count++;
    stats->io_nblocks += nblocks;
    stats->io_in_progress += in_progress;
}

/* 每当需要等待 I/O 完成时 */
static inline void
read_stream_count_wait(ReadStream *stream)
{
    IOStats *stats = stream->stats;
    if (stats == NULL)
        return;

    stats->wait_count++;
}
```

**3. 扫描结束时的统计汇总**：

对于 **SeqScan** 和 **TidRangeScan**（并行查询场景），worker 进程的统计信息通过 DSM（Dynamic Shared Memory）共享给主进程。以 SeqScan 为例（`nodeSeqscan.c` 的 `ExecEndSeqScan`）：

```c
if (node->sinstrument != NULL && IsParallelWorker())
{
    SeqScanInstrumentation *si;
    Assert(ParallelWorkerNumber < node->sinstrument->num_workers);
    si = &node->sinstrument->sinstrument[ParallelWorkerNumber];

    if (scanDesc && scanDesc->rs_instrument)
    {
        /* 将 worker 的 IO 统计合并到共享结构 */
        AccumulateIOStats(&si->stats.io, &scanDesc->rs_instrument->io);
    }
}
```

### 1.4 EXPLAIN 输出格式

使用 `EXPLAIN (ANALYZE, IO)` 时，输出示例（TEXT 格式）：

```
Seq Scan on orders  (cost=0.00..658947.00 rows=10000000 width=39)
                     (actual time=0.029..9823.451 rows=9999999 loops=1)
  Prefetch: avg=8.32 max=16 capacity=32
  I/O: count=1523 waits=47 size=8.19 in-progress=1.23
  Buffers: shared hit=0 read=9999999
```

- **Prefetch avg=8.32**：平均每次返回缓冲区时，队列中有 8.32 个缓冲区已预取但尚未被消费
- **Prefetch max=16**：观察到的最大预读深度为 16
- **Prefetch capacity=32**：预读队列最大容量为 32（即 `max_pinned_buffers`）
- **I/O count=1523**：本次扫描共发起了 1523 次异步 I/O 请求
- **I/O waits=47**：其中 47 次在消费缓冲区时需要等待 I/O 完成
- **I/O size=8.19**：平均每次 I/O 请求读取 8.19 个块

### 1.5 与现有 BUFFERS 选项的区别

| 维度 | `BUFFERS` | `IO` |
|------|-----------|------|
| 粒度 | buffer 级别的命中/脏/读取统计 | ReadStream 预读行为统计 |
| 关注点 | 数据是否在 shared buffer 中 | 数据是否从磁盘预读、预读效率如何 |
| 适用场景 | buffer pool 调优 | 预读策略调优、I/O 子系统评估 |
| 依赖 | 只需 `ANALYZE` | 必须 `ANALYZE + IO` |

## 二、应用场景及最佳实践

### 场景一：判断预读是否生效

**调优目标**：确认大表顺序扫描是否充分利用了 PostgreSQL 的预读机制。

**调优前**（无法观测）：
```sql
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM huge_table WHERE status = 'active';
-- 输出只显示Buffers: shared hit=0 read=1000000
-- 无法判断预读是否在工作
```

**调优后**（清晰可见）：
```sql
EXPLAIN (ANALYZE, IO) SELECT * FROM huge_table WHERE status = 'active';
```

```
Seq Scan on huge_table  (cost=0.00..658947.00 rows=5000000 width=39)
                        (actual time=0.029..15234.521 rows=5000000 loops=1)
  Prefetch: avg=7.12 max=15 capacity=32
  I/O: count=1523 waits=112 size=8.19 in-progress=2.01
  Buffers: shared hit=0 read=5000000
```

**判断方法**：
- 如果 `wait_count` 远大于 `io_count` 的 10%，说明 I/O 延迟较高，预读经常跟不上消费速度
- 如果 `distance_max` 远小于 `distance_capacity`，说明预读深度受限，可能需要增大 `effective_io_concurrency`

### 场景二：识别 I/O 合并效率问题

**问题**：某查询 `io_count` 非常高，但 `io_nblocks / io_count`（平均 I/O 大小）很小，说明预读请求没有被有效合并，每次只读很少的块。

**调优前**：
```
I/O: count=50000 waits=1000 size=1.23 in-progress=0.50
```
50000 次 I/O 请求，平均每次才 1.23 块——这是典型的预读失效场景。

**可能原因**：
1. `effective_io_concurrency` 设置过低，无法发起足够多的并发预读请求
2. 随机访问模式导致预读失效（如非连续的 TID 扫描）
3. 存储子系统 I/O 延迟过高，抑制了预读深度

**调优后**（调整 `effective_io_concurrency = 8`）：
```
I/O: count=1523 waits=12 size=8.19 in-progress=1.23
```
I/O 次数从 50000 降到 1523，平均 I/O 大小从 1.23 块提升到 8.19 块，预读效率显著提升。

### 场景三：BitmapHeapScan 的 IO 监控

BitmapHeapScan 是唯一在首个 commit（`681daed`）中就支持 IO 统计的扫描类型，因为它使用 ReadStream 进行位图页面的预取。

```sql
EXPLAIN (ANALYZE, IO)
SELECT * FROM orders WHERE customer_id IN (
    SELECT customer_id FROM preferences WHERE vip = true
);
```

```
Bitmap Heap Scan on orders  (cost=...)
                             (actual time=...)
  Prefetch: avg=4.21 max=8 capacity=16
  I/O: count=234 waits=5 size=5.67 in-progress=1.10
  Recheck Cond: ...
  ->  Bitmap Index Scan on idx_customer_id
```

这对于诊断 BitmapHeapScan 在复杂 WHERE 条件下的预取效率特别有价值。

### 场景四：通过 auto_explain 捕获生产环境的 I/O 异常

对于生产环境中难以复现的 I/O 问题，可以使用 `auto_explain.log_io` 在慢查询日志中自动记录 I/O 统计：

```sql
LOAD 'auto_explain';

SET auto_explain.log_analyze = on;
SET auto_explain.log_io = on;
SET auto_explain.log_min_duration = 1000;  -- 只记录超过1秒的查询
```

这样，Pg 日志中会自动输出类似：
```
2026-04-08 10:23:45.123 UTC [12345] LOG:  duration: 5234.521 ms  plan:
Seq Scan on huge_table  (cost=0.00..658947.00 rows=5000000 width=39)
                        (actual time=0.029..5234.521 rows=5000000 loops=1)
  Prefetch: avg=2.01 max=5 capacity=32
  I/O: count=152300 waits=89500 size=4.12 in-progress=15.30
  Buffers: shared hit=2341 read=4977659
```

当观察到 `wait_count` 异常高（如 `waits=89500`）时，说明 I/O 子系统是瓶颈，应考虑使用 I/O 延迟更低的存储或增加 `effective_io_concurrency`。

### 场景五：并行查询的 Per-Worker I/O 统计

当 `max_parallel_workers_per_gather > 0` 时，`EXPLAIN (ANALYZE, IO)` 还会显示每个 worker 的独立 I/O 统计：

```
Seq Scan on orders  (cost=0.00..658947.00 rows=10000000 width=39)
                     (actual time=0.029..9823.451 rows=9999999 loops=1)
  Prefetch: avg=8.32 max=16 capacity=32
  I/O: count=1523 waits=47 size=8.19 in-progress=1.23
  Buffers: shared hit=0 read=9999999
  Workers: 3
  Worker 0: Prefetch: avg=8.45 max=16
            I/O: count=508 waits=15 size=8.21 in-progress=1.20
  Worker 1: Prefetch: avg=8.18 max=15
            I/O: count=501 waits=14 size=8.17 in-progress=1.18
  Worker 2: Prefetch: avg=8.31 max=16
            I/O: count=514 waits=18 size=8.19 in-progress=1.31
```

这对于识别并行查询中 worker 之间的 I/O 不均衡（如某个 worker 遇到 I/O 热点）非常有帮助。

## 四、小结与思考

### 技术价值总结

这组 commit 的核心贡献，是将 PostgreSQL 底层 ReadStream 的 I/O 行为第一次以结构化的方式暴露给了 SQL 层使用者：

1. **量化预读效率**：`Prefetch avg/max/capacity` 让 DBA 能直接判断预读深度是否合理，不再依赖猜测
2. **诊断 I/O 瓶颈**：`wait_count` 和 `in-progress` 指标能快速定位 I/O 子系统是否成为查询瓶颈
3. **评估 I/O 合并效果**：`io_count` 和 `io_nblocks` 的比值揭示预读请求的合并效率
4. **并行查询可观测性**：per-worker 的 IO 统计让并行扫描的负载均衡分析成为可能
5. **与 auto_explain 集成**：生产环境的持续 I/O 监控成为可能

### 对 DBA 的实际意义

- **从经验调优到数据调优**：过去调整 `effective_io_concurrency`、`random_page_cost` 等参数时，DBA 往往只能通过反复实验和观察整体响应时间来判断效果。现在可以直接观察 `wait_count` 和 `distance_max` 的变化，用数据验证调优假设
- **缩短问题定位时间**：当 I/O 相关查询变慢时，IO 统计能快速区分是"预取不足"还是"I/O 延迟过高"，避免在错误的方向上排查
- **为存储选型提供依据**：通过对同一查询在不同存储上运行时的 IO 统计对比，可以客观评估存储子系统的预读和 I/O 吞吐能力

### 潜在的限制与后续演进

- 目前只有 `BitmapHeapScan`、`SeqScan`、`TidRangeScan` 三种扫描节点支持 `IO` 统计，其他使用 ReadStream 的节点（如 IndexScan 未来可能的支持）尚未覆盖
- `IO` 选项默认关闭，以避免在不需要时引入额外的统计开销
- 预读统计的精度受 `pg_config.h` 中 `max_pinned_buffers` 等编译时常量影响，在某些极端配置下可能存在差异

### 思考

这组 commit 体现了 PostgreSQL 近年来在"可观测性"方向上的持续投入——从 `EXPLAIN (ANALYZE, BUFFERS)` 到 `EXPLAIN (ANALYZE, WAL)` 再到现在的 `EXPLAIN (ANALYZE, IO)`，每一次新增的统计维度都在帮助 DBA 从更细粒度理解数据库的内部行为。ReadStream 作为 PostgreSQL 异步 I/O 战略方向的核心组件，其 I/O 行为对应用层的可见性提升，标志 PostgreSQL 在企业级 I/O 可观测性方面迈出了重要一步。
