## 背景 

当生产数据库发生连接异常中断时，DBA 常常面临一个棘手的问题：**是谁（哪个进程、哪个用户）发起了终止操作？** 是运维人员执行的 `pg_terminate_backend()`？是监控系统的自动清理？还是某位误操作的开发者？在此之前，PostgreSQL 的错误日志只能显示 "terminating connection due to administrator command"，却无法告诉您这个"administrator command"究竟来自何方。

本文介绍的 commit `55890a9` 为 PostgreSQL 带来了**信号溯源（Signal Source Attribution）**能力：当后端进程因 `SIGTERM` 被终止时，错误日志现在会通过 `errdetail` 附加发送者的 **PID（进程 ID）** 和 **UID（用户 ID）** 信息，使得在多用户、多服务混合的复杂环境中，能够快速定位终止信号的来源。

  

**背景痛点：** 在大型 PostgreSQL 部署中，多个管理员、自动化脚本和监控系统可能同时具备终止后端会话的权限。一旦出现非预期的连接中断，传统的错误日志只能告知"被管理员命令终止"，却无法指明是哪个进程、哪个用户发起了终止操作，导致故障排查如同大海捞针。

**核心价值：** 本次提交通过在信号处理层面捕获 `siginfo_t` 中的 `si_pid` 和 `si_uid`，并将其注入到所有 `ProcDiePending` 相关错误消息的 `errdetail` 中，实现了对终止信号来源的完整溯源能力。

---

## 原理介绍

### 1. 整体架构

PostgreSQL 的后端进程通过信号机制完成多种控制操作，其中 **SIGTERM** 是最常见的终止信号。当调用 `pg_terminate_backend(pid)` 或向一个 PostgreSQL 后端进程直接发送 SIGTERM 时，该后端进程会进入 `ProcDiePending = true` 状态，并在下一次中断处理点（`ProcessInterrupts()`）中触发终止流程。

本次修改的核心，是在信号捕获阶段（而非调用方）记录发送者的身份信息：

```
信号发送方 (pg_terminate_backend / kill/SIGTERM)
         │
         ▼
    [SIGTERM 携带 siginfo_t]
         │
         ▼
  pqsignal wrapper_handler()  ← 新增：提取 si_pid / si_uid
         │
         ▼
  全局变量 ProcDieSenderPid / ProcDieSenderUid
         │
         ▼
  ProcessInterrupts()         ← 新增：ERDDETAIL_SIGNAL_SENDER 宏
         │
         ▼
  错误日志包含 "Signal sent by PID XXX, UID YYY"
```

### 2. 平台能力检测：SA_SIGINFO

Unix 系统的信号处理有两种模式：
- **基本模式**（`sa_handler`）：仅接收信号编号
- **扩展模式**（`SA_SIGINFO`，通过 `sa_sigaction`）：可接收 `siginfo_t` 结构，其中包含信号发送者的详细信息

`siginfo_t` 中的关键字段：
- `si_pid`：发送信号的进程 ID
- `si_uid`：发送进程的真实用户 ID

configure 系统通过以下方式检测平台是否支持 `SA_SIGINFO`：

```c
// configure.ac / configure
AC_CACHE_CHECK([for SA_SIGINFO], [ac_cv_have_sa_siginfo], [
    AC_COMPILE_IFELSE([
        AC_LANG_PROGRAM([[
            #include <signal.h>
            #include <stddef.h>
        ]], [[
            struct sigaction sa;
            sa.sa_flags = SA_SIGINFO;
        ]])
    ],
    [ac_cv_have_sa_siginfo=yes],
    [ac_cv_have_sa_siginfo=no])
])

// meson.build 中同样通过 cc.has_header_symbol 检测
if cc.has_header_symbol('signal.h', 'SA_SIGINFO')
  cdata.set('HAVE_SA_SIGINFO', 1)
endif
```

**适用平台：** Linux、FreeBSD 和大多数现代 Unix 系统均支持。Windows 不支持 `SA_SIGINFO`，此功能在该平台上会自动降级（不显示 PID/UID）。

### 3. 信号处理器的改造

**文件：`src/port/pqsignal.c`**

原有的 `wrapper_handler` 使用基本信号签名：
```c
static void
wrapper_handler(SIGNAL_ARGS)  // SIGNAL_ARGS = int postgres_signal_arg
```

修改后，当 `HAVE_SA_SIGINFO` 可用时，切换为扩展签名：
```c
#if !defined(FRONTEND) && defined(HAVE_SA_SIGINFO)
static void
wrapper_handler(int signo, siginfo_t * info, void *context)
#else
static void
wrapper_handler(SIGNAL_ARGS)
#endif
```

在 `pqsignal()` 注册信号处理时，增加了 `SA_SIGINFO` 标志：

```c
#if !defined(FRONTEND) && defined(HAVE_SA_SIGINFO)
if (use_wrapper)
{
    act.sa_sigaction = wrapper_handler;
    act.sa_flags |= SA_SIGINFO;  // 启用扩展信息
}
else
    act.sa_handler = func;
#else
act.sa_handler = use_wrapper ? wrapper_handler : func;
#endif
```

在 `wrapper_handler` 内部，捕获 SIGTERM 的发送者信息：

```c
#ifdef HAVE_SA_SIGINFO
if (signo == SIGTERM && info)
{
    ProcDieSenderPid = info->si_pid;   // 发送者 PID
    ProcDieSenderUid = info->si_uid;   // 发送者 UID
}
#endif
```

### 4. 全局状态变量

**文件：`src/backend/utils/init/globals.c`**

```c
volatile int ProcDieSenderPid = 0;
volatile int ProcDieSenderUid = 0;
```

**文件：`src/include/miscadmin.h`**

```c
extern PGDLLIMPORT volatile int ProcDieSenderPid;
extern PGDLLIMPORT volatile int ProcDieSenderUid;
```

这些变量使用 `volatile` 修饰，因为它们在信号处理器中被写入（异步上下文），需要确保每次读取都从内存读取最新值而非寄存器缓存。

### 5. 错误消息宏

**文件：`src/backend/tcop/postgres.c`**

```c
#define ERRDETAIL_SIGNAL_SENDER(pid, uid) \
    ((pid) == 0 ? 0 : \
     errdetail("Signal sent by PID %d, UID %d.", (int) (pid), (int) (uid)))
```

这个宏的定义非常精妙：
- 当 `pid == 0` 时（平台不支持 SA_SIGINFO 或信息不可用），展开为 `0`（什么都不做）
- 否则，通过 `errdetail()` 追加信号发送者的 PID 和 UID

### 6. ProcessInterrupts 中的应用

**文件：`src/backend/tcop/postgres.c`（`ProcessInterrupts` 函数）**

```c
if (ProcDiePending)
{
    int  sender_pid = ProcDieSenderPid;
    int  sender_uid = ProcDieSenderUid;

    ProcDiePending = false;
    ProcDieSenderPid = 0;       // 捕获后立即清除，防止泄漏
    ProcDieSenderUid = 0;
    QueryCancelPending = false;

    // ... 清理逻辑 ...

    if (AmAutoVacuumWorkerProcess())
        ereport(FATAL,
                (errcode(ERRCODE_ADMIN_SHUTDOWN),
                 errmsg("terminating autovacuum process due to administrator command"),
                 ERRDETAIL_SIGNAL_SENDER(sender_pid, sender_uid)));
    else if (IsLogicalWorker())
        ereport(FATAL,
                (errcode(ERRCODE_ADMIN_SHUTDOWN),
                 errmsg("terminating logical replication worker due to administrator command"),
                 ERRDETAIL_SIGNAL_SENDER(sender_pid, sender_uid)));
    // ... 其他进程类型 ...
    else
        ereport(FATAL,
                (errcode(ERRCODE_ADMIN_SHUTDOWN),
                 errmsg("terminating connection due to administrator command"),
                 ERRDETAIL_SIGNAL_SENDER(sender_pid, sender_uid)));
}
```

注意：在将 `ProcDieSenderPid/Uid` 复制到局部变量后，立即将其清零。这是因为：
1. 防止同一个值被重复使用（信号只应被处理一次）
2. 避免在后续逻辑中泄露敏感信息

### 7. 同步复制等待中的特殊处理

**文件：`src/backend/replication/syncrep.c`（`SyncRepWaitForLSN` 函数）**

在同步复制等待期间收到终止信号时，PostgreSQL 不能简单地抛出 FATAL（因为事务已提交），而是发出 WARNING 并内联附加信号发送者信息：

```c
ereport(WARNING,
        (errcode(ERRCODE_ADMIN_SHUTDOWN),
         errmsg("canceling the wait for synchronous replication and terminating connection due to administrator command"),
         errdetail("The transaction has already committed locally, but might not have been replicated to the standby.%s",
                   ProcDieSenderPid == 0 ? "" :
                   psprintf("\nSignal sent by PID %d, UID %d.",
                            (int) ProcDieSenderPid,
                            (int) ProcDieSenderUid))));
```

这里采用内联方式而非 `ERRDETAIL_SIGNAL_SENDER` 宏，是因为该位置的 `errdetail` 消息需要追加到已有的 detail 字符串之后（"The transaction has already..."），而宏是独立的追加操作。

## 应用场景及最佳实践

### 场景一：多管理员环境下的误操作溯源

**背景：** 企业 PostgreSQL 集群由多个 DBA 团队共管，每位 DBA 都拥有 `pg_terminate_backend` 的执行权限。

**Commit 前的行为：**
```
2026-04-08 10:23:45.123 UTC [12345] FATAL:  terminating connection due to administrator command
2026-04-08 10:23:45.124 UTC [12345] LOG:  connection received from 192.168.1.100 port 54321
```

问题：只知道是"管理员命令"导致，却不知道是哪位管理员（哪个进程）发起的。

**Commit 后的行为：**
```
2026-04-08 10:23:45.123 UTC [12345] FATAL:  terminating connection due to administrator command
    DETAIL:  Signal sent by PID 67890, UID 1005.
2026-04-08 10:23:45.124 UTC [12345] LOG:  connection received from 192.168.1.100 port 54321
```

溯源过程：
1. PID 67890 → 通过 `ps` 或 `/proc` 确认为某监控脚本
2. UID 1005 → 通过 `id` 或 `/etc/passwd` 确认为 `dba_user_zhang`

**操作建议：** 结合进程审计（auditd 或 systemd-journald），可以追溯 PID 67890 在当日的完整命令历史，确认为 `pg_terminate_backend(12345)` 调用。

### 场景二：自动化运维工具的故障诊断

**背景：** 使用 pgBadger / pgCluu 等工具进行慢查询分析时，某个后台会话在分析过程中被终止。

**Commit 前：**
```
FATAL:  terminating connection due to administrator command
```

运维人员需要手动对比时间戳和所有可能的调用来源，效率极低。

**Commit 后：**
```
FATAL:  terminating connection due to administrator command
    DETAIL:  Signal sent by PID 31415, UID 0.
```

- UID 0 表示由 root 用户或系统服务发起
- 结合 systemd journal：`journalctl _PID=31415` 可快速定位是哪个 systemd 服务

### 场景三：Kubernetes / 容器环境中的信号追踪

**背景：** 在 K8s 环境中，`kubectl delete pod` 会向容器发送 SIGTERM。如果 PostgreSQL pod 在服务期间被滚动更新终止，日志可以清晰显示是 K8s 控制器（kubelet）发起的终止：

**Commit 前的日志：**
```
FATAL:  terminating connection due to administrator command
```

**Commit 后的日志：**
```
FATAL:  terminating connection due to administrator command
    DETAIL:  Signal sent by PID 1, UID 0.
```

- PID 1 在容器中通常是容器的 init 进程（如 `/pause` 或 docker-init）
- 结合 K8s events：`kubectl get events` 可确认是滚动更新还是手动删除

### 场景四：云数据库（ RDS / Cloud SQL ）环境审计

在托管数据库环境中，当客户连接被平台管理员强制终止时，错误日志会包含平台的操作进程信息，配合云平台的审计日志（CloudTrail / Audit Log），可完整还原操作链路。

## 小结和思考

### 核心要点回顾

1. **实现机制：** 利用 POSIX 的 `SA_SIGINFO` 扩展信号能力，在 `wrapper_handler` 中提取 `siginfo_t` 的 `si_pid` 和 `si_uid`，存储于全局变量，并在所有 `ProcDiePending` 相关错误路径中通过 `errdetail` 展示。

2. **平台兼容性：** 仅在支持 `SA_SIGINFO` 的平台上生效（Linux、FreeBSD 等），其他平台优雅降级，不影响功能。

3. **信息粒度：** 精确到 PID 和 UID（而非用户名），确保了信息的精确性和可追溯性。UID 相比用户名更可靠，因为用户名在用户退出后可能被回收。

4. **安全性考量：** 捕获后立即清零全局变量，防止信息泄露或重复使用；PID/UID 为 0 时（不可用）不输出 `errdetail`，保证日志格式一致性。

### 对 DBA 的实际意义

| 维度 | 改进前 | 改进后 |
|------|--------|--------|
| 故障定位时间 | 数小时（大海捞针式排查） | 数分钟（直接定位进程） |
| 审计追溯能力 | 依赖外部监控/审计系统 | 直接内嵌于 PostgreSQL 日志 |
| 多租户环境 | 难以区分不同租户的终止操作 | 可精确区分不同用户的终止 |
| 自动化运维 | 难以判断是哪个自动化工具 | 每个工具的 PID/UID 可追溯 |

### 延伸思考

1. **未来可能的增强方向：**
   - 增加用户名解析（通过 `getpwuid(uid)` 将 UID 转换为用户名）
   - 支持 `pg_cancel_backend()` 的溯源（目前仅处理 `SIGTERM`/`ProcDiePending`）
   - 在 `pg_stat_activity` 的 `state_change` 时间点字段之外，增加信号来源信息

2. **与现有监控体系的协同：** 此功能并不替代外部监控/审计系统（如 `auditd`、云平台审计日志），而是与之互补——PostgreSQL 日志提供了最直接的上下文，外部系统提供了更宏观的操作历史。

3. **日志管理建议：** 随着 PID/UID 信息的引入，日志分析工具（如 pgBadger）可以新增"按终止信号来源统计"的维度，帮助识别异常的终止行为模式。
