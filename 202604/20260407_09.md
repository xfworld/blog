## 背景 

本文深入解读 PostgreSQL 提交 `0d3dba38c777384a9dd7dffe924355c9683a6b71` 实现的"数据库级逻辑复制快照"优化。该优化允许逻辑解码在 startup snapshot 构建阶段、且不需要访问共享系统目录时，只关注当前数据库内的事务，从而避免因其他数据库长事务导致的阻塞问题。

  

**背景痛点**: 在 PostgreSQL 多数据库集群中，当某个数据库执行 `REPACK (CONCURRENTLY)` 或其他需要逻辑解码的操作时，其他数据库中运行相同操作的后端会被无端阻塞。这是因为原有的快照构建器需要等待集群中所有数据库的所有运行中事务完成，而插件实际上可能只依赖当前数据库对象和非 shared catalogs。

**核心价值**: 通过在输出插件中声明"不需要共享目录访问"，快照构建器可以将**启动阶段**的等待范围缩小到当前数据库，避免不必要的跨数据库等待，显著提升 REPACK (CONCURRENTLY) 等操作的集群级并发能力。注意：db-specific `xl_running_xacts` 仅用于加速达到一致点；达到一致点后的资源清理与实际解码仍按常规 cluster-wide `running_xacts` 处理，shared catalog invalidation 只是其中一种相关场景。

---

## 1. 原理介绍

### 1.1 逻辑复制快照机制回顾

在 PostgreSQL 逻辑复制中，**快照（Snapshot）** 是理解数据库在某一时刻一致性状态的核心机制。每次逻辑解码开始时，快照构建器（Snapshot Builder）需要确定：

- 哪些事务已经提交（对解码可见）
- 哪些事务正在运行（对解码不可见）
- 哪些事务尚未开始

原有的快照构建逻辑假设解码过程可能访问**共享系统目录（shared catalogs）**（如 `pg_database`、`pg_tablespace`、`pg_authid` 等），这意味着：

```
快照必须包含：集群中所有数据库的所有运行中事务
原因：其他数据库的事务可能修改了我们需要的共享目录
```

### 1.2 问题场景：跨数据库 REPACK 阻塞

`REPACK`（含 `REPACK (CONCURRENTLY)`）是 PostgreSQL 内核提供的在线表重组命令，其 `CONCURRENTLY` 路径下使用内置输出插件 `pgrepack` 做逻辑解码驱动的表重建。其工作原理简述：

```
1. 创建表的新版本（空 relfilenode/文件）
2. 把旧表现有内容全量复制到新版本
3. 通过逻辑解码捕获复制期间发生的并发 DML 增量
4. 在短暂 ACCESS EXCLUSIVE 锁下完成文件切换
```

问题发生在以下场景：

```
时间点 | 数据库 A (repack) | 数据库 B (repack) | 快照状态
------|------------------|------------------|-------------------
T1    | 事务开始，分配XID | -                | 等待数据库A事务
T2    | -                | 事务开始，分配XID  | 等待 A 和 B 事务
T3    | 运行中...        | 运行中...         | 数据库A和B的事务都被包含
T4    | 等待其他事务完成   | 等待其他事务完成     | **跨数据库串行化，长时间等待**
```

**根本原因**：快照构建器不知道 REPACK 插件实际上只依赖当前数据库对象、且不访问 shared catalogs，因此它保守地等待所有数据库的事务完成。

### 1.3 核心优化：数据库级快照

该 commit 的核心思想是：**如果输出插件声明不访问共享目录，则快照构建器在**启动阶段**只需要按数据库过滤运行中事务列表（xids[] / subxids），以避免等待跨数据库的无谓事务**。

> 注意：当 `dbid` 有效时，db-specific 语义主要体现在 `dbid` 过滤后的 `xids[]/subxids[]` 与 `oldestRunningXid`；`oldestDatabaseRunningXid` 是额外维护的当前数据库最老运行事务信息，不等同于"db-specific 模式下的真正 xmin 边界"。达到一致点后的资源清理与实际解码仍按常规 cluster-wide `running_xacts` 处理。

#### 1.3.1 新增 OutputPluginOptions 字段

```c
// src/include/replication/output_plugin.h
typedef struct OutputPluginOptions
{
    OutputPluginOutputType output_type;
    bool        receive_rewrites;
    bool        need_shared_catalogs;  // 新增字段
} OutputPluginOptions;
```

插件在 startup 回调中设置此标志：

```c
// src/backend/replication/pgrepack/pgrepack.c
static void
repack_startup(LogicalDecodingContext *ctx, OutputPluginOptions *opt, bool is_init)
{
    // ...
    /*
     * REPACK doesn't need access to shared catalogs, so we can speed up the
     * historic snapshot creation by setting this flag.  We'll only have to
     * wait for transactions in our database.
     */
    opt->need_shared_catalogs = false;  // 关键声明
}
```

#### 1.3.2 GetRunningTransactionData 按数据库过滤

原有函数获取整个集群的事务信息：

```c
// 原有签名
RunningTransactions GetRunningTransactionData(void);

// 优化后签名
RunningTransactions GetRunningTransactionData(Oid dbid);
```

当 `dbid` 参数有效时，**运行中事务数组（xids[] / subxids[]）与 `oldestRunningXid`** 都会按数据库过滤；`oldestDatabaseRunningXid` 单独维护，是针对当前数据库的 oldest xid：

> 注意：此处"按库过滤"指的是 xids[]/subxids[] 数组和 `oldestRunningXid` 都按 `proc->databaseId == dbid` 筛选；`oldestDatabaseRunningXid` 是额外的当前数据库最老运行事务信息，不等同于 db-specific 模式下的真正 xmin 边界。

```c
// src/backend/storage/ipc/procarray.c
RunningTransactions
GetRunningTransactionData(Oid dbid)
{
    // ...
    for (index = 0; index < arrayP->numProcs; index++)
    {
        // ...
        if (OidIsValid(dbid))
        {
            int pgprocno = arrayP->pgprocnos[index];
            PGPROC *proc = &allProcs[pgprocno];

            if (proc->databaseId != dbid)
                continue;  // 跳过其他数据库的事务
        }
        // 处理当前数据库的事务...
    }
    // ...
    CurrentRunningXacts->dbid = dbid;  // 记录这是哪个数据库的快照
}
```

#### 1.3.3 xl_running_xacts 新增 dbid 字段

WAL 记录结构体新增数据库标识：

```c
// src/include/storage/standbydefs.h
typedef struct xl_running_xacts
{
    Oid         dbid;           /* only track xacts in this database */
    int         xcnt;           /* # of xact ids in xids[] */
    int         subxcnt;        /* # of subxact ids in xids[] */
    bool        subxid_overflow; /* snapshot overflowed, subxids missing */
    TransactionId nextXid;      /* xid from TransamVariables->nextXid */
    TransactionId oldestRunningXid; /* *not* oldestXmin */
    TransactionId latestCompletedXid;   /* so we can set xmax */
    TransactionId xids[FLEXIBLE_ARRAY_MEMBER];
} xl_running_xacts;
```

#### 1.3.4 SnapBuildProcessRunningXacts 优化逻辑

```c
// src/backend/replication/logical/snapbuild.c
void
SnapBuildProcessRunningXacts(SnapBuild *builder, XLogRecPtr lsn,
                              xl_running_xacts *running,
                              bool db_specific)  // 新增参数
{
    if (builder->state < SNAPBUILD_CONSISTENT)
    {
        if (db_specific)
        {
            /*
             * 如果只需要当前数据库的事务信息，
             * 但收到的记录是其他数据库的，
             * 则主动触发一条当前数据库的 xl_running_xacts WAL 记录，
             * 以加速达到 CONSISTENT 状态（避免等待 bgwriter/checkpointer 下次自然写出）。
             * 这不是在逻辑上"切换快照来源"。
             */
            if (running->dbid != MyDatabaseId)
            {
                LogStandbySnapshot(MyDatabaseId);
                return;
            }

            /*
             * 进入可按当前数据库启动的解码路径后，关闭“允许访问 shared catalogs”
             * 的默认假设，以便后续断言检查插件没有违背声明。
             * accessSharedCatalogsInDecoding 初始为 true（表示当前仍按
             * “可能访问 shared catalogs”处理），此处将其置为 false，
             * 仅作为断言辅助状态，不等同于插件的声明本身
             *（真正的声明是插件在 startup 回调中设置 opt->need_shared_catalogs = false）。
             */
            if (accessSharedCatalogsInDecoding)
                accessSharedCatalogsInDecoding = false;
        }
        // ...
    }
    // ...
}
```

#### 1.3.5 安全检查机制

为防止插件错误声明不访问共享目录导致数据不一致，在 catalog scan 路径新增断言检查：

```c
// src/backend/access/index/genam.c
Assert(!HistoricSnapshotActive() ||
       accessSharedCatalogsInDecoding ||
       !heapRelation->rd_rel->relisshared);
```

当前新增的是基于 catalog scan 路径的断言式安全检查，不是完整的运行时防护机制。如果插件声明不需要共享目录但实际访问了共享目录，在 debug/assert 构建下会触发断言失败；在关闭断言的构建中，如果插件错误声明，风险更接近静默错误或不一致。

### 1.4 WAL 版本号更新

由于 `xl_running_xacts` 结构体新增 `dbid` 字段，WAL 格式发生变化，需要更新 WAL 版本号：

```c
// src/include/access/xlog_internal.h
-#define XLOG_PAGE_MAGIC 0xD11E
+#define XLOG_PAGE_MAGIC 0xD11F
```

物理 standby 侧对数据库级记录有显式忽略逻辑：

```c
// src/backend/storage/ipc/standby.c
/*
 * Records issued for specific database are not suitable for physical
 * replication because that affects the whole cluster.
 */
if (OidIsValid(xlrec->dbid))
    return;  // 跳过数据库级记录
```

## 2. 应用场景及最佳实践

### 2.1 场景一：多数据库并行 REPACK

**场景描述**：DBA 需要在生产库上同时对两个大型表进行在线重组，这两个表位于不同的数据库中。

**优化前（阻塞场景）**：

```
数据库 prod_a 的表 orders (1亿行) - 执行 REPACK (CONCURRENTLY)
数据库 prod_b 的表 inventory (5000万行) - 期望同时执行 REPACK (CONCURRENTLY)

时间轴：
T0: prod_a.REPACK 和 prod_b.REPACK 同时开始，各自启动逻辑解码
T1: prod_a 快照构建时发现 prod_b 有一个长事务 TX（跨数据库，已分配XID）
     prod_b 快照构建时发现 prod_a 也有一个长事务 TX'（跨数据库，已分配XID）
T2: prod_a 等待 TX 提交才能进入一致点 ← 阻塞发生
     prod_b 等待 TX' 提交才能进入一致点 ← 阻塞发生
     若 TX/TX' 需要 2 小时，两个 REPACK 将长时间卡在各自的一致点等待阶段
     跨数据库 REPACK 会被串行化，后启动者可能被先启动者阻塞
```

**优化后（并行执行）**：

```
prod_a.REPACK 设置 need_shared_catalogs = false
prod_b.REPACK 设置 need_shared_catalogs = false

时间轴：
T0: prod_a.REPACK 开始，快照构建时 xids[]/subxids 只包含 prod_a 的事务
T1: prod_b.REPACK 开始，快照构建时 xids[]/subxids 只包含 prod_b 的事务
T2: 两者可以真正并行执行，互不阻塞
```

**预期效果**：在存在跨数据库长事务/并发 REPACK 时，可显著减少跨数据库无关等待，从而提升跨数据库并行运行的机会和上限。

### 2.2 潜在适用方向（未实现）：逻辑复制订阅并发初始化

该 patch 也可能减少某数据库中 REPACK 对其他数据库 subscription 相关 walsender 启动的影响，但 `pgoutput` 目前未启用此优化，且是否可用取决于插件实现是否完全不依赖 shared catalogs。

> **注意**：此场景尚未落地。当前仅有 PostgreSQL 内核 `REPACK` 命令在 `CONCURRENTLY` 路径下使用内置输出插件 `pgrepack`；逻辑复制插件理论上也许可用，但 commit message 仅说"could possibly be used ... need thorough analysis"，并未承诺本文之外的更多已落地场景。

### 2.3 最佳实践建议

#### 2.3.1 REPACK 使用建议

```bash
# REPACK (CONCURRENTLY) 是 PostgreSQL 内核内置命令
# 在包含 commit 0d3dba38c... 的版本中，通过内置 pgrepack 输出插件自动启用数据库级快照优化
psql -d mydb -c "REPACK (CONCURRENTLY) my_schema.my_table;"
```

#### 2.3.2 监控逻辑解码状态

```sql
-- 通用逻辑复制槽排查：查看当前逻辑槽状态
SELECT slot_name, plugin, database, temporary, active, restart_lsn
FROM pg_replication_slots
WHERE slot_type = 'logical';

-- 通用逻辑复制槽排查：查看持久逻辑槽的确认位置
-- 注意：REPACK (CONCURRENTLY) 使用的是临时槽 repack_<pid>，
-- 且其 ad-hoc decoding 路径不依赖 confirmed_flush_lsn 作为主要进度信号；
-- 快照构建状态 SnapBuildState 目前也没有直接的 SQL 级观测视图
SELECT slot_name, confirmed_flush_lsn
FROM pg_replication_slots;

-- 排查可能影响逻辑槽启动/一致点构建的已分配 XID 事务
-- 仅统计 backend_xid 非空的会话（已分配 XID 的运行事务）
-- 注意：这是近似排查视角，与 GetRunningTransactionData() 的 running xact 判断不完全等价
SELECT datname, count(*)
FROM pg_stat_activity
WHERE backend_xid IS NOT NULL
GROUP BY datname
ORDER BY count(*) DESC;
```

#### 2.3.3 避免的错误用法

如果输出插件**确实需要访问 shared catalogs**（如 `pg_database`、`pg_authid`、`pg_tablespace`），**不应**设置 `need_shared_catalogs = false`，否则可能导致：

- 读取到不一致的目录状态
- 解码出的变更与实际不一致
- 在断言启用时触发崩溃

## 3. 小结和思考

### 3.1 技术总结

本次优化是 PostgreSQL 逻辑复制快照机制的精细化改进：

1. **新增 API**：在 `OutputPluginOptions` 中引入 `need_shared_catalogs` 布尔标志

2. **快照粒度控制**：`GetRunningTransactionData(Oid dbid)` 和 `LogStandbySnapshot(Oid dbid)` 支持按数据库过滤

3. **安全机制**：`accessSharedCatalogsInDecoding` 作为断言式安全检查的辅助状态，用于在部分 catalog scan 路径验证插件没有违背声明（见 `src/include/miscadmin.h` 中 "as a safety check" 注释）

4. **WAL 兼容性**：该 commit 修改了 WAL record layout，引入了新的 WAL magic；db-specific `xl_running_xacts` 记录在物理 standby redo 时会被显式忽略，避免级联问题

5. **首次应用**：pgrepack 输出插件率先采用此优化，验证了设计有效性

### 3.2 深层思考

**为什么 REPACK 可以使用数据库级快照？**

REPACK 的核心操作是：
1. 创建一个与原表结构相同的新表（新的 relfilenode/文件）
2. 将原表数据 COPY 到新表
3. 创建索引
4. 通过逻辑解码捕获复制期间的并发 DML 增量
5. 在短暂 ACCESS EXCLUSIVE 锁下完成文件切换（relfilenode 层面的底层存储交换）

REPACK 可能访问当前数据库内的普通系统目录，但不需要访问跨数据库共享的 shared catalogs（如 `pg_database`、`pg_tablespace`、`pg_authid` 等），因此可以安全地声明 `need_shared_catalogs = false`。

**未来扩展方向**：

1. **其他输出插件**：如 `pgoutput`、`wal2json`、`test_decoding`、`pglogical` 的服务端插件等可以考虑采用此优化。像 Debezium 这类上层 CDC 客户端是否受益，取决于其所使用的服务端输出插件是否支持并启用该选项
2. **自定义插件开发**：如果插件不访问共享目录，应被鼓励设置此标志
3. **文档完善**：需要更多最佳实践指导插件开发者

### 3.3 DBA 视角的启示

1. **WAL 兼容性说明**：该 commit 修改了 WAL record layout，因此引入新的 WAL magic；带有效 `dbid` 的 `xl_running_xacts` 记录在物理 standby redo 时会被显式忽略（`standby.c` 中的 `return` 逻辑）。是否需要针对新服务端版本重新编译输出插件，取决于目标版本和扩展构建方式，不宜一概而论

2. **性能问题排查新思路**：
   ```
   复制槽创建、walsender 启动或 REPACK worker 建立一致快照时变慢，
   但本库内又无明显相关事务时 → 可能是跨数据库已分配 XID 的事务造成了保守等待
   ```

3. **运维建议**（条件化）：
   - 如果业务天然分库，且对应输出插件不访问 shared catalogs，该优化可减少跨数据库无关等待（主要体现在 replication slot 创建/启动阶段；逻辑解码文档也明确说明 `need_shared_catalogs = false` 可加速相关 slot 创建）
   - REPACK 任务可安全地并行调度到不同数据库
   - 不宜将此优化外推为"把表拆分到独立数据库"的架构级性能建议
